# 방법론 백서

> **본 문서의 위상**
> 이 백서는 본 방법론 프로젝트의 **철학·목표·원칙**을 정의하는 최상위 문서이다.
> 모든 가이드(`20_guides/*`), 운영 규칙(`CLAUDE.md`, `AGENTS.md`), 자산 스키마(Catalog/Skeleton)는 이 백서의 원칙에 종속된다.
> 백서와 하위 문서가 철학·목표·운영 원칙에서 충돌할 경우, **백서가 이긴다**.
> 단, 구현 사실은 코드가 이기고, 결정 근거는 ADR이 이긴다.
> 백서를 수정할 때는 사람이 게이트 통과시켜야 한다(Class C 변경, ADR 필수).

---

## 0. 한 줄 요약

> **"같은 문제에 두 번 부딪히지 않는 작업 자산 시스템.
> 도구는 갈아탈 수 있고, 자산은 남는다."**

---

## 1. 배경 — 우리가 해결하려는 문제

소규모·다(多)도메인으로 일하는 작업자(혼자 또는 소수 팀)는 다음 세 가지 비효율을 반복적으로 겪는다.

1. **반복 마찰의 누적**
   프로젝트마다 기술 스택·도메인이 달라 보여도, 실제로 시간을 잡아먹는 마찰점은 *놀라울 정도로 자주 동일하다*. 매번 "처음 보는 듯이" 같은 함정에 빠진다.

2. **작업물 단위의 단절**
   각 프로젝트가 "끝난 산출물 1건"으로 종료되며, 그 과정에서 얻은 *암묵지*(어떤 의존성 조합이 잘 작동했는가, 어떤 프롬프트가 효과적이었는가, 어떤 결정이 후회로 이어졌는가)가 *다음 프로젝트로 이전되지 않는다*. 머릿속에만 남고, 머리 밖으로 나오면 즉시 휘발한다.

3. **AI 도구·환경의 휘발성**
   AI 모델은 빠르게 교체되고, 도구(Claude Code / Cursor / Codex / 기타)는 각자 다른 인터페이스를 가지며, 작업자는 여러 PC를 오간다. 한 도구·한 모델·한 PC에 종속된 워크플로는 6개월 안에 부패한다.

본 방법론은 이 세 가지를 **단일 자산 저장소**로 흡수하여, 다음 작업이 항상 *지난 작업의 잔재 위에서 시작*되도록 한다.

## 2. 핵심 철학 (3대 원칙)

원칙은 **충돌 시 우선순위가 명확**해야 한다. 본 방법론의 원칙은 위에서부터 강하다.

### 제0원칙 — 이식성 (Portability First)

> **도구는 갈아탈 수 있다. 자산은 남는다.**

어떤 AI 모델·어떤 PC·어떤 시점에서 누가 이어받든, **저장소 하나만 있으면 즉시 같은 컨텍스트로 진입**할 수 있어야 한다.

- **모델 이식성**: Claude / GPT / Gemini / Local LLM — 어떤 AI에서도 작동
- **세션 이식성**: 한 AI가 중단한 지점에서 다른 AI가 즉시 재개
- **환경 이식성**: macOS / Linux / Windows(WSL) / 클라우드 컨테이너 어디서도 동작

이식성과 다른 원칙이 충돌할 때 **이식성이 이긴다**. 가장 강력한 자동화라도 특정 도구에 종속된다면, 그 자동화는 코어가 아닌 *어댑터*로 격리한다.

### 제1원칙 — 비반복 (No Twice Hit)

> **같은 문제에 두 번 부딪히지 않는다.**

이미 한 번 해결된 문제는 다음 작업에서 다시 발견 가능해야 하고, 반복 확인된 문제는 다음 작업 시작 시점에 해결책이 적용되어야 한다. 문서로 적어두는 것만으로는 부족하다 — 적용까지가 학습이다.

이 원칙은 시스템이 *무엇을 기록할지·승급시킬지·버릴지* 결정할 때의 유일한 가치 함수다. "유용해 보인다"가 아니라 **"이게 다음 번 같은 문제를 막는가?"** 가 통과 조건.

비반복은 3단계로 실행된다.

| 단계 | 조건 | 저장 위치 | 다음 작업에 미치는 영향 |
|---|---|---|---|
| **관찰** | 1회 발생 | `50_resources/ai_observations/` | 검색 가능한 raw 데이터로 남김 |
| **후보** | 1회 해결 + 재사용 가능성 있음 | `50_resources/catalog/_pending/` | 사람이 참고할 수 있으나 Skeleton에 굽지 않음 |
| **활성** | 동일 마찰 N≥2회 또는 사람이 명시 승인 | `50_resources/catalog/` | Skeleton에 굽고 대시보드에 노출 |

### 제2원칙 — 인계 가능성 (Handoff-Ready by Default)

> **누구든 이 자산을 상속받은 사람은, 과거의 시행착오 비용을 0으로 시작한다.**

본인의 미래 자아도 "다른 사람"이다. 6개월 뒤의 자신은 오늘의 컨텍스트를 잃는다. 따라서 *모든 자산은 항상 인계 시점을 가정하고 작성*된다. 친절함이 아니라 *생존 조건*이다.

판정 기준: **15분 안에 첫 의미 있는 작업을 시작할 수 있는가?** 못 하면 시스템 실패.

## 3. 목표 (Goals)

### G1 — 누구나 쉽게 학습해서 쉽게 적용한다

- 신규 사용자(또는 신규 AI 도구)가 **단일 진입점**(`README.md` → `ONBOARDING.md` → `10_foundation/WHITEPAPER.md`)을 통해 부팅 가능
- 첫 작업 시작까지 **15분 이내**
- 도메인별 첫 PR까지 **1일 이내**(스켈레톤 적용 기준)

### G2 — 대시보드를 통한 단일 가시성

- 모든 활성 프로젝트의 *현재 상태*, *다음 액션*, *과거 결정*에 단일 페이지로 도달
- 검색 박스 하나로 모든 도큐멘트·결정·과거 마찰·솔루션 즉시 조회
- 어떤 시점에서도 "지금 우리가 어디까지 와 있는가"를 30초 이내 파악

### G3 — 자가발전 씽크탱크

- AI 관찰 로그 + Git/PR 메타데이터를 *주기적으로 마이닝*하여 다음 항목을 **제안(자동 적용 아님)**:
  - 반복되는 마찰 → Catalog 승급 후보
  - 자주 하는 실수 패턴 → 가이드 보강 제안
  - 효과적인 프롬프트 패턴 → 슬래시 명령·skill 후보
  - 의존성·스택 변동 추세 → Skeleton 업데이트 신호
- 모든 제안은 **사람의 게이트(머지)** 를 거친다. 시스템이 자기 자신을 마음대로 바꾸지 않는다.

## 4. 비-목표 (Non-Goals) — 명시적 스코프 제한

이 방법론은 다음을 *지향하지 않는다*. 이걸 명시해야 스코프 침식을 막을 수 있다.

| 비-목표 | 이유 |
|---|---|
| **본격적인 강화학습 시스템** | 보상함수 자동 정책 개선이 아니라, *사람 게이트가 있는 패턴 마이닝*이다. "씽크탱크"는 제안 엔진이지 자율 의사결정자가 아니다. |
| **모든 지식의 단일 진리원** | 코드는 코드의 진리, ADR은 결정의 진리. 백서·Catalog는 *이전 가능한 자산*만 담는다. 위키 대체가 아니다. |
| **다중 사용자 협업 플랫폼** | 1인~소수 팀을 1차 타깃으로 한다. 동시 편집·권한 관리·실시간 협업은 비-목표. |
| **특정 AI 도구의 강화** | Claude Code 스킬이나 Cursor 룰은 *어댑터*로만 존재. 코어가 특정 도구를 더 강력하게 만드는 일은 하지 않는다. |
| **완벽한 자동화** | 자동화는 항상 *이식성을 해치지 않는 범위* 에서만 추구한다. 제0원칙이 항상 우선. |

## 5. 시스템 구조 (개요)

> 상세 구현은 별도 가이드(`20_guides/`)와 ADR에서 정의한다. 본 백서는 *왜 이런 구조인가*만 설명한다.

```
┌────────────────────────────────────────────────────────────────┐
│ L4. 대시보드 + 온보딩          ← 사람을 위한 단일 진입점        │
├────────────────────────────────────────────────────────────────┤
│ L3. 씽크탱크 (Telemetry)       ← 패턴 마이닝, 제안 생성         │
├────────────────────────────────────────────────────────────────┤
│ L2. 활성 자산 (Catalog/Skeleton) ← 다음 프로젝트가 실제로 쓰는 것 │
├────────────────────────────────────────────────────────────────┤
│ L1. AI 관찰 로그 (raw)         ← 매 세션 자동 기록              │
├────────────────────────────────────────────────────────────────┤
│ L0. 이식성 코어 (.ai/, HANDOFF) ← 어떤 AI/PC도 즉시 부팅 가능    │
└────────────────────────────────────────────────────────────────┘
```

### L0 — 이식성 코어
- `.ai/context.json`: 모든 AI가 부팅 시 읽는 정형 컨텍스트(스키마 버전 포함)
- `.ai/checkpoint.md`: 세션 종료 시 AI가 남기는 *다음 사람을 위한 1페이지 인계서*
- `.ai/adapters/{claude,cursor,gpt,generic}.md`: 도구별 보조 지침 (코어와 분리)
- `HANDOFF.md`: 살아있는 단일 진리. 150줄 한도.

### L1 — AI 관찰 로그 (raw)
- 사용자가 채우지 않는다. **AI가 매 세션 종료 시 자동 기록**.
- 부패해도 OK — 정제는 L3가 한다.
- 고정 6필드 + 자유서술 1필드. *친절·요약·서술 금지*, 다음 자기 자신을 위한 사실만.

### L2 — 활성 자산
- **Problem-Solution Catalog**: 동일 마찰 N≥2회 목격 또는 사람의 명시 승인 시 활성 승급. 6개월 미사용 시 자동 아카이브.
- **Pending Lessons**: 1회 해결된 문제 중 재사용 가능성이 있는 항목. 활성 Catalog와 분리하고, Skeleton에는 굽지 않는다.
- **Skeleton Registry**: 도메인별 부트스트랩 자산. `base/`는 사람이 최소한으로 관리하고, `skeleton.lock.json`과 적용 결과는 Catalog 엔트리 합성으로 결정적으로 빌드된다.

### L3 — 씽크탱크
- 입력: L1 관찰 + Git/PR (어느 AI/도구든 남기는 것). 트랜스크립트는 *보너스 신호*.
- 출력: Catalog 승급 후보 PR, 가이드 보강 제안, 프롬프트 개선 제안.
- **자동 적용 절대 없음**. 사람이 머지해야 시스템에 반영.

### L4 — 대시보드 + 온보딩
- 단일 페이지: NOW / NEXT / LIBRARY / THINKTANK
- `ONBOARDING.md`: 신규 사용자 15분 부팅 가이드
- 분기마다 *신규 사용자 시뮬레이션* — 시스템이 자기 자신을 검증

## 6. 자가발전 메커니즘

```
       ┌─── 작업 수행 ───┐
       │                 │
       ▼                 │
   L1 자동기록           │
       │                 │
       ▼                 │
   L3 주간 마이닝        │
       │                 │
       ▼                 │
   승급 후보 PR           │
       │                 │
       ▼ (사람 머지)      │
   L2 Catalog 갱신        │
       │                 │
       ▼                 │
  Skeleton 자동 재빌드   │
       │                 │
       ▼                 │
   다음 작업 시작 ───────┘
```

**이게 "강화학습"인가?** 아니다. 그렇게 부르지 않는다(비-목표 참조). 이건 **사람이 게이트를 가진 패턴 마이닝 + 결정적 자산 합성**이다. 이 정직한 명명이 시스템의 안전성을 보장한다.

## 7. 성공 지표 (Measurable)

"발전하고 있는가"를 *말이 아니라 숫자로* 검증한다. 대시보드 상단에 박는다.

### 7-1. 비반복 효과 (제1원칙)
| 지표 | 목표 | 의미 |
|---|---|---|
| Catalog 적중률 | 분기별 ↑ | L1 관찰의 `repeat_of` 필드 채워진 비율 |
| 동일 마찰 평균 해결 시간 | 회차별 ↓ | 1회차 N분 → 2회차 0분(자동 패치)이 이상치 |

### 7-2. 인계·이식성 효과 (제0·제2원칙)

**지표 정의**

| 지표 | 목표 | 의미 |
|---|---|---|
| 인계 성공률 | ≥ 95% | 다른 AI가 checkpoint만 보고 추가 질문 없이 진행한 비율 |
| 부팅 시간 | ≤ 60초 | 새 AI/PC에서 첫 의미 있는 출력까지 |
| 벤더 잠금 점수 (`.ai/` + 공통 CLI) | 0 | 이식성 코어와 공통 도구에 특정 AI/도구 종속 코드 없음 |
| 검증된 모델 다양성 | ≥ 3 | Catalog `verified_with`의 고유 모델 수 |

**베이스라인 측정 이력**

| 회차 | 일자 | 환경 | 부팅 시간 | 인계 성공률 | 벤더 잠금 | 모델 다양성 | 종합 |
|---|---|---|---|---|---|---|---|
| #1 | 2026-05-07 | 동일 PC, claude-sonnet-4-6 | ~30–60s (이론치) 🟡 | N/A (1회) | **0** 🟢 | 0 🔴 | 베이스라인 확립 |
| #2 | 2026-05-08 | 동일 PC, transfer drill | (미측정, v0 인프라 확인 중심) | **Pass for v0 scope** 🟢 | 0 🟢 (불변) | 0 🔴 (불변) | v0 인프라 가동 검증 |

**#2 회차 보강 증거** (`40_dev/snapshots/transfer-drill-2026-05-08.md`):
- `.ai/context.json` 의 모든 `must_read` 경로 존재 ✅
- `TODO.md` 활성 항목 0건 (Ready/InProgress/Blocked) — 인계 시점 명료 ✅
- CLI 동작 검증: `observe`, `catalog status`, `skeleton build/apply`, `thinktank`, `generate-dashboard.py` ✅

**남은 격차** (#2 회차에서 드러남):
- 다른 PC 클론 시뮬 미실시 (현재까지는 동일 PC) → 회차 #3 대상
- 인계 성공률은 *진짜 다른 모델 1회+* 실시 후 첫 비율 산출 가능
- 모델 다양성은 Stage 2의 실제 Catalog 승급 이후

### 7-3. 학습/적용 용이성 (G1)
| 지표 | 목표 | 의미 |
|---|---|---|
| 첫 작업 시작까지 | ≤ 15분 | ONBOARDING만 보고 |
| 도메인별 첫 PR까지 | ≤ 1일 | Skeleton 적용 기준 |

### 7-4. 자가발전 신호 품질 (G3)
| 지표 | 목표 | 의미 |
|---|---|---|
| 승급 PR 머지율 | ≥ 70% | 씽크탱크 제안의 사람 채택률(낮으면 시그널 노이즈) |
| 프롬프트 평균 turn 수 | ↓ | 프롬프트 디자인 개선 효과 |

## 8. 운영 원칙

### 8-1. 충돌 해결 우선순위

설계·운영 결정 시 다음 순서로 판단한다.

```
1. 이식성 (L0 무결성)
2. 비반복 (L2 정확성)
3. 인계 가능성 (온보딩 단순성)
4. 자가발전 신호 품질 (L3)
5. 자동화 깊이
```

### 8-2. 게이트 원칙

- **백서 변경**: Class C. ADR 필수, 사람의 명시적 승인 필수.
- **Catalog 승급**: Class B. 사람 머지 필수, 단 자동 PR 생성은 허용.
- **Pending Lesson 추가**: Class A. AI가 제안할 수 있으나 활성 자산으로 취급하지 않는다.
- **Skeleton 재빌드**: Class A. Catalog 변경에 따라 결정적으로 재빌드.
- **L1 관찰 로그**: 게이트 없음. 자동 기록 자체가 가치이며, 정제는 L3가 한다.

### 8-3. 자정(self-cleaning) 원칙

- Catalog: 6개월 hit 0 → 자동 아카이브
- Adapter: 해당 도구가 더 이상 쓰이지 않으면 archived 태그
- L1 관찰: 절대 삭제하지 않는다(원천 데이터). 검색·마이닝 대상에서만 노후화 가중치 적용.

### 8-4. 정직한 명명 원칙

- "강화학습", "자율", "유기적", "지능형" 등 *과장된 단어*를 시스템 컴포넌트 이름에 쓰지 않는다.
- 제안은 "제안"이라고 부른다. 추론은 "추론"이라고 부른다. 자동 적용이 없는 것을 자동이라고 부르지 않는다.

### 8-5. Guardrails-by-Construction (가드레일 우선)

품질·안전 규율은 *사후 정리*가 아니라 *발생 방지*로 강제한다 — 위반을 커밋/머지 단계에서 fail-closed로 막으면 부채가 애초에 안 쌓인다.

- **강제 위치**: 프롬프트·리뷰 구두 지적이 아니라 *시스템*(린트·CI·pre-push 게이트). 프롬프트 한 줄로 끌 수 없다.
- **장애 모드**: fail-closed(`error` → 차단)가 기본. fail-open(`warn` 방치) 금지.
- **래칫 도입**: day-1 `warn` 베이스라인 → 0 수렴 → `error` 승격(역행 불가).
- **두 영역에 동일 적용**: *AI 안전* 가드레일(지침 17 §4.2)과 *코드 품질* 가드레일(지침 19) 모두 같은 원리. 규칙 우회는 ADR 근거 필요.

## 9. 리스크와 한계 (정직 모드)

본 방법론이 *실패할 수 있는 지점*을 명시한다. 위험을 숨기는 백서는 가치가 없다.

| 리스크 | 영향 | 완화 |
|---|---|---|
| L1 자동기록 누락 | 데이터 단절, 자가발전 정지 | CI 체크: 7일 관찰 0건이면 fail |
| L3 마이닝 노이즈 | 잘못된 패턴이 Catalog 승급 | 최소 데이터 임계치(예: 100건) 전까지 인사이트 비신뢰. N≥2 승급 + 사람 게이트. |
| 게이트키퍼 적체 | 승급 PR 쌓이면 시스템 정지 | 주 1회 30분 "Catalog Review" 운영 약속 |
| 도메인 분류 부패 | 잘못된 라벨링 누적 → Skeleton 어긋남 | 분기마다 도메인 택소노미 자체 회고 |
| 체크포인트 품질 편차 | 약한 모델의 부실한 인계서 | 체크포인트 lint CI(필수 섹션 누락 시 fail) |
| 1인 사용 시 한계 효용 | 시스템 운영 비용이 작업 시간 잡아먹음 | 8주 이후 정량지표로 ROI 검증, 부족하면 축소 |
| 백서 자체의 부패 | 현실과 백서가 어긋남 | 백서 변경은 Class C(ADR + 사람 승인). 단, 분기마다 백서-현실 정렬 점검 회고 의무. |

## 10. 로드맵 (개요)

상세 일정·산출물은 `HANDOFF.md`와 `TODO.md`에서 관리한다. 백서에는 **단계 골격**만 둔다.

| 단계 | 핵심 산출물 | 종료 조건(Exit Criteria) |
|---|---|---|
| **Stage 1 — 토대** | L0 이식성 코어 + L1 관찰 로그 + 첫 인계 시뮬 | 다른 AI/PC에서 임의 TODO 1건 추가 질문 없이 이어 작업 성공 |
| **Stage 2 — 자산화** | L2 Catalog 시드 5건 + Skeleton 1개 도메인 | 새 프로젝트 1건을 Skeleton으로 부트스트랩 |
| **Stage 3 — 씽크탱크** | L3 마이닝 v0 + 첫 인사이트 리포트 | 승급 PR 1건이 사람 게이트 통과해 머지 |
| **Stage 4 — 가시성** | L4 대시보드 통합 + ONBOARDING 완성 | 신규 사용자 시뮬 15분 안에 첫 작업 시작 |
| **Stage 5 — 검증·확장** | 정량지표 측정 + 2번째 도메인 Skeleton + 2번째 AI 검증 | 7장의 모든 지표 측정 가능 + 검증 모델 다양성 ≥ 2 |

각 단계 종료 시 **사람 게이트**(휴먼 리뷰 + 명시적 다음 단계 지시)를 통과해야 다음 단계로 진입한다. 자기 자신을 자동 승급시키지 않는다.

## 11. 용어집 (Glossary)

| 용어 | 정의 |
|---|---|
| **L0~L4** | 본 방법론의 5계층 구조. 위로 갈수록 사람 친화적, 아래로 갈수록 기계 친화적. |
| **Catalog** | 재발한 문제 1건당 1엔트리로 정제된 활성 자산 라이브러리. 손으로 관리. |
| **Pending Lesson** | 1회 해결된 문제 중 재사용 가능성이 있으나 아직 활성 Catalog로 승급되지 않은 후보. |
| **Skeleton** | 도메인별 부트스트랩 자산. `base/`는 수동 관리하고, lock/apply 결과는 Catalog 엔트리 합성으로 결정적 빌드. |
| **Observation (L1)** | AI가 세션 종료 시 자동 기록하는 raw 사실 로그. |
| **Checkpoint** | 세션 인계서. 다음 사람(또는 다른 AI)이 즉시 이어받을 수 있도록 작성하는 1페이지 문서. |
| **Adapter** | 특정 AI 도구(Claude Code, Cursor, GPT 등)에 대한 보조 지침. 코어와 분리되어 격리됨. |
| **Promotion (승급)** | L1 관찰이 N≥2회 매칭되었거나 사람이 명시 승인했을 때, Pending Lesson 또는 관찰을 활성 Catalog로 올리는 행위. 사람 머지 필요. |
| **Verified vs Tentative** | Catalog 엔트리가 둘 이상의 AI 모델에서 검증되었는가의 상태. |
| **Vendor Lock Score** | `.ai/`와 공통 CLI 표면에 특정 AI/도구 종속 코드가 섞인 개수. 목표는 0. |
| **Class A/B/C** | 변경의 위험도 분류. `CLAUDE.md` §3 참조. |
| **Meta-Methodology** | 본 저장소를 진화시키는 *방법론에 대한 방법론*. `70_meta/`에 격리되며 외부 프로젝트에 주입되지 않음. §13 참조. |
| **RFC** | `70_meta/rfc/RFC-NNN_*.md` — 백서·운영 규칙 변경 *제안*. 머지 시 ADR로 승급. |
| **Meta Catalog (MC-NNN)** | `70_meta/catalog/` — 방법론 운영 자체의 반복 마찰. 도메인 Catalog(`C-NNN`)와 분리. 주입 안 됨. |

## 12. 백서 변경 절차

본 백서는 시스템의 헌법이다. 가볍게 바꾸지 않는다.

1. 변경 제안은 **ADR**로 시작한다(`40_dev/adr/`).
2. ADR은 다음을 포함한다: 변경 사유, 영향 범위, 어느 원칙·목표·비-목표가 어떻게 바뀌는가, 롤백 시나리오.
3. 사람의 명시적 승인(merged PR 또는 linked issue 승인)이 있을 때만 백서를 수정한다.
4. 수정 시 `version`을 의미적으로 올린다(MAJOR.MINOR.PATCH).
   - **MAJOR**: 원칙(§2)·비-목표(§4) 변경
   - **MINOR**: 목표(§3)·구조(§5)·메커니즘(§6) 변경
   - **PATCH**: 명확화·오타·예시 추가
5. `supersedes` 필드에 이전 버전 doc_id를 기록한다.

## 13. 메타-방법론 (Meta-Methodology)

본 저장소는 **두 역할을 동시에** 갖는 *재귀 구조* 다:

1. **외부 프로젝트들이 적용하는 방법론의 원천** — `methodology init` / `sync` 로 외부에 주입되는 모든 자산
2. **자기 자신을 진화시키는 메타-방법론의 운영 공간** — `70_meta/` (외부 주입 안 됨)

이 분리는 *주입 안전성*과 *원천성 보존*을 위한 결정적 보장이다.
방법론 자체의 운영에서 발생하는 마찰·결정·실험이 *외부 프로젝트의 도메인 자산*과 섞이지 않도록 한다.

### 13-1. 격리 보장 (이중 안전망)

| 계층 | 메커니즘 |
|---|---|
| **1차 (whitelist)** | `60_tools/methodology.py` MANIFEST는 *명시된 경로만* 복사. `70_meta/`가 어디에도 명시되지 않으면 *애초에* 복사 안 됨. |
| **2차 (excluded_paths)** | MANIFEST의 `excluded_paths: ["70_meta"]`. init/sync 시작 시 *겹침 검증* 통과해야 진행. 실수로 다른 카테고리에 추가해도 차단됨. |

검증 명령:
```bash
python3 60_tools/methodology.py manifest-check
```

CI에서 본 검증이 실패하면 **반드시** 배포 차단 — 실패는 격리 깨짐을 의미.

### 13-2. `70_meta/` 내부 구조

```
70_meta/
├── _README.md          ← 메타-방법론 운영 규칙 (단일 출처)
├── rfc/                ← 방법론 변경 제안 (RFC-NNN)
├── retrospectives/     ← 분기 회고 (YYYY-QN)
├── experiments/        ← 실험 (EXP-NNN)
├── observations/       ← 본 저장소 운영 관찰 (메타-L1)
└── catalog/            ← 메타-Catalog (MC-NNN)
    ├── _pending/
    └── archived/
```

상세 규칙은 [`70_meta/_README.md`](../70_meta/_README.md).

### 13-3. 도메인 자산 vs 메타 자산 판정

판정 기준 한 줄: **"외부 프로젝트가 이 자산의 혜택을 받는가?"**

| Yes (도메인) | No (메타) |
|---|---|
| `50_resources/catalog/` (`C-NNN`) | `70_meta/catalog/` (`MC-NNN`) |
| `50_resources/ai_observations/` | `70_meta/observations/` |
| `50_resources/skeletons/` | (해당 없음 — 스켈레톤은 본질적으로 도메인 자산) |
| `40_dev/adr/` (방법론 적용 결정) | `70_meta/rfc/` (방법론 자체 변경 제안) |

분류 실수는 분기 회고(`70_meta/retrospectives/`)에서 점검.

### 13-4. 본 §의 변경

본 §은 백서 §12 변경 절차를 따른다. *메타-방법론의 격리 자체* 를 깨는 변경은 **Class C** (이식성·원천성에 영향).

### C.2 끝번호 페어링 규칙

`20_guides/`의 일의 자리와 `30_planning/`의 일의 자리는 **동일 도메인을 의미한다**.
