# 00_briefs/ — 인간 입력 공간

> **사용자가 raw 메모·기획·리서치를 던지면, AI 가 매 세션·필요 시 다시 읽고 반영한다.**
> 위상: 모든 작업의 *출발점*. 백서 §3-G1 단일 진입점.

---

## 1. 무엇을 넣는가

| 종류 | 예시 |
|---|---|
| **아이디어 노트** | "사용자 onboarding 단순화 해보면 어떨까" |
| **리서치 결과** | 시장 분석·경쟁사 조사·기술 문서 요약 |
| **회의록** | 사용자 인터뷰·내부 회의 내용 |
| **참고 자료 링크** | URL·PDF 경로 (raw 인용 OK) |
| **방향성 선언** | "이번 분기는 X 에 집중" |

→ *정형 산출물* 은 아님. **raw·자유 형식 OK**. AI 가 읽고 *기획서·개발 산출물* 로 변환.

## 2. 어디에 넣는가

```
00_briefs/
├── _README.md              ← 본 문서
├── standing/               ← 상시 SOP (날짜 없음·아카이브 안 됨·boot ★ 최상단)
│   └── SOP_<topic>.md          반복·정기 작업 절차
├── meetings/               ← 회의록·인터뷰
├── research/               ← 리서치·조사·분석 (시장/유저/기술 — 내가 조사했거나 조사해달라 한 것)
├── reference/              ← 참고 자료 (외부 문서·링크·경쟁사 캡처·벤치마크 원본·스펙 발췌)
├── ideas/                  ← 아이디어·방향·전략 선언 (유형 불분명 시 기본)
└── archived/               ← 옛 브리프 (유형 불문, 참고 보관)
```

**파일명**: 유형 폴더 = `YYYY-MM-DD_<topic-slug>.md`(시간순). **`standing/` = 날짜 없음** `SOP_<topic>.md`.

### 자동 분류 (AI가 넣을 때 폴더를 고른다)
> 사용자가 브리프를 던지면 **AI가 내용을 보고 유형을 판별해 해당 폴더에 배치**한다(파일명 컨벤션 적용). 문서 채널 분리(guide 05)와 같은 원리.

| 내용이 이러면 | 폴더 | 신호 예시 |
|---|---|---|
| 회의·인터뷰 논의·결정 기록 | `meetings/` | "회의했는데", 참석자·결정사항 나열 |
| 직접 조사·분석·데이터·시사점 | `research/` | 시장/경쟁/유저 조사, 수치·인사이트 |
| 외부 원본을 그대로 참고 | `reference/` | 링크·PDF·경쟁사 화면·타사 스펙·기사 |
| 반복·정기 작업의 *절차* | `standing/` | "이건 반복 작업", "~할 때마다" (→ SOP) |
| 아이디어·방향·전략 선언 | `ideas/` | "이거 해보면", "이번 분기 X 집중" |
| **애매하면** | 사용자에게 유형 확인 → 그래도 불명확 시 `ideas/` | — |

> research vs reference 구분: **내가 소화·정리한 것**=research / **외부 원본을 날것으로 참조**=reference. 큰 파일·다수 링크는 reference, 요약·판단이 들어가면 research.

### `standing/` — 반복 작업이 잊히지 않게 하는 곳
> **문제**: 정기·반복 작업(예: 업무기술서 처리·주간 리포트·배포 절차)의 *방법*이 관찰 로그·커밋에만 흩어져 있으면, 새 세션이 능동적으로 캐야만 알고 — 못 캐면 프로세스를 모른 채 시작한다(실제 사고 사례).
> **해결**: 그 절차를 `standing/SOP_<topic>.md` 한 장으로 박제한다. 날짜가 없어 *아카이브되지 않고*, `boot` 이 **항상 최상단에 노출**한다. 반복 작업을 시작하기 전 이 파일부터 확인.
> **무엇을 담나**: 트리거(**인식 신호**=어떤 요청/말이 이 작업을 의미하는가 + 주기/이벤트)·입력·단계별 절차·산출물 위치·주의점. 인식 신호가 명확할수록 새 세션이 "이건 그 반복작업"이라고 정확히 매칭한다. 절차가 바뀌면 이 파일을 갱신(SSOT).

## 3. AI 가 언제 읽는가

| 시점 | 동작 |
|---|---|
| **매 세션 시작** | `boot` 이 활성 유형 폴더 전부를 그룹별로 나열(standing ★ 최상단·archived 제외) → 본문 전부 읽음 |
| **사용자 요청 시** | "브리프 다시 봐줘" → AI 가 활성 브리프 재로드 |
| **자동 트리거 (향후)** | brief 파일 mtime 변경 감지 → 다음 세션에 highlight |

## 4. 갱신 패턴

- 인간이 *수시로* `current/` 에 새 파일 추가 또는 기존 파일 수정
- 일정 기간 후 (인간 판단) `archived/` 로 이동
- *삭제 금지* — 옛 맥락도 학습 데이터
- **`standing/` 은 아카이브하지 않는다** — 반복 작업이 살아 있는 한 유지. 절차가 바뀌면 *덮어써* 갱신(AI 도 절차 변경을 감지하면 갱신 제안)

## 5. AI 측 규칙 (CLAUDE.md / AGENTS.md 반영)

- **세션 부팅 시** `boot` 이 `standing/*`(항상 먼저) + `current/*.md`(날짜순) 나열 → 본문 *전부 읽음*
- 작업 진행 중 *그 브리프 내용을 어떻게 반영했는지* 명시 (예: "TALMOCOM-042 는 2026-05-14_onboarding.md 의 §3 반영")
- *옛 브리프와 충돌* 발생 시 사용자에게 확인 — 자동 결정 금지
- **반복 작업 기억 요청 시** ("기억해줘/이건 반복 작업이야") → `standing/SOP_<topic>.md` 작성(`SOP_template.md` 형식). 절차 변경 감지 시 갱신 제안

## 6. 안티패턴

- ❌ 정형 산출물을 brief 에 넣음 — `30_planning/` 또는 `40_dev/` 로
- ❌ 비밀번호·API 키 — `.env` 또는 secret manager 사용
- ❌ 너무 길게 — 1 파일 200줄 이내 권장 (긴 리서치는 `40_dev/snapshots/` 또는 외부 링크)
- ❌ archived 삭제 — 학습 데이터 손실

## 7. 첫 시드

신규 프로젝트면 `current/` 비어있음. 사용자가 첫 브리프 1장 던지면 AI 가 그걸 baseline 으로 작업 시작.
