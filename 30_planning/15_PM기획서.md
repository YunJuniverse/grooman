# grooman 프로젝트 관리 기획서

> 작성 기준: `20_guides/15_프로젝트_관리_기획서_작성_지침.md` §8.1 + §6.16~6.24.
> **위상**: grooman의 PM 시스템은 새로 만들지 않는다 — **방법론 라이브 체계(CLAUDE.md·HANDOFF·TODO·checkpoint·wrap/ship·Change Class·PR 게이트)가 이미 PM 시스템**이다(지침 §6.24의 살아있는 사례). 본 문서는 그 체계를 grooman에 **인스턴스화**하고, 방법론이 커버하지 않는 프로젝트 고유 층(OKR·게이트 카탈로그·RAID·프리모템·인간 액션 레지스트리)만 원본으로 소유한다. 중복 정의 금지 — 정본이 있으면 인덱스.

## 0. 딜리버리 모델 선언 (§6.18)

- **하이브리드**: Phase(P0~P3) 마일스톤은 *예측형*(조건 기반 게이트), 태스크 실행은 *플로우형*(TODO 칸반·WIP 캡 3 — 방법론 표준).
- **근거·테일러링**: 1인+AI 사이드 프로젝트 — ① 캘린더 데드라인은 허구가 되기 쉬우므로 **날짜 약속 대신 조건 게이트**로 관리 ② 실행 주체가 AI 에이전트라 배치·세션 단위 흐름이 자연 단위 ③ Monte Carlo 확률 예측(§6.19)은 throughput 이력 축적 전이라 **미적용을 명시**(데이터 쌓이면 도입 검토).

## 1. 프로젝트 개요

- **한 줄 정의**: 이미 구현된 grooman(남성 그루밍 커뮤니티)을 방법론 거버넌스 아래서 **공개(P1) → 전환 검증(P2) → 수익 다변화(P3)**까지 끌고 가는 프로젝트.
- **유형**: retrofit 운영·성장형 (신규 구축 아님 — P0 구축은 완료).
- **문서 범위**: 프로젝트 관리 구조·통제 기준. 개별 기획 원본(10~14)·개발 순서(18 마스터플랜, 미작성)는 각자 소유.

## 2. 목표·완료 기준 + OKR (§6.21)

**Objective**: "검색으로 발견되는 중립적 남성 그루밍 커뮤니티"가 실제로 작동함을 증명한다.

| KR (아웃컴 — 산출물 아님) | Phase | 수치 정본 |
|---|---|---|
| KR1 검색 유입이 생긴다 — 색인 500+·MAU 5천 | P1 | 사업 §9·마케팅 §9 |
| KR2 유입이 기여로 전환된다 — 가입 전환 1%·첫 기여 20% | P2 | 마케팅 §3 |
| KR3 커뮤니티가 자생한다 — 월 재방문 30%·UGC 비율 상승 추세 | P2~3 | 마케팅 §9 |

- **완료 기준(프로젝트 종료 아닌 검증 종료)**: KR2 판정 데이터 확보 시점 — 성공이든 실패든 *판정 가능*해지면 이 프로젝트(검증)는 완료, 이후는 운영·확장.
- **feature factory 가드**: 새 기능 TODO는 반드시 KR에 링크 — 링크 안 되면 비목표 재검토(서비스기획서 §8).

## 3. 범위·단계 (조건 게이트)

| Phase | 목표 | 진입 조건(게이트) | 상태 |
|---|---|---|---|
| P0 구축 | 제품+방법론+기획서 체계 | — | ✅ 완료(기획 시리즈 6/8) |
| **P1 공개·색인** | KR1 | **선행 4건**: GRM-012(신고·제재)·GRM-013(측정)·법률 검토 2건(임시조치·clinic)·릴리스 게이트 SOP(봇 0건) + **Class C 공개 승인** | 대기 |
| P2 전환 검증 | KR2 | P1 KR1 달성 + 색인 안정 | — |
| P3 수익 다변화 | 제휴·후원 | KR2 검증 + 유료 3조건(마케팅 §7) + Class C(예산) | — |

- 제외 범위: 서비스기획서 §8 비목표 6건이 정본(중복 기재 안 함).

[GRM-013 측정 인프라] ─┼→ [법률 검토 2건] ┼→ [릴리스 게이트(봇 0)] → [P1 공개] → [색인 스프린트] → KR1 → [P2]

## 7. 이슈·리스크 — RAID (§6.23, 통합 등록부)

- **R**isks: 사업기획서 §13 R1~R5 정본 + AI 위험 등록부(§10) — 여기 중복 안 함.
- **A**ssumptions (조용히 깨지는 전제 — 분기 리뷰): ① RPM ₩2천+ ② 자동수집이 색인 페널티 안 받음 ③ 사람 승인 SLA 72h 유지 가능 ④ Supabase/Vercel 무료·저가 티어로 P2까지 충분 ⑤ 1인 운영 시간 주 N시간 확보 [미확정 — 수치는 사람만 앎].
- **I**ssues: `HANDOFF.md` Open Issues가 **정본**(현재 G1+G4·G5·SEC-1·CRAWL-1) — 오너·다음 스텝 포함, 세션마다 갱신.
- **D**ependencies: 인간 액션 레지스트리(§4) + 외부(Supabase·Vercel·Anthropic API·AdSense 정책·검색엔진 알고리즘).

**프리모템 (§6.23 — 실행 완료, 결과 박제)**: "P2 말, 프로젝트가 실패했다. 왜?"

| 실패 시나리오(역산) | 기존 리스크 커버? | 보강 액션 |
|---|---|---|
| 유입은 있는데 가입 0.1% — 자동수집 사이트로 인식됨 | R1 ✓ | UGC 병치·전환 지표 조기 측정(G5가 전제 — 이래서 P1 필수) |
| 자동수집 대량 색인 → 구글 스팸 판정 → 도메인 페널티 | R2 ✓ | 비중 상한 50% 준수 + Search Console 수동 점검(주간) |
| clinic 글로 의료광고법 신고 접수 | R2 유사, **커버 약함** | **법률 검토를 크리티컬 패스로 승격(§5)** — 이 프리모템의 최대 수확 |
| 운영자 3개월 방치 → 크롤만 도는 유령 사이트 | R5 ✓ | 가정 ⑤를 분기 리뷰 — 방치 감지 시 크롤 중단 규칙(운영 §13) |
| AI 에이전트 대량 작업 중 회귀(빌드 깨짐·데이터 오염) | **신규** | §9 에이전트 품질 가드(빌드·wrap 게이트) + Class B 롤백 규칙 준수 |

## 8. 의사결정·산출물·변경 (전부 인스턴스 — 정본 인덱스)

- **산출물**: 기획 시리즈 상태=TODO GRM-011 체크리스트 · 자식 산출물 인덱스=서비스기획서 §4 정본 · 버전=git+각 문서 Change Log(Trigger 포함).
- **변경 통제**: `CLAUDE.md §3 Change Class`(A/B/C)가 정본 — 범위 변경은 TODO 스코프 수정 + 서사(checkpoint) 기록. 문서→코드 순서(운영 §10) 준수.
