# grooman 서비스기획서

> 작성 기준: `20_guides/11_서비스기획서_작성_지침.md` §8.1 — **부모=오케스트레이팅 인덱스**(§2.2). 이 문서는 *의도·결정·경계*만 소유하고, 상세(IA·기능·정책·화면)는 자식 산출물이 원본이다 — 여기에 전수 복사하지 않는다(단일출처).
> 상위 입력: `10_사업기획서.md` v1(BM·Phase) · `14_브랜드기획서.md` v1(가치·톤·Surface).
> **retrofit 주의**: 제품이 먼저 구현됐다(자율빌드). 본 문서는 구현을 역문서화하며 *의도를 정본화*한다 — 구현 상태를 사실로 표기하고, 코드와 문서가 다르면 코드가 우선(발견 시 본 문서를 갱신).

## 1. 서비스 개요 [원본]

- **한 줄 정의**: 그루밍이 궁금한 남성이 검색으로 진입해, 광고 아닌 실경험 정보를 얻고, 자신의 경험을 남기게 되는 개방형 커뮤니티.
- **목적(정성)**: 남성 그루밍 정보 탐색의 공백(사업기획서 §1)을 "개방형 + 중립 + 실경험"으로 메운다.
- **기대효과(정량, §19.12 이항 분리)**: P1 MAU 5천 → P2 MAU 3만·검색유입→가입 전환 1%+ (사업기획서 §9 로드맵의 서비스 측 지표).
- **해결하는 문제**: ① 검색해도 광고 블로그뿐 ② 남성 맥락의 종합 정보처 부재 ③ 물어볼 커뮤니티의 신뢰 부족.
- **이번 문서 범위**: 서비스 전 영역의 의도·결정·경계 + 자식 산출물 인덱스. 운영 절차(검수·제재·CS)는 12 운영기획서, 채널·캠페인은 13 마케팅기획서.

## 2. 사용자 정의 [원본] (§19.8 피라미드 90/9/1)

| 계층 | 비율[가정] | 정의 | 목적 | 핵심 기대 액션 |
|---|---|---|---|---|
| **탐색형** | ~90% | 검색 유입 비로그인 방문자 | 특정 고민의 답 찾기 | 글 열람 → (재방문) → 가입 전환 |
| **기여형** | ~9% | 가입 회원 | 질문·후기 작성, 반응 | 글/댓글 작성, 추천, 북마크 |
| **코어** | ~1% | 헤비 기여자 (레벨 상위) | 정체성·평판 | 고품질 후기 축적, 커뮤니티 톤 형성 |
| 관리자 | — | 운영자(현재 1인) | 품질·수익 관리 | 크롤 소스 관리, 콘텐츠·사용자·광고 관리 |
| (시스템) | — | 자동수집 게시자(user_id=null) · 테스트 봇(배포 전 삭제, ADR-0002) | 콘텐츠 볼륨 | — 사용자 아님, 권한 없음 |

- **기능 우선순위 원칙**: 다수(탐색형)의 *읽기 경험* > 소수(기여형)의 *쓰기 경험* > 거버넌스. P1은 읽기, P2는 쓰기 전환에 집중(§8).

## 3. 사용자 시나리오 — 상위 서사 [원본]

핵심 3개 + 이탈 1개. 카드 상세(Gherkin AC)는 `user-story.md`로(§4 인덱스).

- **S1 진입(탐색형)**: "미녹시딜 후기" 검색 → grooman 글 랜딩(비로그인 열람) → 관련글·HOT보드로 2~3글 연속 소비 → 브라우저 이탈 or 북마크 시도 → 가입 유도. *종료 조건: 재방문 쿠키/가입.*
- **S2 전환(탐색형→기여형)**: 재방문 n회 → 댓글 작성 욕구 발생 → 카카오/구글 1클릭 가입 → 첫 댓글 → 알림(답글)으로 재방문 루프 형성. *이 전환이 사업 최대 미검증 가설(사업기획서 R1).*
- **S3 기여(기여형)**: 자기 고민 검색 → 유사 글 발견 못함 → 글 작성(Tiptap·이미지) → 추천/댓글 반응 → 레벨·포인트 상승 → 기여 반복.
- **S-X 이탈 리스크**: 랜딩 글이 자동수집 요약뿐이라 얕음 → "광고 블로그와 다를 게 없네" → 즉시 이탈. *대응 결정: 자동수집 글에 관련 UGC·댓글 유도 모듈을 병치(P2), 자동수집 비중 상한 [미확정 — 12 운영기획서에서 결정].*

## 6. IA — 개념·탐색 논리 [인덱스 → `ia-spec.md`]

- **왜 이 구조인가**: 최상위 분류 = 고민 도메인 6종(hair/skin/shaving/fragrance/clinic/deals) — 남성의 탐색은 "고민 단위"로 시작한다는 가설. 기능(글쓰기·검색·마이)은 GNB 유틸로 분리.
- **탐색 논리**: 진입은 *검색(외부)* 이 정문, 카테고리 브라우징은 후문 — IA는 랜딩 글에서의 *횡이동*(관련글·태그·HOT)을 우선 설계.
- 라벨링 결정(§19.9): 게시판명은 한국어 보통명사(브랜드기획서 §6 네이밍 규칙) — "핫딜·이벤트"(← "deals"), "시술·성형 후기"(← "clinic"). 전체 라벨 사전은 `ia-spec.md`로.
- 화면 ID·depth·전체 메뉴트리·권한 매트릭스 → `ia-spec.md` (미착수, §4).

## 7. 정책 — 의도·목록 [인덱스 → `service-policy.md`]

정책 상세(조건문·ON/OFF 2-시트)는 `service-policy.md`가 원본. 여기엔 **정책 영역 목록 + 이미 코드에 내린 핵심 결정**만:

| 정책 영역 | 코드에 구현된 핵심 결정 (역문서화) |
|---|---|
| 가입·인증 | OAuth(카카오·구글)+이메일. username 2~20자 UNIQUE·트리거 자동 생성 |
| 작성·수정·삭제 | 본인만 수정·삭제(RLS `auth.uid()=user_id`). 삭제 = posts.status enum(soft) — **이력 보존 원칙(§19.3)**: 계정 탈퇴 시 글·댓글은 `SET NULL`로 익명 보존(ADR-0003) |
| 공개 범위 | published 전체 공개(비로그인 열람) · draft 본인만(RLS 검증됨) · hidden/deleted 관리자 |
| 자동수집 | clinic 금지(AI enum 차단)·출처/뱃지 의무·spam_score 0.7↑ 폐기 (ADR-0001) |
| 레벨·포인트 | 레벨 1~5·포인트 적립(글·댓글·추천 트리거) — *승급 임계·보상표는 service-policy로* [미확정] |
| 신고·제재 | reports 스키마만 존재(사유 enum·reporter RLS). **신고 접수 후 처리 기준 없음** → 제재 단계·기준은 12 운영기획서 원본, 서비스 측 UI는 F7 갭 해소 시 |
| 광고 | 광고 지면과 콘텐츠 구분 표기(브랜드 §11 Surface 매트릭스 준수) |

## 8. MVP 범위 + 비목표 [원본·결정]

**MVP = 현재 구현 범위(F1~F6)** — retrofit이므로 MVP는 계획이 아니라 사실. 단 **공개 전 필수 갭**을 명시:

| 갭 | 왜 MVP에 필요한가 | Phase |
|---|---|---|
| G1 **신고 UI** (F7) | UGC 커뮤니티를 신고 수단 없이 공개하면 자정 불가 + 정보통신망법상 임시조치 요청 창구 필요 | **P1 필수 (2026-07-22 사람 결정)** → GRM-012. 처리 기준은 12 운영기획서 §4.1 |
| G2 봇 teardown 실행 | 릴리스 게이트(SOP·ADR-0002) | P1 공개 직전 (수단 구현 완료) |
| G3 크롤 소스 실소스 교체·저작권 검토 | 현재 example 플레이스홀더 소스 | P1 공개 전 필수 |

**비목표(Non-Goals) — 제외 사유 + 진입 조건 게이트(§19.9)**:

| 하지 않을 것 | 제외 사유 | 후속 진입 조건(게이트) |
|---|---|---|
| 쪽지/DM | 1:1 채널은 스팸·분쟁 벡터, 운영 비용 大 | 코어 사용자 요구 반복 + 운영 인력 확보 시 |
| 네이티브 앱 | 웹 SEO가 유입 전략의 전부(사업 §3) | P3 재방문율 30%+ 달성 시 재검토 |
| 자체 커머스·구매 | 중립성(브랜드 가치 1)과 직접 충돌 | 안 함 — 제휴 링크(P2)까지만 |
| 실시간 채팅·라이브 | 아카이브형(검색 자산) 콘텐츠와 상충 | 안 함 |
| 전문가(의사) 인증 답변 | 의료광고·책임 문제, clinic은 UGC 후기로 한정 | 법률 검토 + 파트너십 확보 시에만 |
| 여성 카테고리 | 포지셔닝 붕괴(브랜드 §3) | 안 함 |
