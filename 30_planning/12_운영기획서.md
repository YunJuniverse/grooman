# grooman 운영기획서

> 작성 기준: `20_guides/12_운영기획서_작성_지침.md` §8.1 — 단, **1인+AI 운영 현실에 맞춰 슬림화**(가짜 조직도 금지: 다인 역할 체계를 흉내내지 않고 겸직·자동화로 정직하게 매핑). 기준은 전부 **조건형**으로 쓴다("필요 시 제한" 금지).
> 상위 입력: `11_서비스기획서.md` §10이 넘긴 항목(신고 처리·제재·자동수집 상한·clinic 운영·알람 임계) + **G1 신고 UI = P1 필수(2026-07-22 사람 결정)**.
> 릴리스 게이트의 정본은 `00_briefs/standing/SOP_public-release-gate.md` — 본 문서는 인덱스만.

## 1. 운영 개요

- **목적**: 1인 운영자가 커뮤니티 품질(중립·신뢰)과 시스템 안정을 *지속 가능하게* 유지한다.
- **운영 3원칙**: ① **자동화 우선** — 사람 개입은 판단이 필요한 곳에만(Toil 캡: 일일 루틴 ≤ 15분 목표, §12) ② **조건형 기준** — 모든 판단 기준은 조건→조치로 명문화 ③ **이력 보존** — 모든 운영 조치는 상태 변경(soft)으로, 파괴적 삭제 금지(서비스기획서 §7 이력 보존 원칙).
- **범위**: 콘텐츠·회원·신고·자동수집·광고·시스템 운영. 기능 정의는 서비스기획서, 채널·캠페인은 마케팅기획서 원본.

## 2. 운영 조직·역할 (1인+AI 정직 매핑)

| 역할(지침 표준) | grooman 실제 | 비고 |
|---|---|---|
| 운영자·CS·검수·IC(인시던트 지휘) | **운영자 1인(hayden) 겸직** | 에스컬레이션 대상 = 본인. "상위 보고"가 없으므로 대신 **조건형 기준을 미리 문서화**해 판단 일관성 확보(본 문서의 존재 이유) |
| 자동 검수 1차 | **AI 파이프라인** (spam_score·is_advertisement·clinic 차단) | 사람은 2차 — 사후 검토(§10 AI 운영) |
| 외부 에스컬레이션 | 법률 자문(의료광고·임시조치 분쟁 시) · Supabase/Vercel 지원(플랫폼 장애) | 트리거를 §7에 조건으로 명시 |
| (금지) | 봇 계정의 운영 활용 | 봇=테스트 픽스처, 배포 전 전량 삭제(ADR-0002). 운영 단계에서 봇으로 활동 위장 금지 |

## 3. 핵심 운영 대상

| 객체 | 주요 상태값 | 개입 필요 상황 |
|---|---|---|
| 게시글(UGC) | published / hidden / deleted(soft) | 신고 접수·금칙 위반·법적 요청 |
| 게시글(자동수집) | 동일 + is_auto_crawled | 오분류·저품질·출처 문제·원저작자 요청 |
| 댓글 | 활성 / 삭제 | 신고·금칙 위반 |
| 회원 | 일반 / 관리자 / **(정지 상태 없음 — G4 갭)** | 반복 위반·스팸 계정 |
| 신고(reports) | 접수 / 처리중 / 완료 [DB enum 확인 필요] | 전 건 — 사람 판단 대상 |
| 크롤 소스·큐 | active/inactive · done/failed | 실패 누적·품질 저하·저작권 요청 |
| 광고(ads) | active/inactive | 게재 요청·부적합 광고 |

## 4. 핵심 운영 프로세스

### 4.1 신고 처리 (P1 필수 — G1 결정 반영)

> 서비스 측 신고 UI(G1)와 관리자 측 처리 큐가 세트. 구현 범위는 GRM-012.

1. **접수**: 사용자가 글/댓글에서 신고(사유 선택: 스팸·광고 / 욕설·비하 / 의료법 위반 소지 / 저작권 / 기타) → reports 기록.
2. **분류(운영자, SLA 24h 내 1차 확인)**: 명백 위반 → 즉시 조치 / 판단 필요 → 보류 큐 / 허위·중복 신고 → 기각.
3. **조치**: §5 제재 기준 적용(콘텐츠 숨김 → 작성자 조치). 조치·사유를 기록.
4. **통보**: 신고자에게 처리 결과 표시 [P2 — P1은 상태 표시만], 피신고자에게 사유 통보.
5. **권리침해 신고(명예훼손·저작권)**: 정보통신망법 임시조치 — 요청 접수 시 **즉시 숨김(30일)** → 이의 없으면 유지, 이의 시 재게시 판단. *[미확정 — 임시조치 절차 상세는 법률 검토 후 확정]*

### 4.2 자동수집 운영

- **소스 추가**: 후보 RSS → ① 이용약관·저작권 확인(전문 전재 금지 소스 배제) ② 3일 시험 수집(비공개 검토) ③ 품질 통과 시 활성화. **소스별 저작권 검토 기록 의무**(사업기획서 R2).
- **일일 확인(자동화 목표)**: crawl 실행 결과(processed/skipped/failed) 확인 — failed 연속 3회 소스는 비활성화 후 원인 확인.
- **비중 상한 (확정 2026-07-22)**: 카테고리 피드 첫 화면(최신 20건) 중 자동수집 글 **≤ 50%** — 초과 시 크롤 볼륨 하향. UGC 성장에 따라 P2에 30%로 조임.
- **원저작자 삭제 요청**: 확인 즉시 해당 글 deleted 처리 + 해당 소스 재검토. SLA 24h.

### 4.3 clinic(시술·성형) 카테고리 운영 — 최고 민감 표면

- 허용: **본인 경험담**(비용·경과·부작용 포함 가능), 정보 질문.
- 조치 대상(조건형): ① 특정 병원·의사 연락처/예약링크/가격 이벤트 포함 → **홍보성 — 숨김** ② 전후사진 중심 후기 → 의료광고법 리스크 — 숨김 후 재작성 안내 ③ "진단·처방" 성 조언 댓글 → 숨김 ④ 병원 관계자 추정 반복 게시 → 스팸 계정 처리(§5).
- *[미확정 — 의료광고법 경계(환자 후기 허용 범위)는 공개 전 법률 검토 필수. 그 전까지 위 기준으로 보수 운영]*

### 4.4 봇·릴리스 운영 (인덱스)

- 공개 배포 게이트: `SOP_public-release-gate.md` 정본(봇 0건 검증·teardown 절차). 본 문서는 중복 기술하지 않음.

## 5. 운영 정책 — 제재 기준 (조건형)

**콘텐츠 조치**: 금칙어lexicon(브랜드기획서 §6)·스팸·홍보 위장 → `hidden` (사유 기록). 명백 불법(불법의약품 거래 등) → `deleted` + 계정 조치.

**계정 제재 단계**:

| 단계 | 조건 | 조치 |
|---|---|---|
| 1차 | 콘텐츠 조치 1회 | 경고 통보 |
| 2차 | 30일 내 재위반 | **작성 정지 7일** |
| 3차 | 정지 후 재위반 | 작성 정지 30일 |
| 즉시 영구 | 업체 침투(홍보 계정)·스팸 봇·불법 콘텐츠 | 계정 정지 + 콘텐츠 일괄 숨김 |
| 복구 | 이의신청 시 | 운영자 재검토 1회, 결과 기록 |

> ⚠️ **G4 갭**: 현재 "작성 정지"를 집행할 수단이 없다 — `profiles`에 정지 상태 컬럼·RLS 반영 없음, 관리자 액션은 글 숨김/삭제뿐(실측). **신고 UI(G1)와 세트로 P1 구현 필요** → GRM-012 범위.

## 6. CS 정책 (1인 슬림)

- **채널**: 이메일 단일 [미확정 — 주소 확정 필요, 예: contact@grooman.kr]. 실시간 채팅·전화 안 함(비목표 정합).
- **SLA**: 1차 응답 **48h 내** (1인 현실 — 과속 약속 금지). 유형: 계정 문의 / 신고·권리침해(→§4.1로 이관) / 광고 게재 문의 / 기타.
- **응대 톤**: 브랜드기획서 §11 Surface(시스템 메시지 — 격식 3, 사과는 구체적으로).

## 7. 이슈 대응 — 심각도·인시던트 (1인 경량 SEV)

| SEV | 정의 (grooman 기준) | 대응 | 목표 |
|---|---|---|---|
| SEV1 | 서비스 전면 다운 · 데이터 유출 의심 · RLS 우회 발견 | 즉시 — 필요 시 서비스 비공개 전환. 유출 시 KISA·이용자 통지 의무 확인 | 즉시 착수 |
| SEV2 | 핵심 기능 불능(로그인·글쓰기) · 법적 요청(임시조치) · 크롤이 대량 오게시 | 당일 완화 — 우회로·숨김 처리 | 24h 완화 |
| SEV3 | 부분 기능 저하 · 크롤 실패 누적 · 오분류 발견 | 주간 루틴에서 처리 | 7일 |
| SEV4 | 사소한 버그·개선 요청 | TODO 백로그로 | 백로그 |

- **1인 IC**: 지휘=운영자. 외부 에스컬레이션 트리거 — 플랫폼 장애→Supabase/Vercel 상태 페이지·지원, 법적 분쟁→법률 자문, 개인정보 사고→KISA 신고 절차.
- **포스트모템(경량·블레임리스)**: SEV1~2 해소 후 **방법론 관찰 로그(`methodology.py observe --friction`)로 기록** — 타임라인·원인·재발 방지. 별도 문서 체계를 만들지 않고 기존 학습 루프에 통합.

| 지표 | 목표 | 측정 |

- Error Budget 정책(경량): 주간 루틴에서 SEV2+ 재발 2회면 신규 작업 중단하고 원인 해소 우선.

## 10. AI 프로덕션 운영 (§6.22 — 크롤 파이프라인)

- **품질 모니터링**: 사후 전수 검토(일 ~30건 규모라 가능) — 오분류·저품질은 숨김+골든셋 후보로 축적(17 평가·가드레일 연계, 16번 작성 시 정식화).
- **비용 캡·알람 (확정 2026-07-22)**: 월 토큰 비용 **₩10,000 초과 → 원인 조사**, **₩30,000 초과 → 크롤 일시 중단**(사용자 영향 없음 — 자동수집만 멈춤). 근거: 정상 운영비 ~₩3천/월의 3배·10배 이상치.
- **Provider 장애**: 크롤은 비동기·비대면 — 장애 시 **skip하고 다음 주기 재시도**(fallback 모델 불필요, 멱등키=url_hash 구현됨). 모델 deprecation 시 프롬프트는 provider 중립(사업기획서 §15) — 모델명 교체로 대응.
- **프롬프트 인젝션**: 크롤 원문(외부 입력)이 프롬프트에 들어감 — 출력이 JSON 스키마 검증을 통과 못 하면 폐기(구현: jsonMatch 파싱 실패 시 null). 고위험 도구 호출 없음(분류·요약만).

## 11. 변경·릴리스 운영 (인덱스)

- 변경 등급·게이트 = 방법론 `CLAUDE.md §3 Change Class`(A/B/C)가 정본 — 운영 문서에서 중복 정의하지 않음.
- 공개 릴리스 = `SOP_public-release-gate.md` + Class C 사람 승인.
- 롤백: DB 변경은 마이그레이션 롤백 절 필수(Class B 규칙), 배포는 Vercel 이전 배포 복원.

## 12. 운영 매뉴얼 초안 (일일/주간 루틴)

- **일일 (≤ 15분, Toil 캡)**: ① 신고 큐 확인·조치 ② 크롤 결과(failed) 확인 ③ 신규 자동수집 글 훑기(clinic 유출·저품질).
- **주간**: ① 크롤 성공률·토큰 비용 확인 ② SoS·유입 지표(브랜드기획서 §12) ③ 소스 품질 재평가 ④ SEV 재발 확인.
- **월간**: 지표 리포트(사업기획서 §11 가정 실측치 교체) · 골든셋 보강.
- 루틴 중 반복 마찰 발견 시 → 관찰 로그 friction 기록 → 자동화 후보(Toil 제거를 KPI로).
