# 08. 서브에이전트 오케스트레이션

> 큰 작업을 *격리된 서브에이전트*에 분산하고 **요약만 돌려받아** 합성하는 패턴.
> 위상: RFC-002 R4 구현. Anthropic sub-agents(컨텍스트 격리 → 1~2k 토큰 요약 반환)를 방법론 자산으로 코드화. 백서 [10_foundation/WHITEPAPER.md](../10_foundation/WHITEPAPER.md) 제0원칙(이식성)·context engineering(신호>노이즈)의 연장.
> 핵심 가치: **탐색·검증의 원시 컨텍스트는 서브에이전트 안에 가두고, 리드는 결론만 받아 판단한다.**

---

## 1. 왜

긴 작업의 원시 컨텍스트(파일 덤프·탐색 궤적·다관점 토론)를 메인 컨텍스트에 다 쌓으면 context rot이 온다. 서브에이전트는 각자 깨끗한 컨텍스트에서 일하고 **압축된 요약만 반환**해, 리드 에이전트의 컨텍스트를 보호한다. 이 방법론의 격상 토론·무게 감사·SOTA 평가가 실제로 전부 서브에이전트로 수행됐으나 — 패턴은 쓰면서 자산화되지 않았다. 이 지침이 그 갭을 메운다.

## 2. 언제 쓰나

- **Fan-out 탐색**: 여러 파일·디렉터리·명명 규칙을 훑어 *결론만* 필요할 때(원시 덤프 불요).
- **독립 관점**: 한 판단을 서로 다른 렌즈로 교차(찬성/반대/방법론적합성, 4렌즈 감사).
- **적대적 검증**: 발견을 *반증하도록* 시킨 독립 검증자 다수 → 다수가 반증하면 기각.
- **대규모 스윕/마이그레이션**: 항목별 병렬 처리(격리로 충돌 방지).

한 세션 컨텍스트로 감당되는 소규모 작업은 서브에이전트가 오히려 과하다 — 직접 한다.

## 3. 규약 (Contract)

| 규약 | 내용 |
|---|---|
| **요약 반환** | 서브에이전트는 원시 출력(파일 덤프·긴 로그)이 아니라 **결론·근거·구조화 데이터**만 반환한다(1~2k 토큰 목표). |
| **컨텍스트 격리** | 탐색의 원시 맥락은 서브에이전트 안에 남긴다 — 리드는 그걸 다시 받지 않는다(context 보호). |
| **관점 다양화** | 검증은 동일 프롬프트 N개보다 *서로 다른 렌즈* N개가 낫다(중복이 못 잡는 실패 모드를 잡음). |
| **적대적 기본값** | 검증자는 "확인"이 아니라 "반증"을 기본으로 — 확신 없으면 기각 쪽으로. |
| **파이프라인 우선** | 다단계는 기본적으로 항목별 파이프라인(단계 간 배리어 최소화). 전체 취합이 꼭 필요할 때만 배리어. |
| **침묵 절단 금지** | 서브에이전트 수·범위를 제한하면(top-N·무재시도·샘플링) *무엇을 뺐는지* 보고한다(지침 07·백서 §8-5). |

## 4. 스케일 매칭

작업 요청 강도에 맞춘다: "빠른 확인"은 소수 에이전트·단일 검증 / "철저히 감사·평가"는 큰 풀 + 다표 적대적 검증 + 합성 단계. 과한 오케스트레이션은 비용만 늘린다(예산 = 지침 07).

## 5. 다른 지침·구조와의 경계

- **지침 06 (compaction)**: 서브에이전트 요약 반환은 *공간 축의 compaction*(격리로 원시 맥락 배제), compaction은 *시간 축*(세션 수명). 같은 원리(신호>노이즈)의 두 축.
- **지침 07 (예산)**: 서브에이전트 팬아웃은 토큰을 크게 쓴다(멀티 ~15×) → 예산 선언·상한 필수.
- **지침 05 (산출물 채널)**: 서브에이전트 요약은 *메시지 채널*(작업 메타)이다 — 산출물 아님.

## 6. 2025-26 SOTA 보강 (웹리서치, METH-093)

> 출처: Anthropic *How we built our multi-agent research system* · *Building Effective Agents* · Claude Code *subagents*/*Workflows* · Cognition *Don't Build Multi-Agents* · OpenAI agents SDK. **[CC]**=Claude Code/SDK 특유.

### 6.1 결정축: fan-out vs single-writer (§2 보강 — 언제 *안* 쓰나)
- **읽기·탐색·검증의 breadth는 fan-out**, **응집 산출물의 생성은 single-writer**. 공유 컨텍스트 없는 병렬 생성은 **충돌 산출물**을 만든다(Cognition: 한 서브가 마리오 배경, 다른 서브가 안 맞는 새를 그림 — 서로의 암묵 결정을 못 봄). **안 쓸 때**: 순차 의존(한 출력이 다음을 틀 지음)·긴밀 결합·짧은 단발. "대부분의 코딩 작업은 리서치보다 병렬화 가능 부분이 적다."

### 6.2 sizing heuristic (§4 보강)
- 위임 프롬프트에 규모를 박는다: **단순 사실확인=1에이전트/3–10 tool call · 비교=2–4서브 · 복잡=10+**. 과소/과다 스폰 방지. 초기 실패 모드 = *단순 쿼리에 50 서브 스폰*.

### 6.3 위임 계약 필드 (§3 보강)
- 서브에이전트마다 **목표 + 출력 형식 + tool/source 가이드 + 명확한 경계**를 준다(모호하면 중복 작업). **[CC] parent→child 채널은 Agent 툴 프롬프트 문자열 뿐** — 파일 경로·에러·결정을 *명시적으로 넣어야* 서브가 안다(대화 이력 상속 없음).

### 6.4 model/effort 스케일링 (§4 보강)
- **[CC] 서브당 `model`·`effort` 조절**: breadth 탐색=값싼 모델+낮은 effort / 검증·합성=강한 모델. "모델 업그레이드 > 토큰 예산 2배". 토큰량이 성능 분산의 ~80% 설명하나 model/effort가 더 큰 레버.

### 6.5 concurrency cap + 집계 (§3·§4 보강)
- 리드는 **동시 3–5 서브**(각 3+ tool 병렬) 권장·상한 — 복잡 쿼리 시간 ~90%↓. LLM-judge **completeness rubric**(사실·인용 정확성·완전성·소스 품질·tool 효율; ~20 실쿼리로 시작)로 fan-out 누락을 잡는 **completeness critic**. dedup은 *경계 명확화로 상류에서* 방지 + 필요 시 최종 전용 패스.
- **artifact/외부메모리 집계**: 서브가 결과를 *독립 파일로 영속* 쓰고 리드는 포인터만 — 동기 리드 병목(가장 느린 서브에 전체가 블록) 완화. 지침 06(공간축 compaction)과 연동.

### 6.6 스케일 escape + 결정론 (§4 보강)
- **[CC] 수십~수백 에이전트는 turn별 서브 대신 런타임 `Workflow`**(코드 주도 제어흐름)로. 원칙: **"복잡도는 이득이 입증될 때만 추가"** — 예측 가능한 오케스트레이션은 코드에, 모델 주도 위임은 진짜 불확실한 데만.

## 7. 격상 이력

- RFC-002 R4(accepted) 구현. 이 방법론이 *이미 사용 중*인 패턴(격상 토론·감사·평가·리서치)을 사후 명문화 — 관찰→반복→자산화 루프의 실례.
- 강제: 별도 CLI 없음(오케스트레이션은 하네스 기능). 이 지침이 *언제·어떻게* 쓸지의 계약.
