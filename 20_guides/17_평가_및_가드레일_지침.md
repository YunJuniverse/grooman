## 2. Eval-Driven Development (EDD) 원칙

### 2.1 정의
EDD는 AI 기능을 만들기 전에 *어떻게 평가할지를 먼저 정의하는* 방법론이다.
"무엇을 성공으로 볼 것인가"가 합의되지 않은 상태에서 프롬프트와 모델을 바꾸면, 개선과 회귀를 구별할 수 없다.

### 2.2 핵심 원칙
1. 메트릭과 골든셋을 *프롬프트보다 먼저* 정한다
2. 골든셋은 작더라도(50~100개) 시작한다 — 양보다 *대표성*이 중요
3. 채점 자동화 — 매 변경마다 사람이 채점하면 회귀가 잡히지 않는다
4. offline + online 평가를 모두 둔다
5. judge 모델은 가능한 한 *대상 모델과 다른 모델*을 쓴다

### 2.3 EDD 사이클
```
1. 메트릭 정의 → 2. 골든셋 구축 → 3. baseline 평가
                    ↓
6. 회귀 감지 ← 5. 변경 적용 ← 4. 합격 기준 합의
```

## 3. 평가 표준

### 3.1 메트릭 카탈로그

| 메트릭 유형 | 언제 쓰는가 | 측정 방식 |
|-------------|-------------|-----------|
| Exact Match | 정답이 단일하고 결정론적일 때 | 룰 |
| F1 / Precision / Recall | 분류·추출 | 룰 |
| BLEU / ROUGE | 번역·요약 (전통적) | 룰 |
| LLM-as-Judge 점수 | 정답이 다양하고 정성적일 때 | LLM |
| Human Rating | 매우 주관적이거나 high-stakes | 사람 |
| Latency p50/p95/p99 | 모든 운영 기능 | 운영 |
| Cost per request | 모든 운영 기능 | 운영 |
| Refusal rate | 가드레일 트리거 빈도 | 운영 |
| Hallucination rate | RAG/사실 응답 기능 | judge 또는 사람 |

### 3.2 골든셋 구축 원칙
- **최소 50, 권장 100~200개**로 시작
- 다음 카테고리를 *반드시* 포함한다
  - Happy path (정상 입력)
  - Edge case (긴 입력, 특수 문자, 모호한 질문)
  - Adversarial (prompt injection 시도, 금지 요청)
  - Out-of-scope (서비스 범위 밖 질문)
- 각 항목은 (입력, 기대 출력 또는 기대 행동, 메타 라벨)을 갖는다
- 골든셋 자체도 버전 관리한다 (`evals/<feature>/golden-v1.jsonl`)

### 3.3 LLM-as-Judge 사용 가이드
- judge 프롬프트는 *별도 파일*로 관리하고 골든셋과 같이 버전을 매긴다
- judge는 calibration set으로 검증한다 (사람이 채점한 결과와 judge 점수를 비교)
- judge 모델은 가능한 한 대상 모델과 다른 provider/family를 쓴다
- judge 점수는 절대 점수보다 *상대 비교*에 더 신뢰성이 있다 (변경 전 vs 변경 후)

### 3.4 합격·회귀 기준
- 합격 기준: 골든셋에서 [메트릭] [임계값] 이상 [N%] 통과
- 회귀 기준: 직전 승인 버전 대비 [메트릭] [임계값] 하락 시 자동 차단
- high-stakes 기능은 *추가로* 사람 검토를 거쳐야 한다

### 3.5 Offline / Online 평가
- **Offline (개발 시)**: 골든셋 기반 자동 평가, CI에서 차단 게이트
- **Online (운영 시)**: 실사용 트래픽의 일부를 샘플링하여 judge 또는 사람 검토
- 두 결과의 괴리가 커지면 골든셋이 실제 분포를 반영하지 못하는 것 — 골든셋을 보강한다

### 3.6 평가 운영 KPI (15 PM기획서 연동)
- 골든셋 갱신 주기
- 회귀로 차단된 PR 수
- judge calibration 점수
- online 평가 표본 비율

### 3.7 LLM-as-Judge Bias & 완화 (§3.3 심화)
> judge는 편향된다 — 각 편향에 완화를 의무화.

| 편향 | 증상 | 완화 |
|------|------|------|
| Position/order | 먼저(또는 나중) 나온 답 선호 | pairwise 시 **순서 스왑 후 평균/양방향 합의** 필수 |
| Verbosity(length) | 길수록 고득점 | 길이 통제 rubric·근거 없는 부연 감점 |
| Self-preference | 자기 모델 계열 편애 | **생성기 ≠ judge 계열**·교차 계열 panel |
| Sycophancy | 프롬프트 주장에 동조 | reference-guided 채점·confidence 단서 숨김 |

- **G-Eval 기본형**: rubric에서 평가 스텝 생성(CoT) → form-filling 채점(확률가중으로 분산↓).
- **미묘한 품질은 pointwise보다 pairwise**. high-stakes는 **panel(교차 계열)**.
- **judge calibration 게이트**: 사람 라벨 표본과 κ/일치율이 임계 미달이면 CI에서 judge 사용 금지. judge/모델 업그레이드 시 재검증. **judge 모델 버전 pin**.
- **judge 불신 조건**: 사람 일치 낮은 task · 안전/법률 고위험 · 동일 계열 judge·생성기 · 세밀한 절대 점수.

### 3.8 에이전트 / Trajectory 평가 (16 에이전트 기능과 짝)
> 최종 출력만이 아니라 *경로*를 평가. 조직 표준 4지표:
- **task 성공률**(end-state 검증) · **tool-call 정확성**(맞는 tool·유효 인자·순서 — 환각 인자는 답이 그럴듯해도 실패) · **trajectory 건전성**(스텝/서브골 완료·스텝·루프 수·실패 복구) · **task당 비용·지연**.
- (state, action) 쌍을 judge 1~5 채점 후 집계(예: 기하평균 — 한 스텝만 나빠도 벌점). 3레벨: end-to-end / trajectory / component.

### 3.9 RAG 메트릭 카탈로그 (RAGAS — 16 feature-level이 참조)
- **Context Precision**(검색 chunk 신호/잡음·랭킹) · **Context Recall**(필요 정보 전부 검색? — ground-truth 라벨 필요) → *검색* 실패 진단.
- **Faithfulness/groundedness**(컨텍스트가 답을 뒷받침·환각 없음) · **Answer Relevancy**(질문에 답하나) → *생성* 실패 진단.

### 3.10 Eval 데이터 위생
- **오염 검사**: 벤치/골든셋의 verbatim + *패러프레이즈/번역* 누수까지. **홀드아웃 규율**(eval셋은 학습·few-shot에 절대 안 씀). **eval셋 버전**(hash+provenance). **합성 eval 데이터는 사람 검토 필수**(오염 재유입 위험). 공개 벤치는 회전/동적 항목 고려.

## 4. 4-카테고리 가드레일

### 4.1 카테고리 정의

#### A. 콘텐츠 안전 (Content Safety)
유해/혐오/성인/폭력/자해 등 *출력의 사회적 위험*을 통제.
- 입력: 사용자가 안전하지 않은 출력을 유도하는지 탐지
- 출력: 모델이 생성한 텍스트가 안전 기준을 위반하는지 검사
- 위반 시 동작: 표준 거절 메시지로 대체, 로그 기록, 빈도 알람

#### B. 보안 (Security)
*시스템 차원의 위협* 방어.
- Prompt injection 탐지 (사용자 입력에서 system prompt를 우회하려는 시도)
- Jailbreak 패턴 탐지
- Tool abuse 차단 (권한 없는 액션 시도)
- 위반 시 동작: 차단, 보안팀 알람, 사용자 차단 정책 적용

#### C. 데이터 보호 (Data Protection)
*개인·기밀 정보의 누출* 방어.
- 입력 PII 자동 마스킹 (이메일, 전화, 주민번호, 카드번호)
- 출력에서 PII/secret 패턴 탐지 및 차단
- 외부 모델로 전송되는 데이터의 redaction
- 위반 시 동작: 차단, 보안팀 알람, 감사 로그

#### D. 컴플라이언스 (Compliance)
*법규·정책 위반* 방어.
- 도메인별: 의료(HIPAA), 금융(PCI/GLBA), 아동(COPPA)
- 지역별: GDPR, CCPA, **EU AI Act (2026.8 기준)**
- 정책별: 광고 가이드, 자동 의사결정 고지 의무
- 위반 시 동작: 차단, 컴플라이언스 알람, 거부 사유 로그

### 4.2 Guardrails-by-Construction 원칙
가드레일은 *프롬프트가 아니라 시스템 컴포넌트*로 강제한다.

| 원칙 | 권장 (✓) | 지양 (✗) |
|------|----------|----------|
| 위치 | 입력 단/출력 단의 별도 모듈 | 시스템 프롬프트 한 줄 |
| 장애 모드 | fail-closed (위반 시 차단) | fail-open (위반 시 통과) |
| 검증 | calibration set으로 정밀도/재현율 측정 | 사례 몇 개로 확인 |
| 우회 방지 | 사용자 입력으로 가드 비활성화 불가 | 프롬프트 1줄로 끔 |

### 4.3 가드레일 적용 매트릭스 (16번에서 인스턴스화)
각 AI 기능은 본 카탈로그를 참조하여 자기 기능에 맞는 매트릭스를 작성한다.

| 카테고리 | 적용? | 통제 컴포넌트 | 위반 시 동작 | 알람 |
|----------|-------|---------------|--------------|------|
| 콘텐츠 안전 | yes/no | (예: provider safety filter + 룰) | 거절 메시지 / 차단 | high |
| 보안 | yes/no | injection detector / tool guard | 차단 / 사용자 제재 | high |
| 데이터 보호 | yes/no | PII redactor / output scanner | 마스킹 / 차단 | high |
| 컴플라이언스 | yes/no | policy.yaml + 룰 | 거절 / 사람 검토 | medium |

### 4.4 EU AI Act 매핑 (2026.8 기준)
EU 시장에 노출되는 AI 시스템은 다음 분류를 명시해야 한다.

- Minimal risk
- Limited risk (사용자에게 AI임을 고지 의무)
- High risk (이력 관리, 인간 감독, 정확성 입증 의무)
- Unacceptable (금지)

high risk에 해당하는 도메인 예: 채용, 신용평가, 의료 진단 보조, 법 집행 보조, 교육 평가.
프로젝트가 이에 해당하면 16 AI 기능 기획서의 `guardrail_class`를 `high`로 두고, 본 지침의 컴플라이언스 절을 별도 부록으로 채워 운영기획서에 연결한다.

**GPAI 갱신(2025-26)**: 금지 관행+AI 리터러시 2025.2 시행. **GPAI(범용 AI) 모델 의무 2025.8 적용**, **GPAI Code of Practice**(2025.7, 3장: 투명성·저작권·안전보안 — 안전보안은 systemic risk GPAI[Art.55]에만) 서명이 조화표준 전까지 적합성 추정 경로. 학습데이터 요약 템플릿 확정. **집행·과징금 2026.8**(레거시 모델 2027.8). 고위험(Annex III) 의무 2026.8/2027.8 단계 시행. **Art.50 투명성**(AI 상호작용 고지·합성 콘텐츠 표시)은 별도 의무.

### 4.5 레드팀 pre-release 게이트 (런타임 가드와 별개)
> 배포 *전* 적대적 테스트를 표준화 — 런타임 가드레일과 다른 축.
- **적대 스위트**: jailbreak + 프롬프트 인젝션. 도구: Garak·PyRIT·Promptfoo·DeepTeam.
- **finding→regression 루프**: 확인된 레드팀 발견은 *영구 CI 테스트*로 → 재유입 시 배포 차단.
- "레드팀 게이트 통과"를 평가 합격/회귀 임계와 함께 pre-release 체크리스트 항목으로.

### 4.6 거버넌스 프레임워크 매핑 (NIST AI RMF · ISO 42001 · EU AI Act)
> EU AI Act(법적 준수)만이 아니라 *방법론*(NIST)·*경영시스템*(ISO)까지 정렬.
- **NIST AI RMF**(AI 100-1) 4기능: **GOVERN**(정책·책임·문화) · **MAP**(맥락·리스크 식별/인벤토리) · **MEASURE**(정량·정성 벤치·추적) · **MANAGE**(우선순위·대응·완화). **GenAI Profile(AI 600-1)** = 12 GenAI 리스크 범주 × ~200 액션.
- **ISO/IEC 42001:2023** = 인증 가능 **AI 경영시스템(AIMS)** — 10절 + Annex A(~38 controls / 9 objectives). 3년 인증·연 사후심사.
- **매핑 표**(작성): 본 지침 eval/guard 통제 → NIST Govern/Map/Measure/Manage(+600-1 범주) → ISO 42001 절/Annex-A → EU AI Act 조항. 위상: EU AI Act=법적 준수 · NIST=자발적 리스크 방법 · ISO 42001=인증 가능 경영시스템 척추.

## 5. 인간 검토 게이트 (Human-in-the-Loop)

### 5.1 표준 게이트 시점
다음 시점에는 *반드시* 사람 검토를 거친다.

| 게이트 ID | 시점 | 승인 증거 |
|-----------|------|-----------|
| `eval-spec-approval` | 16번 작성 직후 (구현 전) | PR 리뷰 또는 ADR |
| `pre-launch-eval` | 운영 배포 전 | 평가 리포트 첨부 PR |
| `regression-override` | 회귀 차단을 우회하려 할 때 | ADR 필수 |
| `high-stakes-action` | high-stakes Tool 액션 실행 시 | 사용자 명시 동의 + 감사 로그 |
| `guardrail-incident` | 가드 위반 사고 발생 시 | Incident 보고서 |
| `model-swap` | 운영 모델 교체 시 | 회귀 평가 통과 + PM 승인 |
| `policy-change` | 가드레일 정책 변경 시 | 컴플라이언스 검토 ADR |

### 5.2 게이트 통과 증거의 일반 규칙
- 구두 합의 / 메신저 OK는 증거가 아니다
- 증거는 항상 (1) 머지된 PR (2) 링크된 ADR (3) 승인된 issue 셋 중 하나
- AI 기능 기획서(16)의 `human_in_loop_gates`에 게이트 ID와 승인자를 미리 등록한다

## 6. 관측·런타임 평가

### 6.1 트레이스 표준 필드
모든 AI 호출은 아래 필드를 트레이스에 기록한다.

- `request_id`
- `user_id_hash` (PII가 아닌 형태)
- `feature_id`
- `model_provider`, `model_id`, `model_version`
- `prompt_version`
- `input_tokens`, `output_tokens`
- `latency_ms`
- `cost_usd`
- `eval_score` (online sampling 결과가 있을 시)
- `guardrail_triggers` (어떤 카테고리가 발동했는지)
- `fallback_chain` (어느 단계가 최종 응답인지)

> **OpenTelemetry GenAI semconv 정렬**(이식성): 위 필드를 OTel GenAI 표준 속성명에 매핑 — `gen_ai.operation.name`·`gen_ai.provider.name`·`gen_ai.request.model`·`gen_ai.usage.input_tokens`/`output_tokens`, span명 `{operation} {model}`, 메트릭 `gen_ai.client.token.usage`·`gen_ai.client.operation.duration`(히스토그램). 에이전트 오케스트레이션·MCP tool 호출·eval 결과까지 커버. 아직 experimental — `OTEL_SEMCONV_STABILITY_OPT_IN` dual-emission으로 독점 스키마 잠금 회피.

### 6.2 표준 알람 룰
- p95 latency가 목표의 1.5배 초과
- 시간당 비용이 캡의 30% 초과
- guardrail trigger 비율이 baseline 대비 2배 초과
- refusal rate가 baseline 대비 2배 초과
- online judge 점수가 7일 이동평균에서 임계값 하락

### 6.3 데이터 보존
- 트레이스 기본 보존: 30일
- PII 포함 가능 필드: 7일 또는 즉시 redaction 후 보존
- 평가 결과/회귀 이력: 영구 보존

## 7. 자주 하는 실수

### 7.1 골든셋 없이 회귀 평가
"좋아 보인다" 기반 배포는 회귀를 누적시킨다.

### 7.2 가드를 프롬프트에만 의존
사용자가 system prompt를 우회하면 통제가 사라진다.

### 7.3 self-eval 신뢰
같은 모델·같은 프롬프트로 자기 출력을 평가하면 점수가 부풀려진다.

### 7.4 online 평가 없이 offline만
실사용 분포는 골든셋과 다르다. 둘 다 본다.

### 7.5 fail-open 가드
"확실하지 않으면 통과시킨다"는 보안 사고로 직결된다. fail-closed가 기본이다.

### 7.6 게이트 통과를 메신저로 처리
승인 증거가 없으면 사후 사고 시 책임 추적이 불가능하다.

### 7.7 EU AI Act 분류를 미루는 것
"나중에 한다"는 출시 직전 큰 리스크가 된다. high risk 가능성이 있다면 기획 단계에서 분류한다.

## 8. AI 실행용 축약 지시문

평가와 가드레일에 관련된 모든 작업에서 다음 원칙을 따르라.
- 골든셋과 메트릭을 *프롬프트보다 먼저* 정의하라
- judge 모델은 가능한 한 대상 모델과 다른 family를 사용하라
- 가드레일은 4-카테고리(콘텐츠/보안/데이터/컴플라이언스)로 나누어 시스템 컴포넌트로 강제하라
- 모든 가드는 fail-closed를 기본으로 하라
- offline 평가에 더해 online 샘플 평가를 운영하라
- high-stakes 행동, 모델 교체, 회귀 우회는 사람 게이트를 통과해야 하며 증거는 PR/ADR/issue 중 하나여야 한다
- 트레이스 표준 필드를 모두 기록하고, 비용·지연·refusal·가드 트리거에 대한 알람을 켜라
- EU AI Act에 해당할 가능성이 있는 도메인은 기획 단계에서 분류하라
