# 06. 컨텍스트 컴팩션 프로토콜

> 긴 세션이 컨텍스트 한계에 닿아 *요약·재시작(compaction)*될 때, **무엇을 반드시 살리고 무엇을 버리는가**의 규칙.
> 위상: RFC-002 R2 구현. 백서 [10_foundation/WHITEPAPER.md](../10_foundation/WHITEPAPER.md) 제0원칙(이식성)·제2원칙(인계 가능성)의 *세션 내 런타임 실천*. context engineering(신호>노이즈)을 세션 수명에 적용.
> 핵심 가치: **compaction은 "세션 중간 인계"다. checkpoint가 세션 종료용 인계라면, 이 프로토콜은 세션이 스스로에게 넘기는 인계다.**

---

## 1. 왜 (백서 종속)

컨텍스트는 유한 자원이다 — 토큰이 쌓이면 정확도가 떨어진다(context rot). 긴 세션은 하네스가 컨텍스트를 *요약하고 새 창으로 재시작*하는 **compaction**을 거친다. 이때 **무엇을 요약에 담느냐가 재개 품질을 결정**한다. 과하게 버리면 나중에야 중요성이 드러나는 맥락을 잃고, 과하게 남기면 context rot이 재발한다.

이는 제2원칙(인계 가능성)의 런타임 버전이다: *"미래의 나(=compaction 이후의 나)도 다른 사람이다."* 세션 종료 시 checkpoint를 남기듯, compaction 경계에서도 **홀로 서는 요약**을 남겨야 한다.

## 2. Compaction이란

컨텍스트 한계 근처에서 대화 내용을 요약해 새 컨텍스트 창으로 재시작하는 기법(SOTA: Anthropic context engineering의 장기 실행 1순위 기법). 원칙: **recall 최대화 후 precision** — 먼저 놓치면 안 되는 것을 다 담고, 그다음 군더더기를 제거한다.

## 3. 보존 / 폐기 규칙 (핵심)

| 반드시 **보존** (compaction 요약에 살림) | **폐기** (요약에서 버림) |
|---|---|
| 활성 아키텍처·설계 **결정**과 그 이유 | **원시 툴 출력** — 파일 덤프·grep 결과·긴 로그·명령 출력 |
| 사용자의 **명시 제약·결정·스코프 확정** | 탐색 중 **dead-end**(가봤지만 아닌 길) |
| 현재 **게이트 상태** — 어느 휴먼 게이트를 대기 중인가 | **중복 재서술** — 같은 내용의 반복 |
| 진행 중 작업의 **다음 액션**(구체적 첫 행동) | **이미 라이브 파일·frontmatter에 반영된** 내용(그건 파일이 들고 있음 → 포인터만) |
| 미해결 **open questions** | 완료·검증된 중간 산출물의 **raw 형태**(결론만 남김) |
| 라이브 파일·핵심 소스 **경로 포인터**(JIT 재로드용) | 되돌린(revert) 시도의 상세 과정 |
| **되돌리기 어려운 것**의 이유·전제 | |

> 판정 한 줄: *"compaction 후의 내가 이걸 잃으면 사용자에게 다시 물어야 하나?"* — 그렇다면 **보존**. 파일을 다시 읽으면 되는 것은 **폐기**(경로만 남김 — JIT 재검색).

## 4. checkpoint = compaction survival kit

`.ai/checkpoint.md`(백서 §2-2)의 세 섹션 — **방금 한 것 · 다음 사람에게 · 미해결 결정사항** — 이 곧 compaction 생존 키트다. 따라서 checkpoint 갱신 **트리거를 확장**한다:

- **(기존)** 세션 종료 시.
- **(신설)** **compaction 임박 시** — 하네스가 요약을 예고하거나 컨텍스트가 한계에 근접하면, 요약 전에 checkpoint를 먼저 갱신한다.
- **(신설)** **긴 세션의 자연 경계** — 큰 작업 단위가 끝날 때마다. checkpoint가 최신이면 compaction이 언제 일어나도 안전하다.

즉 checkpoint는 "세션 끝에 쓰는 것"이 아니라 **"항상 최신으로 유지하는 것"**으로 바뀐다. 라이브 파일(HANDOFF·TODO)도 같은 원리 — compaction 이전에 상태가 파일에 있으면, 요약이 그걸 잃어도 파일에서 복원된다.

## 5. Pre-compaction 체크리스트

컨텍스트가 한계에 근접했다고 판단되면, 요약 전에:

1. **라이브 파일 먼저 갱신** — 진행 상태·결정·다음 액션을 HANDOFF·TODO·checkpoint에 *지금* 반영(§3 보존 항목이 파일에 있으면 요약이 실패해도 복원 가능).
2. **미해결 결정 명시** — open questions를 checkpoint에 적는다(요약이 가장 잘 잃는 것).
3. **경로 포인터 확보** — 다음에 읽어야 할 파일 경로를 checkpoint "다음 사람에게"에 적는다(요약은 파일 *내용*이 아니라 *경로*만 담아도 됨 — JIT 재로드).
4. **원시 출력 미보존** — 긴 grep/파일 덤프는 결론만 남기고 버린다.

## 6. 다른 지침·구조와의 경계

- **지침 03 (관찰 로그)**: *사후* 학습 기록(세션 종료 후 L1). compaction은 *진행 중* 생존. 시점이 다르다.
- **지침 05 (산출물 채널 분리)**: compaction 요약은 *메시지 채널*(작업 메타)이다 — 산출물이 아니다.
- **checkpoint (백서 §2-2)**: 이 프로토콜은 checkpoint의 *갱신 시점*을 확장할 뿐, 형식은 §2-2 그대로.
- **HANDOFF·TODO**: compaction 이전 상태의 최종 방어선. "파일에 있으면 요약이 잃어도 산다."

## 7. 운영 통합

- `CLAUDE.md`·`AGENTS.md` 세션 절차에 **compaction 경계 checkpoint 트리거**를 추가(세션 종료 절차의 형제 규칙).
- 강제 메커니즘: 별도 CLI 없음(compaction은 하네스가 일으키므로 사전 CI 훅 부적합). 대신 **운영 계약(CLAUDE.md)에 편입**해 부팅 시 항상 로드 — 지침 03이 백서 원칙 없이도 구속력을 갖는 것과 동일.

## 8. 2025-26 SOTA 보강 (웹리서치, METH-093)

> 출처: Anthropic *Effective context engineering* · *Effective harnesses for long-running agents* · Claude Code compaction/context-window 문서. **[CC]**=Claude Code 특유, **[일반]**=일반 원칙.

### 8.1 두 층의 compaction — 임계치 (§2 보강)
- **[CC] 하네스 auto-compact ~95–98%** 자동 발동 — 가장 가벼운 것부터: **이미지/PDF 제거 → 큰 툴 결과 절삭 → 나머지 요약**. `/compact` 수동·`/context` 사용량·`/clear`는 요약 없이 전부 삭제(≠compact).
- **[일반] proactive는 ~60%·자연 경계에서** — 경고를 기다리면 이미 80–95%로 저하 중. 큰 작업 단위가 끝나는 지점에서 *먼저* checkpoint 갱신하고 필요 시 `/compact`.
- **[CC] API-층 compaction**(별도): input 토큰이 trigger(기본 150k, 최소 50k)에 닿으면 요약 블록 1개로 압축·이전 블록 폐기. 방법론은 하네스 층에서 돌지만 보존/폐기 로직 동일.

### 8.2 무엇이 자동 생존하나 — *그래서 포인터가 안전하다* (§3·§4 보강)
- **[CC] 디스크 파일은 compaction 후 다시 읽힌다.** 루트 `CLAUDE.md`·무조건 로드 rule·auto-memory(`MEMORY.md` ~200줄)도 재주입. → §3 "경로 포인터만 남긴다"가 안전한 *이유*. 내용 아닌 경로를 남겨라.
- **[CC] ⚠️ 재주입 안 되는 것**: `paths:`-스코프 rule·하위 디렉터리 `CLAUDE.md`는 *매칭 파일을 읽어야* 들어온 것이라 요약에 버려지고 **자동 복귀 안 됨**. 필요하면 compaction 후 **매칭 파일을 다시 읽어 re-anchor**.

### 8.3 폐기 순서 — safest-first (§3 보강)
- **원시 툴 출력을 *가장 먼저* 버린다**(Anthropic: "safest lightest touch form"). 그다음 dead-end → 중복. **결정·이유·제약·다음 액션은 마지막까지 절대 안 버림.** 과한 compaction은 *"나중에야 중요성이 드러나는 subtle하지만 결정적인 맥락"*을 잃는다(위험 인용).

### 8.4 post-compaction 검증 (§5 보강)
- 재개 직후 **"지금 어디·다음 액션" 한 줄을 스스로 확인**하고 checkpoint와 대조. 어긋나면 라이브 파일에서 복원.

### 8.5 subagent isolation = *공간축* compaction (§6 보강)
- **[일반] 고토큰 탐색은 서브에이전트로 오프로드** — clean 컨텍스트에서 수만 토큰 쓰고 **1–2k 요약만** 반환하므로 상세가 애초에 메인 창에 안 들어옴(요약out 대상도 안 됨). 이 프로토콜(시간축)의 짝. 상세 지침 08.

## 9. 격상 이력

- RFC-002 R2(accepted, 2026-Q3 회고 비준)의 **첫 실제 구현**. SOTA(Anthropic context engineering)의 compaction 기법을 방법론 구조(checkpoint·라이브 파일)에 접합.
- 향후: 세션 지표(compaction 후 재개 시 사용자 재질문 횟수)를 회고에서 추적 — 이 프로토콜의 효과 측정.
