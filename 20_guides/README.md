# 기획 지침서 카탈로그 (Planning Guides)

> 본 폴더는 AI가 기획 프로젝트를 수행할 때 따라야 하는 표준 작성 지침을 담는다.
> 8종의 핵심 지침과 2종의 AI 보조 지침으로 구성된다.

---

## 1. 폴더의 목적

이 폴더는 다음을 위해 존재한다.

1. **AI에게 기획 작업의 표준을 제공한다** — 어떤 문서를 언제·어떤 구조로 만들지에 대한 단일 출처
2. **사람과 AI가 공유하는 계약을 만든다** — 사람이 검토하고, AI가 실행하는 공용 산출물 표준
3. **문서 간 경계를 강제한다** — 원본 정보의 중복을 막고 책임을 분명히 한다
4. **AI 시대의 기획 표준을 통합한다** — 전통 기획 지침에 평가·가드레일·관측을 결합한다

## 2. v3 업데이트 요약 (2026-05)

이전 버전(v2)은 전통 PMI 스타일 기획 지침이었다. v3에서는 2026 AI 엔지니어링 실무 표준에 맞춰 다음을 추가했다.

### 2.1 새로 추가된 원칙 (00 운영원칙에 삽입)
- **Eval-First 원칙** — 작성보다 평가 정의를 먼저
- **Harness 분리 원칙** — Plan / Generate / Evaluate를 같은 세션·에이전트로 묶지 않음
- **Guardrails-by-Construction 원칙** — 프롬프트가 아닌 아키텍처 차원의 안전 통제
- **사람-AI 공용 산출물 원칙** — 자연어 + 기계 판독 가능 형태(OpenAPI/llms.txt/policy.yaml/prompts)

## 5. 사용 가이드

### 5.1 새 프로젝트를 기획할 때
1. 00 원칙을 먼저 읽는다 (전체 철학 이해)
2. 01 오케스트레이션으로 어떤 문서를 만들지 결정
3. 단계별 권장 (00의 9절)을 참고해서 작성 순서를 정한다
4. AI 기능이 있다면 16 + 17을 동시에 시작한다 (Eval-First)

### 5.2 단일 문서 작성 시
- 해당 번호의 지침서 17번 절(축약 지시문)을 AI에 전달한다
- frontmatter는 00의 5.7을 참조해서 작성한다
- 다른 문서와의 경계는 12번 절을 확인한다

### 5.3 AI 기능을 포함한 프로젝트
1. **반드시** 16 AI 기능 기획서 작성
2. **반드시** 17 평가/가드레일 지침 참조
3. 부모 서비스기획서(11)에 6.13~6.15 절(AI 진입점·실패 UX·Eval Spec)을 채워둔다
4. 운영기획서(12)에 6.12~6.14 절(AIOps·게이트 운영·Incident 분류)을 채워둔다
5. PM기획서(15)에 6.15~6.17 절(Eval 포트폴리오·게이트 카탈로그·AI 위험 등록부)을 채워둔다
6. EU 시장 노출 가능성이 있다면 17의 4.4(EU AI Act 매핑)를 *기획 단계에서* 결정한다

## 7. v3에서 받아들인 외부 표준

본 버전은 다음 외부 동향을 반영했다.

- **Anthropic Harness Engineering** — Plan / Generate / Evaluate 분리, 구조화 handoff artifact
- **Evaluation-Driven Development (EDDOps)** — 평가를 거버닝 함수로 다루는 폐쇄 루프
- **Spec-Driven Development (Kiro/EARS)** — Requirements/Design/Tasks 3단계, 검증 가능한 요구사항 표기
- **AI PRD 모범사례** — 150~200 instruction 단위 분해, 명시적 non-goals, 실패 UX 의무화
- **Guardrails-by-Construction** — 프롬프트가 아닌 시스템 컴포넌트로 강제
- **EU AI Act (2026.8 발효)** — Minimal/Limited/High/Unacceptable risk 분류
- **Agent Experience (AX)** — 사람이 읽는 문서 + AI가 읽을 수 있는 기계 판독 형태 병행
