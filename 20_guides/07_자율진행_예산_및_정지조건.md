# 07. 자율 진행 예산 및 정지 조건

> AI가 사람 확인 없이 여러 단계를 *자율로* 진행할 때(대규모 전파·리트로핏·스윕), **얼마나 쓰고 언제 멈추는가**의 규칙.
> 위상: RFC-002 R3 구현. loop engineering(budget·no-progress·iteration cap)을 방법론에 접합. 백서 [10_foundation/WHITEPAPER.md](../10_foundation/WHITEPAPER.md) §5 휴먼 게이트의 *런타임 보조* — 게이트가 "언제 사람이 판단하나"라면, 이 규칙은 "자율 구간이 언제 스스로 멈추고 사람을 부르나".
> 핵심 가치: **예산 내에서는 자율, 예산을 넘거나 진전이 없으면 멈추고 보고.**

---

## 1. 왜

방법론의 유일한 정지 장치는 휴먼 게이트뿐이다. 그러나 자율 진행 구간(예: "슬라이스 전체 알아서 진행")에는 *compute 규율*이 없어, 잘못하면 무한 반복·과소비·헛도는 작업이 사람 개입 없이 누적된다. loop engineering은 자율 루프에 **예산·no-progress 감지·반복 캡**을 필수로 본다(단일 에이전트 ~4×, 멀티 ~15× 토큰).

이는 사용자의 자율 진행 선호(메모리 [[user-autonomy-multistep]])와 상보적이다: **자율을 넓히려면 그만큼 "언제 멈출지"를 명시**해야 폭주가 안 난다.

## 2. 규칙 (Hard Rules)

| 규칙 | 내용 |
|---|---|
| **예산 선언** | 대규모 자율 작업 착수 전, 개략 예산을 선언한다 — 예상 파일 수·PR 수·반복 횟수 또는 토큰 규모. 선언 없는 무제한 자율 금지. |
| **no-progress 정지** | 같은 실패·같은 지점에서 **2회 이상 진전 없이 반복**되면 멈추고 사람에게 보고한다(원인 가설 포함). 세 번째 재시도 금지. |
| **반복 캡** | 루프성 작업(항목별 반복 처리)은 상한을 정한다. 상한 도달 시 남은 항목을 *보고*하고 자동 계속하지 않는다. |
| **예산 초과 = 정지 조건** | 선언 예산을 초과하면 그것이 새로운 정지 조건이다. 초과분은 사람 승인 후 재개(휴먼 게이트로 승격). |
| **침묵 절단 금지** | 예산·캡으로 범위를 줄이면 *무엇을 남겼는지* 반드시 보고한다(백서 §8-5 정신 — 조용한 truncation은 "다 했다"로 오독됨). |

## 3. "예산 내 자율, 초과 시 보고" 절차

1. **선언**: "이 전파는 5개 repo·5 PR 예상" 처럼 착수 시 예산을 말한다.
2. **진행**: 예산 내에서는 추천+실행 후 보고(자율). 기계적·검증가능 단계는 멈추지 않는다.
3. **정지 트리거 감시**: no-progress 2회 / 반복 캡 도달 / 예산 초과 / 되돌리기 어려운 결정 → **멈춤**.
4. **보고**: 멈춤 시 현재까지 진행·남은 것·정지 사유·다음 선택지를 사람에게.

> 자율 진행 중에도 방법론 가드레일은 유지된다: 빅뱅 금지, Class C·브랜드·스코프 결정은 사람(메모리 [[user-autonomy-multistep]]). 예산은 그 위에 얹는 *compute 상한*이다.

## 4. 다른 지침·구조와의 경계

- **백서 §5 휴먼 게이트**: 게이트는 *페이즈 경계*의 사람 판단. 이 규칙은 *자율 구간 내부*의 런타임 정지. 게이트를 대체하지 않고 보조한다.
- **지침 06 (compaction)**: compaction은 컨텍스트 한계 대응, 이 규칙은 compute·진전 한계 대응. 둘 다 "긴 자율 작업의 안전장치".
- **Class B/C**: 예산 초과로 멈춘 뒤 재개는 사람 승인 = 사실상 Class 승격 트리거가 될 수 있다.

## 5. 2025-26 SOTA 보강 (웹리서치, METH-093)

> 출처: Anthropic *Building Effective Agents* · *Harness design for long-running apps* · *Measuring agent autonomy* · Claude Agent SDK *agent-loop* · loop-engineering·agent-failure 문헌. **[CC]**=Claude Code/SDK 특유.

### 5.1 이중 예산 — runtime + declared scope (§2 보강)
- **runtime 예산**(루프 폭주 차단, 런타임 enforced): 반복 수 또는 비용(USD). **[CC] SDK `max_turns`·`max_budget_usd`** — *둘 다 기본 무제한*. 명시 안 한 자율 실행엔 런타임 서킷브레이커가 아예 없다(가장 중요한 caveat).
- **declared scope 예산**(드리프트 차단, 사람 선언): 파일/PR 수(현행 §2). *런타임이 강제 못 하는 의미 예산*.
- → **둘을 짝지어 선언**: runtime는 loop을, scope는 drift를 잡는다.

### 5.2 6 circuit breaker 체크리스트 (§2 보강)
① max-iterations ② 비용 상한 ③ wall-clock ④ error-rate ⑤ permission scope ⑥ human-escalation. **[CC] 이 중 ①②⑤만 SDK가 강제**, 나머지는 운영 계약(사람이 감시).

### 5.3 진척은 ground-truth로 판정 (§2 no-progress 보강)
- no-progress를 **모델 self-report가 아니라 환경 상태**(테스트 통과 수·빌드·diff)로 판정한다. 에이전트는 *"형편없는 결과를 자신있게 칭찬"*하므로 자기평가 신뢰 금지. → **생성과 평가 분리**: 긴 자율 실행은 독립 verify 패스(또는 평가 서브에이전트)를 *완료 선언 전* 게이트로.

### 5.4 ask → clarify → escalate 사다리 (§3 보강)
- **근거 있으면 답 → 불확실하면 질문 → 그래도 막히면 사람 에스컬레이션.** ⚠️ **over-asking도 실패** — 모든 단계를 게이트하지 말 것(Anthropic: 행동마다 승인은 "안전 이득 없이 마찰만"). 자율은 *모델 다이얼이 아니라 감독 설계*에서 나온다 → 잘 놓인 정지조건 + 에이전트의 자발적 일시정지.

### 5.5 "되돌리기 어려운 것" = 독립 정지 클래스 (§3 보강)
- 비가역·대외 행동(데이터 삭제·프로덕션/설정 변경·배포·과금·발송)은 **남은 예산과 무관하게 동기 승인 게이트**. 이는 방법론 **Class C**(중단→사람 승인)와 정확히 매핑 — Class C가 이 정지의 상위 규칙.

### 5.6 stop report 표준형 (§3.4 보강)
멈출 때 보고: **정지 사유**(어느 조건 발동) · **예산 소비 vs 선언**(예: 4/5 PR) · **검증 상태**(ground-truth) · **남은 것**(침묵 절단 금지) · **다음 선택지** · **resume handle**(브랜치/session/checkpoint). [CC] SDK `ResultMessage.subtype`(success/error_max_turns/error_max_budget_usd)가 이 형태.

### 5.7 재선언 전 checkpoint (§3 보강)
- 예산 도달로 "계속?"을 물을 땐 **먼저 checkpoint 갱신 후 새 예산 재선언** — 조용히 연장 금지(지침 06·checkpoint와 연동).

## 6. 격상 이력

- RFC-002 R3(accepted) 구현. loop engineering의 budget/stop 원칙을 방법론의 자율-진행 규율로 명문화.
- 강제: 별도 CLI 없음(자율 판단은 런타임). 운영 계약(이 지침 + 사용자 보고)으로 강제 — 향후 대규모 작업 로그에서 예산 준수 여부를 회고에서 점검.
