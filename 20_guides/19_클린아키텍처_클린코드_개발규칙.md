## 1. 문서 개요

### 1.1 문서명
클린 아키텍처 · 클린 코드 개발 규칙 (Code-Quality Guardrails by Construction)

### 1.2 문서 목적
fullstack 프로젝트가 **첫날부터** 레이어 경계·타입 정직성·파일 크기·로깅 규율을 ESLint(또는 동급 린터) 가드레일로 강제하도록 하는 조직 표준. 사후 리팩토링(아키텍처 부채 정리)을 *애초에 발생하지 않게* 하는 것이 목표다.

### 1.3 문서의 역할
지침 17(평가·가드레일) **§4.2 Guardrails-by-Construction 원칙**을 **코드 품질**에 인스턴스화한다. 17이 *AI 안전* 가드레일이라면, 본 문서는 *코드 품질* 가드레일이다. 동일 원칙: 가드레일은 프롬프트가 아니라 시스템(린트)으로 **fail-closed** 강제하며, 프롬프트 한 줄로 끌 수 없다.

### 1.4 적용 범위
fullstack·dev 트랙(Next.js/TS + Supabase 류 등) 전반. planning-only 트랙은 비적용. 스택이 달라도 §2·§3·§6·§8의 *원칙*은 이식 가능하다. (작업 모드는 `50_resources/templates/_CATALOG.md` 참조.)

### 1.5 예상 독자
개발을 수행하는 AI 모델, 코드 리뷰어, 인계받는 사람.

### 1.6 관련 문서

## 2. 핵심 원칙 — Guardrails by Construction (코드 품질)

> **부채는 사후 정리보다 발생 방지가 압도적으로 싸다.** `any` 1개·401줄 파일·레이어 위반을 *커밋 단계에서* 막으면, 수십 PR짜리 리팩토링이 통째로 사라진다.

| 원칙 | 권장 (✓) | 지양 (✗) |
|------|----------|----------|
| 강제 위치 | 린트 규칙(CI/pre-push) | 코드 리뷰 구두 지적 |
| 장애 모드 | fail-closed (`error` → 머지 차단) | fail-open (`warn` 방치) |
| 도입 방식 | day-1 `warn` 베이스라인 → 0 수렴 후 `error` 승격(래칫) | 한 번에 error로 기존 코드 마비 |
| 우회 방지 | 규칙 끄기는 ADR 근거 필요 | 파일 상단 `eslint-disable` 남발 |

**래칫 원리**: 새 규칙은 `warn`으로 시작(기존 위반 표면화) → 카테고리별로 0까지 감축 → **0 도달 즉시 `error` 승격**. 이후 신규 위반은 자동 차단. (역행 불가 톱니바퀴.)

## 3. 레이어 아키텍처 + 의존성 규칙

### 3.1 4-레이어
| 레이어 | 책임 | 허용 import |
|--------|------|-------------|
| **domain** | 순수 비즈니스 규칙·타입·계산 (IO 없음) | domain, shared 만 |
| **application** | use case·IO 오케스트레이션(DB·외부 API) | domain, application, infrastructure, shared |
| **interface** | UI 컴포넌트 | application(서버액션/use case), shared. *infra 직접 호출 금지* |
| **app** (라우트) | 페이지·라우트 핸들러 | application use case. *DB 직접 호출 금지* |

### 3.2 경계 강제 (`no-restricted-imports`)
- `domain/**` → `@/infrastructure`·`@/application`·`@/interface`·`@/app` import **금지**. IO를 동반한 로직은 application으로.
- `app/**`·`interface/**` → `@/infrastructure/<db>`(예: supabase) 직접 import **금지**. 라우트/컴포넌트는 use case·서버액션 호출.
- 정당한 예외(브라우저 auth/storage 업로드 등)만 린트 override로 명시 화이트리스트.

### 3.3 핵심 패턴
- 라우트/컴포넌트의 인라인 DB 쿼리는 `application/<도메인>/queries.ts`(읽기)·`actions.ts`(쓰기)로 추출.
- 인증만 필요한 라우트는 `getCurrentUser()` 같은 단일 use case로(직접 `auth.getUser()` 호출 금지).

> **레이어 의미 함정**: `domain/*/actions.ts`가 사실 IO를 포함한 use case인 경우가 흔하다 → 그건 application 소속. 진짜 domain은 *IO 없는 순수 계산*(점수·한도·상태 전이 계산 등)만.

## 4. 4개 코드 가드레일 (린트)

day-1에 린트 설정에 세팅(처음엔 `warn`, 래칫으로 `error`).

| 가드레일 | 규칙(ESLint 예) | 의도 |
|----------|------|------|
| 레이어 경계 | `no-restricted-imports` (레이어별) | 의존성 규칙 영구 강제 |
| 타입 정직성 | `@typescript-eslint/no-explicit-any: error` | `any` 금지 → `unknown`+가드 |
| 로깅 규율 | `no-console: [error, { allow: ["warn","error"] }]` | 단일 logger sink만(`shared/utils/logger`) |
| 파일 크기 | `max-lines: [error, { max: 400, skipBlankLines: true, skipComments: true }]` | god파일 방지 → SRP 분할 |

- 생성 타입(예: DB 타입)·i18n 사전 등 *데이터 파일*은 max-lines 면제(`ignores`).
- logger 파일만 `no-console: off`(유일한 sanctioned sink).

## 5. 타입 정직성

- **`any` 0**: 불가피하면 `unknown` + 타입 가드. 경계 캐스트는 `as Tables["x"]["Insert"]`처럼 *좁고 명시적*으로.
- **클라이언트 바인딩**: DB 클라이언트 등은 타입 파라미터로 바인딩(`createClient<Database>`) → `(client as any)` 불필요. `as unknown as` 우회 금지.
- **`any`는 버그 은폐 장치**: 타입 정직화는 곧 버그 발굴이다. 캐스트 제거 중 드러나는 스키마/조인 갭은 *진짜 버그*일 가능성이 높으니 즉시 검증. (사례: GambleScan 리팩토링에서 `any` 제거가 "존재하지 않는 테이블 조회로 KPI 전부 0", "없는 컬럼으로 별점 깨짐" 등 실런타임 버그 4건을 드러냄.)

## 6. 파일 크기 규율 (max-lines=400) + 분할 패턴

400줄 초과 파일은 god파일. **동작 불변**으로 추출해 분할한다.

| 대상 | 패턴 |
|------|------|
| 큰 순수 상수(COPY/배열)·스키마(zod 등) | colocated 시블 `*-copy.ts`/`*-fields.ts`로 |
| 순수 함수·헬퍼 | colocated 시블로(외부 importer 있으면 re-export) |
| 폼 컴포넌트 | 핸들러는 state 클로저라 잔류, *순수 헬퍼·JSX 섹션 서브컴포넌트*만 prop으로 추출 |
| 큰 페이지/컴포넌트 | JSX 섹션을 서브컴포넌트로(props로 데이터 전달) |
| 헤비 로직 모듈(1000줄+) | **N-way 분할**: types/keywords/scoring/build 등 책임별 |

**분할 시 주의(사례 교훈)**:
- 분할 후 *새 시블*도 max-lines 재확인 — 한 시블이 또 god파일이 되면 더 쪼갠다.
- 함수 간 **cross-call이 순환**을 만들면, cross-call 함수를 별도 `internals` 모듈로 묶어 선형 의존(`types/internals → A → B → entry`)으로.
- (Next.js) **"use server" 파일은 async 함수만 export 가능**(re-export·type export 금지) → 배럴은 plain(비-use-server), 모든 액션은 도메인 use-server 파일로. 반드시 `build`로 검증.
- 라인범위 추출 시 *함수 사이에 산재한* 스키마/타입/`let` 캐시 누락 주의 → typecheck로 수렴, 깨진 시도 후 원본은 `git show HEAD`로 재독.

## 8. 테스트 안전망

- **도메인 순수로직은 100% 커버 목표**: 권한 매트릭스·상태 전이·점수/한도 계산 등 고위험 순수 로직에 *특성화(characterization) + 가드* 테스트.
- IO 액션의 순수 규칙은 **domain으로 추출 후 특성화**(현 동작 pin) — IO 통합 테스트는 tier에 따라 선택.
- 비즈니스 로직 변경은 테스트 동반(가능하면 test-first — `CLAUDE.md` §7).
- 특성화 테스트가 드러낸 *동작 갭*(예: 임의 상태 전이 허용)은 별도 정책 결정(Class A/B)으로 처리하되, 테스트는 현 동작을 먼저 고정.

## 10. 안티패턴·교훈 (정직 모드)

- **"일단 동작" → 부채**: 사후 정리는 day-1 강제보다 수십 배 비싸다(사례: REFACTOR-CLEAN ~50 PR).
- **god파일 신호**: 한 파일에 여러 책임(폼+섹션+핸들러·여러 도메인 액션)이 모이면 분할 시점.
- **"COPY 중복"의 함정**: 인라인 i18n COPY는 대부분 *컴포넌트 고유*다. 공유 사전으로 옮기는 건 dedup이 아니라 재배치 — *진짜 cross-file 중복*(공통 라벨)만 중앙화하고 고유 텍스트는 inline 유지가 올바른 분리. 무리한 "COPY 0"은 안티패턴.
- **매 슬라이스 브랜치 먼저**: 브랜치 없이 `ship`하면 main 직푸시 사고. main 히스토리 재작성은 더 위험하니 *예방*이 정답.
- **정직한 명명**: 단계가 *의도적 de-scope*면 "완료"로 표기 금지 — "건너뜀(근거)"으로. (백서 §8-4 정직 명명.)
