# 01_AI_기획_오케스트레이션_지침서

## 1. 문서 개요

### 1.1 문서명
AI 기획 오케스트레이션 지침서

### 1.2 문서 목적
본 문서는 AI가 기획 프로젝트 요청을 받았을 때, 해당 요청을 적절한 문서 유형으로 분류하고, 문서 간 관계를 설계하며, 마스터 문서와 보조 문서를 구분하고, 프로젝트 단계에 맞는 문서 생성 순서를 결정하도록 돕는 오케스트레이션 기준을 정의한다.

### 1.3 문서의 역할
본 문서는 개별 문서를 직접 작성하는 상세 작성 지침이 아니라, **어떤 문서를 언제, 왜, 어떤 구조로 먼저 다룰지 판단하는 총괄 운영 문서**다.

### 1.4 적용 대상
본 문서는 아래와 같은 상황에서 사용한다.

- 새로운 프로젝트 요청이 처음 들어왔을 때
- 하나의 요청이 여러 기획 영역에 걸쳐 있을 때
- 어떤 문서를 먼저 작성할지 정해야 할 때
- 문서 간 중복과 경계를 조정해야 할 때
- 프로젝트 단계별로 필요한 문서 세트를 추천해야 할 때

### 1.5 예상 독자
- AI 시스템 설계자
- PM
- 프로젝트 운영자
- 기획 책임자
- 기획 자동화 프롬프트 설계자

## 2. 오케스트레이션의 기본 정의

### 2.1 오케스트레이션이란
오케스트레이션이란, 하나의 프로젝트 요청을 적절한 기획 문서 단위로 분해하고, 각 문서의 역할을 명확히 구분하며, 프로젝트 단계에 따라 어떤 문서를 먼저 만들고 어떤 문서를 나중에 보완할지 설계하는 작업이다.

### 2.2 왜 필요한가
기획 프로젝트 요청은 대부분 단일 문서로 깔끔하게 떨어지지 않는다.  
예를 들어 “플랫폼을 기획해달라”는 요청에는 다음 요소가 동시에 섞여 있을 수 있다.

- 시장성과 BM
- 사용자 기능과 UX
- 운영 정책
- 런칭 전략
- 브랜드 방향
- 일정, 보고, 이슈, 리스크, 의사결정 관리

오케스트레이션이 없으면 AI는 이 모든 내용을 한 문서에 과도하게 넣거나, 반대로 필요한 문서 분리를 놓칠 수 있다.

### 2.3 오케스트레이션의 목적
- 요청을 정확히 해석한다
- 필요한 문서만 선택한다
- 마스터 문서와 보조 문서를 구분한다
- 문서 생성 순서를 설계한다
- 문서 간 입력과 출력을 연결한다
- 중복과 역할 혼합을 줄인다

## 3. 오케스트레이션의 기본 원칙

### 3.1 먼저 분류하고 나중에 작성한다
AI는 요청을 받으면 즉시 내용을 길게 쓰기 전에, 먼저 요청이 어떤 문서 유형에 속하는지 분류해야 한다.

### 3.2 한 요청은 하나 이상의 문서로 분해될 수 있다
사용자 요청이 한 문장이라도, 실제로는 다음처럼 다수 문서를 요구할 수 있다.

예시:
- “커뮤니티 플랫폼 기획해줘”
  - 사업기획서
  - 서비스기획서
  - 운영기획서
  - 필요 시 마케팅기획서

### 3.3 마스터 문서를 먼저 정한다
복합 요청일 경우 먼저 **마스터 문서**를 정한다.  
마스터 문서는 해당 요청의 중심 질문을 가장 잘 담는 문서다.

예시:
- “이 사업을 왜 해야 하는지와 수익 모델까지 정리해줘” → 사업기획서가 마스터
- “회원가입부터 후기 작성까지 UX를 설계해줘” → 서비스기획서가 마스터
- “운영 기준과 신고 처리 프로세스를 짜줘” → 운영기획서가 마스터

### 3.4 상세 실행 문서는 필요할 때만 분리한다
모든 요청에 5종 문서를 다 만들 필요는 없다.  
요청 범위와 프로젝트 단계에 맞춰 필요한 문서만 제안해야 한다.

### 3.5 문서 간 원본은 한 곳에만 둔다
문서가 여러 개일 경우, 각 핵심 정보의 원본은 하나의 문서에만 둔다.  
다른 문서에는 요약 또는 참조만 둔다.

### 5.7 AI 기능 기획으로 분류하는 기준
아래 질문이 중심이면 AI 기능 기획(16)으로 라우팅한다.

- 어떤 입력을 어떤 출력으로 LLM/생성형 모델이 변환하는가
- 어떤 모델·프롬프트·온도를 쓰는가
- AI 출력의 정답을 어떻게 평가하는가 (골든셋, judge, 메트릭)
- AI가 환각하거나 실패했을 때 사용자는 어떻게 행동할 수 있는가
- AI 호출에 어떤 가드레일(콘텐츠/보안/PII/컴플라이언스)을 적용하는가
- 비용·지연·관측은 어떻게 관리하는가

대표 키워드:
- LLM, 프롬프트, 모델, 추론, 생성형
- 챗봇, 어시스턴트, 에이전트, RAG, **MCP**, tool/function calling
- 골든셋, 평가, judge, eval, 회귀
- 가드레일, prompt injection, 환각, hallucination
- fallback, edit/regenerate/reject
- 토큰 비용, p95 latency, 모델 교체
- **workflow vs agent, 정지조건, 구조화 출력(JSON Schema), 컨텍스트 엔지니어링, 에이전트 메모리** (§16 §5.15~5.21)

분류 시 우선순위:
- AI가 *제품 기능의 일부 또는 전부*로 동작하면 16 라우팅
- AI를 *작성·기획 도구로만* 사용하고 산출물에 AI가 포함되지 않으면 16 라우팅 *불필요*
- 16이 라우팅되면 17(평가/가드레일) 지침이 함께 짝꿍 문서로 동작

### 5.8 평가·가드레일 자체가 요청인 경우 (17 라우팅)
아래 질문이 중심이면 17 평가/가드레일 지침으로 라우팅한다.

- 조직 표준 메트릭은 무엇인가
- 골든셋을 어떻게 운영하는가
- LLM-as-Judge를 언제·어떻게 쓰는가
- 4-카테고리 가드레일은 어떻게 적용하는가
- EU AI Act 분류는 어떻게 하는가
- 인간 검토 게이트의 표준은 무엇인가

이 경우 16번을 *작성하지 않고* 17번 카탈로그 자체를 참고/갱신하는 작업이다.

### 5.9 2025-26 신규 영역 라우팅 (심화분 반영)
> 지침 12~17 심화(METH-073~078)로 추가된 주제들. 아래 키워드가 중심이면 *소유 기획서*로 라우팅한다. AI 관련 주제가 여러 문서에 걸치므로 **경계 disambiguation**에 유의.

| 요청 키워드 | 라우팅 | 소유 근거 |
|---|---|---|
| SLO·error budget·인시던트(SEV/on-call/포스트모템)·DORA·계층지원(L1/L2/L3)·**AI 프로덕션 운영**(가드레일 위반=인시던트·토큰 FinOps) | **운영 12** | §6.15~6.22 |
| **GEO/AEO**(AI 답변 인용)·포스트쿠키 측정(MMM/증분성/MER)·Growth Loops·채널 유닛경제·브랜드/퍼포먼스(60:40)·AI 광고 공시 | **마케팅 13** | §6.15~6.21 |
| Distinctive Brand Assets·포지셔닝(Dunford)·브랜드 아키텍처/네이밍·Share of Search·**브랜드 in AI 답변**·WCAG·소닉 | **브랜드 14** | §6.14~6.21 |
| 딜리버리 모델(애자일/하이브리드)·플로우 메트릭·DORA·아웃컴/OKR·RAID/프리모템·**AI 에이전트 작업 거버넌스** | **PM 15** | §6.18~6.24 |
| 에이전트 아키텍처·MCP·RAG 설계·구조화 출력·컨텍스트 엔지니어링·OWASP LLM Top10 | **AI기능 16** | §5.15~5.21 |
| NIST AI RMF·ISO 42001·레드팀·trajectory eval·거버넌스 3축 매핑 | **평가 17** | §3.7~4.6 |

**AI 주제 경계 disambiguation**(자주 혼동):
- "브랜드가 AI 답변에 어떻게 보이나" = **14**(브랜드 인식) / "우리 콘텐츠를 AI가 인용하게" = **13**(GEO/AEO) / "AI 기능 자체 설계" = **16**.
- "AI **에이전트 기능** 설계"(입출력·tool·평가) = **16** / "AI **에이전트가 한 작업**을 프로젝트로 관리"(스코핑·자율예산·throughput) = **15** / "운영 중 AI **장애·가드 위반 대응**" = **12**.

## 6. 단일 요청과 복합 요청의 처리 규칙

### 6.1 단일 요청 처리
요청이 명확히 하나의 문서 유형에 속하면 해당 문서 기준으로 바로 진행한다.

예시:
- “서비스 IA 짜줘” → 서비스기획
- “브랜드 메시지 체계 정리해줘” → 브랜드기획
- “운영 정책 문서 써줘” → 운영기획

### 6.2 복합 요청 처리
하나의 요청에 둘 이상의 문서 영역이 섞여 있으면 아래 순서로 처리한다.

1. 중심 질문을 찾는다
2. 마스터 문서를 정한다
3. 보조 문서를 정한다
4. 이번 응답에서 어디까지 다룰지 정한다
5. 필요한 후속 문서를 제안한다

### 6.3 복합 요청 예시
예시 1:
“탈모 플랫폼 사업을 기획해줘. 커뮤니티도 있고 커머스도 넣고 싶어.”

분석:
- 시장과 BM → 사업기획
- 기능 구조 → 서비스기획
- 운영 기준 → 운영기획 가능성

판단:
- 마스터 문서: 사업기획서
- 보조 문서: 서비스기획서
- 후속 문서 후보: 운영기획서

예시 2:
“앱 런칭 마케팅 전략 짜고, 브랜드 메시지도 같이 정리해줘.”

분석:
- 런칭 전략 → 마케팅기획
- 브랜드 메시지 → 브랜드기획

판단:
- 마스터 문서: 마케팅기획서
- 보조 문서: 브랜드기획서

예시 3:
“리뉴얼 프로젝트 일정표, 주간보고 체계, 이슈 관리 방식까지 같이 정리해줘.”

분석:
- 일정/보고/이슈 관리 → 프로젝트 관리 기획
- 리뉴얼 방향 자체가 포함되면 사업기획 또는 서비스기획 연계 가능

판단:
- 마스터 문서: 프로젝트 관리 기획서
- 보조 문서 후보: 사업기획서 또는 서비스기획서

### 6.4 통합본과 분리본 선택 기준
아래 조건이면 통합본 대응이 가능하다.

- 요청이 초기 검토 단계다
- 세부도가 낮다
- 의사결정용 요약이 필요하다
- 상세 실행까지 아직 가지 않았다

아래 조건이면 분리본 대응이 좋다.

- 기능/운영/브랜드/마케팅이 모두 중요하다
- 실행 가능 수준까지 구체화가 필요하다
- 각 팀의 독자가 다르다
- 수정 주기가 다르다

## 7. 마스터 문서 선정 규칙

### 7.1 마스터 문서란
마스터 문서란 해당 요청의 중심 질문을 가장 대표하는 문서다.  
프로젝트의 상위 구조와 의사결정의 중심이 된다.

### 7.2 마스터 문서 선정 질문
- 이 요청에서 가장 중요한 판단은 무엇인가
- 지금 당장 결정해야 할 것은 무엇인가
- 이 요청이 한 문서로만 남아야 한다면 어떤 문서가 적절한가

### 7.3 마스터 문서 선정 예시
- 사업 타당성, 시장, 수익 구조가 핵심 → 사업기획서
- 제품 구조와 사용자 흐름이 핵심 → 서비스기획서
- 운영 안정화와 운영 정책이 핵심 → 운영기획서
- 런칭 성과와 유입 구조가 핵심 → 마케팅기획서
- 브랜드 인식 정비가 핵심 → 브랜드기획서
- 일정, 보고, 이슈, 리스크, 산출물 통제가 핵심 → 프로젝트 관리 기획서

## 8. 보조 문서 선정 규칙

### 8.1 보조 문서란
보조 문서란 마스터 문서를 실무적으로 뒷받침하는 상세 문서다.

### 8.2 보조 문서 선정 기준
아래 중 하나라도 해당하면 보조 문서를 제안한다.

- 상위 방향만으로 실행이 어렵다
- 다른 팀이 읽어야 할 상세 기준이 필요하다
- 수정 주기가 크게 다르다
- 문서 길이가 과도하게 길어진다
- 역할 혼합 위험이 크다

### 8.3 보조 문서 추천 예시
- 사업기획서 이후 서비스 정의가 필요함 → 서비스기획서
- 서비스기획서 이후 운영 기준이 필요함 → 운영기획서
- 사업기획서 이후 런칭 구조가 필요함 → 마케팅기획서
- 사업기획서 이후 정체성 정리가 필요함 → 브랜드기획서
- 사업기획/서비스기획 이후 프로젝트 실행 통제가 필요함 → 프로젝트 관리 기획서

## 11. 문서 간 경계 관리 규칙

### 11.1 어디까지 포함하고 어디서 분리할지
AI는 각 문서에 필요한 맥락은 넣되, 원본 세부 내용은 원본 문서로 분리해야 한다.

예시:
- 사업기획서에 서비스 구조 개요는 넣을 수 있다
- 그러나 상세 기능 명세는 서비스기획서로 분리해야 한다

### 11.2 요약과 원본의 구분
요약:
- 다른 문서의 맥락을 이해시키기 위한 수준
- 핵심 포인트만 정리

원본:
- 상세 기준, 구조, 정책, 판단 근거가 담긴 본 문서

### 11.3 참조 방식
권장 문장:
- 상세 기능 구조는 서비스기획서에서 정의한다
- 운영 기준은 운영기획서에서 별도로 다룬다
- 브랜드 메시지 원본은 브랜드기획서 기준을 따른다

### 12.2 문서 분류 결과 제시

### 12.3 범위 정의

예시:
- 이번 응답에서는 사업기획서 중심으로 시장, 포지셔닝, BM까지 다룬다
- 상세 기능 구조는 후속 서비스기획서에서 분리한다

### 12.4 목차 설계
문서 유형에 맞는 목차를 먼저 제안한다.

### 12.5 실무형 초안 제시
구조화된 초안을 작성한다.

### 12.6 후속 문서 제안
필요한 경우 다음 문서 작업을 제안한다.

### 14.3 운영 기준 요청
입력 예시:
“신고 처리 프로세스와 제재 정책 정리해줘”

판단:
- 마스터 문서: 운영기획서

처리:
- 운영 절차, 제재 기준, 관리자 책임 주체 중심으로 작성

### 14.4 런칭 성장 요청
입력 예시:
“앱 런칭 마케팅 전략 짜줘”

판단:
- 마스터 문서: 마케팅기획서
- 보조 문서 후보: 브랜드기획서

처리:
- 타깃, 채널, 퍼널, KPI 중심으로 작성

### 14.5 브랜드 리포지셔닝 요청
입력 예시:
“우리 브랜드를 더 프리미엄하게 재정리해줘”

판단:
- 마스터 문서: 브랜드기획서
- 보조 문서 후보: 사업기획서

처리:
- 브랜드 정의, 포지셔닝, 메시지 체계 중심으로 작성

### 14.6 프로젝트 운영 체계 요청
입력 예시:
“프로젝트 주간보고, 일정, 이슈 관리 체계 정리해줘”

판단:
- 마스터 문서: 프로젝트 관리 기획서

처리:
- 보고 체계, 마일스톤, 이슈 로그, 리스크 관리, 의사결정 구조 중심으로 작성

## 15. 자주 하는 실수

### 15.1 모든 요청을 사업기획으로 몰아넣는 실수
상위 개념이 들어있다고 해서 모든 요청을 사업기획으로 처리하면 안 된다.

### 15.2 기능 요청을 전략 문서로 처리하는 실수
사용자 플로우와 기능 정의는 서비스기획 문서로 분리해야 한다.

### 15.3 운영 이슈를 서비스 정책에만 묶는 실수
운영 후속 처리 기준은 운영기획 문서가 원본이어야 한다.

### 15.4 브랜드와 마케팅을 혼동하는 실수
브랜드는 기준, 마케팅은 실행이다.

### 15.5 문서 순서를 무시하는 실수
사업 방향이 없는 상태에서 서비스와 마케팅을 과도하게 상세화하면 정렬이 무너질 수 있다.

## 16. 최종 판단 원칙 요약

AI는 어떤 요청이 들어와도 아래 원칙으로 판단한다.

- 먼저 요청의 목적과 단계와 독자를 파악한다
- 중심 질문을 기준으로 문서 유형을 분류한다
- 복합 요청이면 마스터 문서와 보조 문서를 나눈다
- 모든 문서를 한 번에 만들지 않고, 필요한 문서만 제안한다
- 문서 생성 순서는 프로젝트 단계와 실행 필요도에 따라 정한다
- 원본 정보는 한 문서에만 두고, 다른 문서에는 요약만 둔다
- 작성보다 분류와 구조 설계를 먼저 한다

## 17. AI 실행용 축약 지시문

앞으로 모든 기획 요청을 먼저 해석하고 분류하라.  
요청의 목적, 단계, 독자, 산출물 수준, 중심 질문을 파악하라.  
그다음 사업기획, 서비스기획, 운영기획, 마케팅기획, 브랜드기획, 프로젝트 관리 기획 중 적절한 문서 유형을 선택하라.  
복합 요청이면 마스터 문서와 보조 문서를 구분하라.  
프로젝트 단계에 따라 어떤 문서를 먼저 만들지 판단하라.  
원본 정보는 한 문서에만 두고, 다른 문서에는 요약과 참조만 허용하라.  
항상 작성 전에 분류와 구조 설계를 먼저 수행하라.
