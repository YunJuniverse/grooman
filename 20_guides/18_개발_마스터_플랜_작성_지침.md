# 18. 개발 마스터 플랜 작성 지침

> 본 지침은 서비스기획서(11)의 기능을 **ID로 참조**하고, 그 위에 빌드 순서·페이즈·게이트를 얹는 단일 문서, `MASTER_PLAN.md`의 작성 표준이다. 기능 정의 원본은 11번(SSOT) — 18은 복제하지 않고 링크한다.
> 개발자가 마스터플랜 하나만 열면 무엇을 어떤 순서로 만들어야 하는지 파악할 수 있어야 한다. 서비스기획서가 변경되면 마스터플랜을 반드시 동기화한다.

---

## 1. 문서의 목적

`MASTER_PLAN.md`는 다음을 위해 존재한다.

1. **빌드 전략의 단일 진실 출처** — 서비스기획서(11)의 기능을 **ID로 참조**하고, 그 위에 *빌드 순서·페이즈·게이트*를 얹는다. 마스터플랜이 소유하는 것은 기능 정의가 아니라 *페이즈 배치·순서·MVP 확정*이다. 기능 정의의 원본은 항상 11번(SSOT) — 18은 복제하지 않고 링크한다. 개발자는 "18에서 빌드 순서 → 링크 따라 11에서 기능 상세"로 진입한다.
   > **SSOT 원칙 (v5).** 이전 v2는 11의 기능 정의를 18에 *인라인 복제*하도록 했으나, 이는 11↔18 이중 관리·동기화 부담을 만들어 방법론의 "각 내용은 집 하나" 원칙과 충돌했다. v5부터 18은 **ID 참조 + 얇은 페이즈 요약**만 담는다. 개발자 편의를 위한 단일 진입점은 *복붙*이 아니라 *링크*로 달성한다.
2. **MVP 확정의 단일 출처** — 11번에 MVP 후보가 있어도, 무엇이 *이 빌드의* MVP인지는 18에서 확정된다.
3. **페이즈 게이트의 인스턴스화** — 15번이 표준 게이트 카탈로그라면, 18은 *이 프로젝트의* M0/M1/M2 게이트를 명시한다.
4. **B/C 변경 트리거의 사전 식별** — 어느 페이즈에서 어떤 B/C 트리거가 발동할 가능성이 있는지를 미리 매핑해서 ADR 작성 타이밍을 당긴다.
5. **AI 기능과 일반 기능의 통합 페이징** — 16번이 AI 기능 단위 명세라면, 18은 *그 AI 기능을 어느 마일스톤에 넣을지* 결정한다.

## 2. 입력 산출물

마스터플랜 작성 시 참조해야 하는 입력의 단일 목록.

| 입력 | 출처 | 사용 목적 |
|------|------|-----------|
| 비즈니스 모델 / 가치 가설 | 10 §6, §6.14 | 페이즈 가치 매핑, 비용 페이즈 분기 |
| 사용자 시나리오 / 기능 ID 목록 | 11 §6, §9 | 페이즈에 채울 기능 단위 |
| AI 기능 ID 목록 | 16 §1, §6.13 (11에서 진입점 참조) | AI 기능 페이즈 배치 |
| 게이트 카탈로그 | 15 §6.16 | 페이즈 게이트 인스턴스화 |
| 위험 등록부 | 15 §6.17 | 페이즈별 리스크 재평가 항목 |
| 평가 포트폴리오 | 15 §6.15 | AI 기능 페이즈 종료 조건 |
| 운영·관리자 도구 요건 | 12 §6 | 운영 가능 시점 페이즈 |
| 외부 시장·법규 제약 | 17 §4.4 (EU AI Act 등) | 페이즈 진입 조건 |

각 입력은 **버전(또는 commit hash)**과 함께 §2에 기록한다. 마스터플랜이 어느 시점의 기획서를 기준으로 만들어졌는지가 명확해야 한다.

## 3. 가정 / 비-목표 / 제약

### 3.1 가정 (Assumptions)
- 검증되지 않은 전제를 명시한다. 예: "사용자의 70%가 모바일 환경", "월 호출 1M 미만 가정".
- 가정이 깨지면 어느 페이즈가 영향받는지 표시.

### 3.2 비-목표 (Non-Goals)
- 이번 빌드에서 *명시적으로 하지 않는 것*. 11번의 비-목표를 그대로 가져오되, *페이즈 단위로* 다시 명시한다.
  - 예: "M0~M2까지 다국어 지원 안 함", "M1까지 결제 기능 안 함".

### 3.3 제약 (Constraints)
- 외부 차단 요인 (법무 리뷰 SLA, 보안 감사, 외부 API 계약, 모델 제공자 SLA 등).
- 인적 제약 (병행 작업 불가능한 역할, 1인 개발 가정 등).

---

## 4. 제품 비전 → 페이즈 매핑

10번/11번에서 끌어온 비전을 **시간 축**으로 변환한다.

| 비전 요소 | 출처 | 매핑된 페이즈 | 검증 지표 |
|-----------|------|--------------|----------|
| 예: "데이터 분석가가 SQL 없이 인사이트 추출" | 10 §3 | M2 | NPS ≥ 40 in 50명 베타 |

- 비전 요소 1개 = 1개 이상의 페이즈에 매핑되어야 한다.
- 어느 페이즈에도 매핑되지 않는 비전 요소는 비-목표로 옮기거나 11번을 갱신.

---

## 5. 아키텍처 결정 요약

### 5.1 컴포넌트 토폴로지
- 시스템을 박스 단위로 나열 (frontend / api / worker / vector-db / model-provider 등)
- 다이어그램은 mermaid `graph TD` 권장.

### 5.2 데이터 흐름
- 주요 read/write path 1~3개를 시퀀스로 표시.
- AI 기능은 16번의 fallback 체인을 여기서 인용.

### 5.3 외부 시스템 의존
- 모델 제공자 / 데이터 소스 / 결제 / 분석 / 인증 등.
- 각 의존마다 *이 페이즈에 대안이 있는가* 표시 (없으면 §9 리스크에 등록).

### 5.4 ADR 인덱스
- 이 마스터플랜이 의존하는 ADR 번호 나열.
- 페이즈 진입 시 재검토할 ADR도 함께 표시.

> 18은 아키텍처 *결정 요약*만 담는다. **상세 결정은 ADR**에 둔다.

---

## 6. 페이즈 / 마일스톤 분할

### 6.1 페이즈 정의

각 페이즈(M0..MN)는 다음 5개 필드를 의무로 가진다.

| 필드 | 설명 |
|------|------|
| 목적 | 1줄. 이 페이즈가 끝나면 무엇이 가능해지는가. |
| 진입 조건 | 시작 가능 조건 (이전 페이즈의 게이트 통과 + 외부 차단 해소). |
| 종료 조건 | 측정 가능한 결과 (기능 ID 목록 + 검증 지표). |
| 휴먼 게이트 | 종료 시점에 누가 무엇을 검증하는가. |
| 산출물 | 코드 외 산출물 (ADR, eval 리포트, 데모 영상 등). |

### 6.2 의존성 그래프

- 기능 간 / 기능 ↔ 외부 시스템 / AI 기능 ↔ eval 인프라 의존을 mermaid로 그린다.
- 사이클이 발견되면 하나를 깨고 ADR로 사유 기록.

### 6.3 페이즈별 산출물 매트릭스

페이즈 × 영역 표. 영역은 [기능 / 운영 도구 / 분석 / 마케팅 콘텐츠 / 테스트 / ADR / Eval 셋].

> 빈 셀이 많은 페이즈는 의도적으로 비웠는지 확인하라. 누락이라면 페이즈 분할이 잘못됐을 가능성.

## 7. MVP 확정

11번에 MVP 후보가 있어도, **이 빌드의 MVP가 무엇인지는 18에서 확정한다**.

### 7.1 in-scope 기능
- 11번 기능 ID로 나열. 새 ID를 18에서 만들지 않는다.

### 7.2 out-of-scope (V2 이후)
- 명시적으로 거부한 기능. 사유 1줄.

### 7.3 MVP 정의 게이트 (휴먼 승인 필요)
- 누가, 어떤 형태로 (PR / ADR / 회의록), 언제까지 승인하는가.
- 승인 전에는 이후 모든 페이즈가 잠정 상태.

---

## 8. AI 기능 통합 전략 (16/17과 연계)

### 8.1 AI 기능 페이즈 배치

| AI 기능 ID (16번) | 진입 페이즈 | Eval-First 게이트 | 가드레일 카테고리 |
|------------------|-------------|------------------|------------------|
| AI-001 (요약기) | M1 | M1 진입 전 골든셋 100건 통과 | A, B, C |

- AI 기능은 **eval 인프라가 준비된 페이즈 이후**에만 진입한다.
- 17번의 4-카테고리 가드레일이 모두 합격되기 전까지는 internal flag 뒤에 둔다.

### 8.2 모델 선정 / 교체 전략의 페이즈 영향
- 페이즈 시작 시 사용하는 모델 (provider/version)을 명시.
- 교체 가능 시점 (예: M2에서 자체 모델 검토).
- 교체 발생 시 어느 게이트가 재실행되는가.

### 8.3 가드레일 시점
- limited-risk → high-risk로 분류 변화 가능성이 있는 페이즈를 표시.
- EU 시장 노출 시점 (페이즈)에 17 §4.4 EU AI Act 매핑 재실행.

## 9. 리스크 & 미해결 결정

### 9.1 B/C 트리거 후보

각 페이즈에서 발생 가능성이 있는 B/C 트리거를 미리 식별한다.

| 페이즈 | 트리거 | Class | 사전 ADR 필요 |
|--------|--------|-------|---------------|
| M1 | 결제 PG 도입 | B | Y (M1 진입 전) |
| M2 | 자체 모델 호스팅 | B | Y (M1 종료 전) |
| M3 | 가격 정책 변경 | C | Y (M2 종료 전) |

> 트리거가 발동하기 전에 ADR 초안을 만들어두라. 페이즈 진입 시 시간 부족으로 ADR을 건너뛰는 것이 가장 흔한 실수다.

### 9.2 결정 보류 항목 (Decision Log)
- 현재 시점에 결정할 수 없는 항목과 *언제까지 결정해야 하는가*.
- 결정 시점이 페이즈 진입 게이트와 일치해야 한다.

### 9.3 페이즈 진입 시 재평가 항목
- 가정 (§3.1)이 여전히 유효한가
- 외부 의존성 SLA가 여전히 유효한가
- AI 모델 deprecation 일정이 여전히 유효한가

## 10. 리소스·일정 가정

### 10.1 인원·역할 가정
- 인원 수, 역할 분담, 병행 가능한 작업 / 불가능한 작업.

### 10.2 처리량(throughput) 가정과 측정 방법
- 주당 완료 가능한 TODO 수의 가정 (초기치). *velocity(포인트/스프린트)가 아니라 throughput* — flow 메트릭(`20_guides/15` §6.19)을 쓴다.
- M0 종료 시점에 실측 throughput으로 갱신하고, Monte Carlo로 "언제 끝나나"를 확률 예측한다.

### 10.3 외부 차단 요인
- 법무 리뷰 SLA, 보안 감사 일정, 외부 API 계약 협상 등.
- 페이즈 진입 조건과 직접 연결된다.

## 11. 게이트 매핑 (15 카탈로그 → 18 인스턴스)

15번 §6.16의 게이트 카탈로그를 *이 프로젝트*의 페이즈에 인스턴스화한다.

| 게이트 ID | 카탈로그 출처 | 인스턴스 (이 마스터플랜) | 승인자 | SLA |
|-----------|--------------|------------------------|--------|-----|
| master-plan-approval | 15 §6.16 | 마스터플랜 v1 승인 | Primary Approver | 5일 |
| phase-entry/M1 | 15 §6.16 | M1 진입 게이트 | Tech Lead + PM | 2일 |
| eval-spec-approval/AI-001 | 15 §6.16 | AI-001 골든셋 승인 | AI Owner | 3일 |

- 게이트 ID는 안정 키. 변경 시 영향 범위 추적 가능해야 한다.

## 12. 환류 트리거

마스터플랜은 살아있는 문서다. 다음 신호 중 하나가 잡히면 갱신한다.

### 12.1 갱신을 부르는 신호
- Eval 회귀 (3회 연속 골든셋 통과율 하락)
- 비용 초과 (예산 대비 +30%)
- 기능 재정의 (11번에 메이저 업데이트)
- 외부 SLA 변경 (모델 provider deprecation 공지 등)
- B/C 변경이 페이즈 중간에 발생한 경우

### 12.2 갱신 시 영향 범위
- 어느 기획서를 같이 손대야 하는가 (§2 입력 매트릭스 역참조).
- 진행 중 작업(TODO.md `milestone:` 태그)과의 정합성 점검.
- 갱신 결과는 *이 문서를 in-place 업데이트* 한다 (스냅샷 신설 X).

> 프로젝트당 마스터플랜은 1개. 메이저 환류 시점에는 *기존 파일을 commit*으로 갱신하고, 변경 이력을 §변경 이력에 누적한다.

## 14. 다른 문서와의 경계

### 14.1 ↔ 11 서비스기획서
- 11번: 기능 *정의의 원본*·SSOT (시나리오, 입력, 출력, 정책)
- 18번: 11번의 기능을 **ID로 참조** + 빌드 순서·페이즈·게이트를 얹는다. 기능 정의를 *복제하지 않는다*.
- 마스터플랜은 페이즈 표에 `기능 ID + 한 줄 요약 + 11번 링크`만 담는다. 시나리오·입력·출력·정책 상세는 11번에서 읽는다(중복 금지 = 동기화 부담 제거).
- **기능을 18에서 신규로 "정의"하지 않는다.** 빌드 중 새 기능이 필요하면 *11번에 먼저 정의*하고 18은 그 ID를 페이징한다(정의는 항상 11, 배치는 항상 18).
- 11번이 변경돼도 18은 ID 참조라 기능 상세를 다시 복사할 필요가 없다 — 18은 *페이즈 배치가 여전히 유효한지*만 재확인한다(범위 변동 시 페이즈 재배치).

### 14.2 ↔ 15 PM기획서
- 15번: 메타 통제 **카탈로그·표준** (게이트, 리스크(RAID), eval 포트폴리오, **딜리버리 모델(§6.18)·플로우 메트릭(§6.19)·DORA(§6.20)·OKR(§6.21)**의 정의와 방법론)
- 18번: 그 표준의 *이 프로젝트 인스턴스* — 15가 "무엇을 어떻게 측정/통제하는가"를 정의하면, 18은 "이 빌드의 페이즈에 그것을 언제 적용하는가"를 확정한다.
- 경계 규칙 (METH-076으로 15 심화 후 재확인):
  - **게이트**: 15 §6.16에서 ID 정의 → 18 §11에서 프로젝트 인스턴스화.
  - **딜리버리 모델**: 15 §6.18에서 예측/하이브리드/애자일 스펙트럼 정의 → 18은 *이 프로젝트가 어느 모델인지* §10에 선언.
  - **플로우/DORA**: 15가 지표 *정의* → 18은 지표를 *어느 페이즈 종료 조건에 거는지*만 참조(측정 방법 복제 금지).
  - **리스크**: 15 §6.17 위험 등록부 표준 → 18 §9는 *이 프로젝트의* B/C 트리거·결정 보류 인스턴스.

### 14.3 ↔ 16 AI 기능 기획서
- 16번: AI 기능 *단위 명세* (입력/출력/평가/가드)
- 18번: 그 AI 기능의 *페이즈 배치 + Eval-First 게이트 시점*
- 16에 명세가 없는 AI 기능을 18에서 페이징하지 않는다.

### 14.4 ↔ ADR
- 18에 등장하는 모든 *결정*은 ADR 링크가 있어야 한다.
- 18은 결정의 *목록과 시점*만 담고, 사유는 ADR이 담는다.

## 15. AI 시대 마스터플랜의 차이점

### 15.1 페이즈별 모델·비용 가정 명시 (의무)
- 각 페이즈가 가정하는 모델 provider/version과 월 호출량 추정을 표로 명시.
- 가정이 깨지면 페이즈 일정이 어떻게 변하는가를 함께 기록.

### 15.2 가드레일 강화 시점의 사전 계획
- limited-risk → high-risk로 변화 가능성이 있는 페이즈를 미리 표시.
- 17 §4 4-카테고리 가드레일 중 *어느 단계에서 어느 카테고리를 강화*하는지.

### 15.3 외부 모델 SLA 의존성
- 외부 모델 deprecation 일정이 페이즈 일정과 충돌 가능성을 주기적으로 점검.
- self-hosting 전환 검토 시점을 페이즈로 명시.

## 16. 자주 하는 실수

1. **기능 정의를 18에 복제** (v5 금지) — 18은 기능을 *ID 참조*만 한다. 시나리오·입력·출력을 18에 복붙하면 11과 이중 관리가 되고 반드시 어긋난다. 상세는 11 링크로.
2. **기능을 18에서 신규 "정의"** — 빌드 중 새 기능이 나오면 11번에 먼저 정의하고 18은 그 ID를 페이징한다. 정의는 항상 11, 배치는 항상 18.
3. **페이즈 게이트 누락** — 휴먼 게이트 없는 페이즈는 자동 종료될 수 없다.
4. **AI 기능을 일반 기능처럼 취급** — Eval-First 게이트를 페이즈 진입 조건에 명시적으로 넣어야 한다.
5. **B/C 트리거를 페이즈 중간에 처음 식별** — §9.1에서 사전에 매핑하지 않으면 ADR 작성 시간이 부족해진다.
6. **마스터플랜을 N개 만듦** — 프로젝트당 1개. 메이저 갱신은 *commit으로* 누적한다.
7. **입력 산출물의 버전 누락** — §2에서 어느 시점의 기획서를 기준으로 만들어졌는지 명시 안 하면 환류 영향 분석이 불가능.
8. **속도(velocity) 가정 갱신 안 함** — M0 종료 시점에 실측치로 갱신해야 이후 페이즈 일정이 신뢰 가능.

## 17. AI 작성 지시문 (축약 프롬프트)

```
[ROLE] 너는 18번 개발 마스터 플랜 작성을 보조하는 PM AI다.

[INPUT] 첨부된 11번 서비스기획서(필수), 10/15/16(필요 시 12/13/14/17) 기획서를 읽고, 다음을 산출하라.

[OUTPUT]
1. §2 입력 매트릭스 (참조 문서 + 버전)
2. §3 가정·비-목표·제약 초안
3. 11번 기능을 ID로 참조한 페이즈 배치 목록 (`기능 ID + 한 줄 요약 + 11번 링크` — 상세 정의는 복제하지 말 것)
4. §6.1 페이즈 정의 (M0..MN, 각 5필드)
5. §6.2 의존성 다이어그램 (mermaid)
6. §7 MVP 확정 후보 (in-scope/out-of-scope)
7. §8.1 AI 기능 페이즈 배치 표
8. §9.1 B/C 트리거 후보 표
9. §11 게이트 매핑 인스턴스 표

[RULES]
- 11번 기능은 ID + 한 줄 요약 + 링크로만 참조하라. 시나리오·입력·출력을 복제하지 마라 (SSOT: 정의는 11번).
- 빌드 중 새 기능이 필요하면 "11번에 먼저 정의 필요"로 표시하라. 18에서 기능을 신규 정의하지 마라.
- AI 기능은 eval 게이트가 보장된 페이즈 이후에만 배치하라.
- 페이즈 종료 조건은 측정 가능해야 한다 (기능 ID + 지표).
- 결정이 필요한 항목은 §9.2 결정 보류로 분리하라 — 추측으로 채우지 마라.
- 외부 의존성에 대안이 없으면 §9 리스크에 등록하라.
- 모든 결정 항목은 ADR 링크가 있어야 한다 — 없으면 "ADR 필요"로 표시.

[STYLE]
- 페이즈 표는 컬럼 5개 (목적/진입/종료/게이트/산출물).
- 다이어그램은 mermaid `graph TD` 또는 `flowchart LR`.
- 본문은 한국어, 코드/메타데이터는 영문.

[BOUNDARY]
- AI 기능 명세 누락 → 16번 작성 후 다시 시도하라고 안내하라.
- 게이트 ID 충돌 → 15번 카탈로그를 먼저 확인하라고 안내하라.
- 18이 페이징하려는 기능 ID가 11번에 없으면 → "11번에 먼저 정의하라"고 안내하라 (18은 정의하지 않음).
```

## 18. 실무 craft 부록 (현장 패턴 · 적용 프로젝트 환류)

> **출처**: 적용 프로젝트 GambleScan 실전 스프린트 플랜·태스크 레지스트리 정제본을 방법론으로 환류(2026-06-23, METH-040). 일반 craft만 수록.

### 18.1 코드 베이스라인 스냅샷 (재기획 금지)
- 마스터플랜의 시작점을 "재기획"이 아니라 *현존 코드의 현재 상태*로 고정한다: `기능영역 | 현재 구현 상태 | 다음 갭` 3열 표 → "이미 있는 것 *위에* 무엇을 더 짓는가". 없는 기능을 새로 정의하는 환각을 차단. §2(입력 산출물)에 *기획 문서*뿐 아니라 *코드 현황*을 페이즈 진입 전 베이스라인으로 박는다.

### 18.2 페이즈 종료 기준 이중화 + KPI 라인
- §6.1 5필드 표와 *별개로* 각 페이즈 헤더에 **종료 기준을 측정 가능한 산문 불릿**으로 선언("운영자가 로그인 가능한 콘솔을 가진다", "E2E가 동작한다") — 검증자가 읽는 게이트 체크리스트.
- 각 페이즈 말미에 **KPI 목표 한 줄**(정량 수치 묶음)을 고정 → §4 비전-페이즈 매핑의 검증 지표를 페이즈 종료 시점에 재확인 가능한 형태로 집약.

### 18.3 동적 스케줄·인라인 가드
- **조건부 작업 슬립**: 외부 차단(계약·파트너 등)이 미충족이면 *해당 작업 묶음을 뒤로 밀고 후속을 당기는* 재배치 규칙을 페이즈 본문에 인라인 명시(§10.3 차단요인 나열 → 발동 시 액션까지).
- **인라인 ⚠️ 가드**: 위험한 순서 위반(예: "법률 검토 전 특정 기능 미오픈", "선행 설계 없이 런칭 금지")을 B/C 트리거 표와 *별개로* 작업 바로 옆 ⚠️ 인용블록으로 못박음(이중 배치).

### 18.4 게이트 라이프사이클 추적
- 전체 Class C 중 *개발 단계에 아직 살아있는* 게이트만 별도 표로 분리 추적하고, 나머지는 어디로 이연됐는지(예: → Phase 5) 명시 → 게이트의 **"활성 vs 이연" 상태**를 한눈에. §11 정적 인스턴스 표를 *수명주기* 추적으로 확장.

### 18.5 개발명세 산출물 체계 (기획 ↔ 빌드 사이 층)
- 기획 산출물(요구·IA·유저스토리) 하류에 **개발명세 4종**: `data-model` / `user-flow` / `wireframe-spec` / `functional-spec` (템플릿 `50_resources/templates/`). 공통 규약:
  - ① frontmatter에 **`기반 문서`(상류 문서 명시) + 범위(Phase/Sprint)**
  - ② 말미 **Change Log(Changed Because)**
  - ③ **Out of Scope / 차기 Phase 예고** 의무 섹션(범위 크리프 차단)
  - ④ 정상 경로뿐 아니라 **Alternate(분기) / Failure(+Recovery) / Empty·Loading·Error 3-state** 강제(스피너 단독 금지, Empty state 전 명시).
