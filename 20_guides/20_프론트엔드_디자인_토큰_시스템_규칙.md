# 20_프론트엔드_디자인_토큰_시스템_규칙

> **출처**: 적용 프로젝트들의 UI 색 하드코딩 누적·드리프트(같은 회색이 6종, arbitrary hex 산재) 회고를 방법론으로 환류. 거대 디자인 시스템이 아니라 *day-1에 까는 4기둥 최소 토대*만 수록.

## 1. 문서 개요

### 1.1 문서명
프론트엔드 디자인 토큰 시스템 규칙 (Design Tokens as the Single Source of Color)

### 1.2 문서 목적
프론트엔드 프로젝트가 **첫날부터** 색·반경·그림자·모션·타이포를 *시맨틱 토큰 단일 출처*로만 쓰도록 가드레일로 강제하는 조직 표준. 색 하드코딩(arbitrary hex, off-system 회색)이 누적돼 드리프트·리브랜딩 마비를 일으키는 사후 정리를 *애초에 발생하지 않게* 하는 것이 목표다.

### 1.3 문서의 역할
지침 17(평가·가드레일) **§4.2 Guardrails-by-Construction 원칙**과 지침 19(클린아키텍처·클린코드)의 *코드 품질 가드레일*을 **UI 레이어(색·디자인 언어)**에 인스턴스화한다. 19가 *구조*(레이어·타입·파일크기) 가드레일이라면, 본 문서는 *시각 일관성* 가드레일이다. 동일 원칙: 가드레일은 프롬프트가 아니라 시스템(린트/CI/pre-push)으로 **fail-closed** 강제하며, 프롬프트 한 줄로 끌 수 없다.

### 1.4 적용 범위
프론트엔드를 포함하는 모든 트랙(웹앱·랜딩·대시보드 등). 스택 기준은 Tailwind CSS v4 `@theme` + React지만, §2·§3의 *원칙*(시맨틱 토큰·가드레일·프리미티브)은 스택 무관 이식 가능. planning-only 트랙은 비적용.

### 1.5 예상 독자
UI를 생성·수정하는 AI 모델, 코드 리뷰어, 디자인-개발 인계자.

### 1.6 관련 문서
- 지침 17 평가 및 가드레일 — §4.2 Guardrails-by-Construction(상위 원칙)
- 지침 19 클린아키텍처·클린코드 — 자매 가드레일(구조 품질). 본 문서는 *시각 품질*
- 지침 14 브랜드기획서 — 6.12 "Style Guide as Code"(토큰 값의 브랜드 출처)
- 스타터: `50_resources/skeletons/frontend-design-tokens/base/`
- 패턴: `50_resources/catalog/C-001_frontend-design-tokens.md` (active, P-002에서 승급)

## 2. 핵심 원칙 — 4기둥만 (거대 시스템 금지)

> **색은 한 곳에서만 정의되고, 코드는 *역할*로만 색을 부른다.** `bg-[#1a1a2e]`·`text-gray-500`을 *커밋 단계에서* 막으면, 리브랜딩 한 번에 토큰 값만 바꾸면 전 화면이 따라온다.

거대 디자인 시스템(수백 컴포넌트·문서 사이트)을 짓지 않는다. **딱 4기둥**:

| 기둥 | 무엇 | 산출물 |
|------|------|--------|
| **① 토큰** | 시맨틱 색(surfaces/text/border/brand/semantic) + 디자인 언어(radius/shadow/motion/type)를 `@theme`에 단일 정의 | `theme/tokens.css` |
| **② 프리미티브** | `cn()`(twMerge+clsx) + 최소 3종(Card/Button/Badge). 토큰만 소비 | `lib/cn.ts`, `components/primitives/` |
| **③ 가드레일** | arbitrary hex·off-system 회색을 빌드 fail시키는 스크립트를 lint:tokens + CI + pre-push에 연결 | `guardrails/check-no-arbitrary-color.sh`, `wiring.md` |
| **④ 제약 문서** | 이 프로젝트의 토큰 표 — AI 생성 시 참조하는 단일 제약 | `design-system.md` |

**원칙: 이름 = 역할.** 토큰 이름은 *값*이 아니라 *역할*을 나타낸다. `--color-surface-raised`(O) / `--color-slate-800`(X). 프로젝트 브랜드에 맞게 바꾸는 것은 **값뿐**, 이름(역할)은 유지한다 → 스켈레톤 간 이식성 보존.

| | 권장 (✓) | 지양 (✗) |
|------|----------|----------|
| 색 정의 위치 | `@theme` 시맨틱 토큰 1곳 | 컴포넌트마다 인라인 hex |
| 코드의 색 호출 | `bg-surface-raised text-text-primary` | `bg-[#1e293b] text-gray-400` |
| 강제 메커니즘 | 가드레일 스크립트(fail-closed) | 리뷰 구두 지적 |
| 의도적 예외 | `ALLOW_HEX` allowlist에 근거와 함께 | `<!-- eslint-disable -->` 남발 |

## 3. 4개 시맨틱 색 그룹 (이름=역할)

값은 프로젝트마다 다르되, **그룹·역할 이름은 고정**한다.

| 그룹 | 토큰(역할) | 용도 |
|------|-----------|------|
| **surfaces** | `surface-base` / `-raised` / `-overlay` / `-sunken` / `-inverse` | 페이지 바닥 / 카드·패널 / 모달·팝오버 / 입력 well / 대비 반전 |
| **text** | `text-primary` / `-secondary` / `-muted` / `-inverse` / `-on-brand` | 본문 / 보조 / 흐림 / 반전 배경 위 / 브랜드 배경 위 |
| **border** | `border-subtle` / `-default` / `-strong` / `-focus` | 구분선 / 기본 / 강조 / 포커스 링 |
| **brand** | `brand` / `-hover` / `-active` / `-subtle` | 주 액션 / hover / active / 배경 틴트 |
| **semantic** | `success` / `warning` / `danger` / `info` (+ 각 `-subtle`) | 상태 표현(성공·경고·위험·정보) |

**디자인 언어 토큰**(색이 아닌 시각 언어): `--radius-*`(sm/md/lg/full), `--shadow-*`(sm/md/lg), 모션(`--ease-*`/`--duration-*`), 타이포(`--font-*`, type scale `--text-*`). 이것도 인라인 magic number 대신 토큰으로.

> **회색 함정**: `text-gray-N`·`bg-slate-N`·`zinc-N`은 *off-system 팔레트*다 — 토큰을 우회한 회색이 곧 드리프트의 90%. 회색이 필요하면 거의 항상 `surface-*`·`text-*`·`border-*` 중 하나로 표현된다. 가드레일이 이를 차단한다.

## 4. 가드레일 (fail-closed) — §4의 핵심

day-1에, **코드를 쓰기 전에** 켠다. 스크립트가 잡는 위반:

| 위반 | 패턴 예 | 판정 |
|------|---------|------|
| arbitrary hex | `bg-[#1a1a2e]`, `text-[#fff]` | **fail** |
| off-system 팔레트(전 family) | `text-gray-500`, `bg-slate-800`, `border-zinc-300`, **`bg-amber-400`, `text-blue-500`, `border-rose-300`** … | **fail** |
| 의도적 예외 | `ALLOW_HEX`에 등록된 hex(순백 max-contrast, 메달 gold/silver/bronze 등) 또는 시맨틱 토큰 | **pass** |

> **off-system은 회색만이 아니다(C-001 교훈②).** 가드레일은 *전 prefix*(`bg/text/border/ring/from/to/shadow/…`) × *전 Tailwind 팔레트 family*(gray·slate… 뿐 아니라 red·orange·amber·yellow·…·rose)를 검사한다. `text-` 회색만·gray-family만 검사하면 비-text·비-회색 색이 게이트를 통과해 누적된다(실세계: gamblescan에서 회색 32건 + amber/orange 251건 누출). 의도적 categorical/메달 색은 토큰 또는 `ALLOW_HEX`로 *명시* 분리한다.

연결 3지점 (`guardrails/wiring.md` 참조):
1. `package.json` → `"lint:tokens": "bash guardrails/check-no-arbitrary-color.sh"`
2. CI(PR 게이트) → lint:tokens 실패 시 머지 차단
3. pre-push 훅 → push 직전 로컬 차단

> 가드레일을 켠 직후 **더미 위반 1건**(예: `<div className="bg-[#123456]">`)으로 *실제로 빌드가 깨지는지* 검증하고 보고한다. 안 깨지면 가드레일이 죽은 것.

### 4.1 C-001 — 기존 프로젝트 리트로핏(빅뱅 금지)
하드코딩이 이미 누적된 프로젝트는 점진·검증 가능하게:
1. **실측**: `grep -roh '-\[#[0-9a-fA-F]\{6\}\]' src | sort | uniq -c | sort -rn` 으로 지배값/드리프트 파악. 토큰 시스템이 여럿(:root/@theme/config)이면 충돌부터 식별.
2. **토큰 정의**: `@theme` 시맨틱 토큰을 *지배적 사용값*으로 → 렌더 byte-identical(무변경). hex→토큰 매핑표 작성.
3. **codemod**: arbitrary hex → 토큰 유틸 치환. exact=무변경, 드리프트=가장 가까운 토큰으로 의도적 흡수(델타 보고). 프리뷰 불가 환경이면 *빌드 CSS 색값 비교*로 회귀 게이트.
4. 프리미티브·가드레일 추가(재발 차단), 화면 단위 슬라이스 PR(빅뱅 금지).

핵심: 지배값=토큰값이라 대부분 무변경이어야 한다. 변하는 것(드리프트 흡수)은 건수·델타를 보고한다.

## 5. 운영 트리거 (복사용 지시문)

AI에게 작업을 시킬 때 그대로 전달하는 트리거. 기준: 본 문서, 스타터 `50_resources/skeletons/frontend-design-tokens/base/`.

### A. 신규 프로젝트 (greenfield) — 토대 주입
```
이 프로젝트에 디자인 토큰 토대를 day 1에 깐다. 거대 시스템 금지 — 딱 4기둥만.
```
1. `theme/tokens.css`의 `@theme` 시맨틱 토큰(surfaces/text/border/brand/semantic) + 디자인 언어 토큰(radius/shadow/motion/type)을 주입. 값은 이 프로젝트 브랜드에 맞게 *값만* 조정(이름=역할 유지).
2. `cn.ts`(twMerge+clsx) + primitives(Card/Button/Badge) 주입. deps: `clsx`, `tailwind-merge`.
3. 🔑 가드레일을 *코드 쓰기 전에* 켠다 — `guardrails/check-no-arbitrary-color.sh`를 `package.json` `lint:tokens` + CI(PR 게이트) + pre-push 훅에 등록(`guardrails/wiring.md`). arbitrary 색·off-system 회색 발견 시 빌드 fail.
4. `design-system.md`를 이 프로젝트 토큰 표로 채워 AI 생성 제약 문서로 둔다.
- 규칙: 이후 모든 UI는 토큰·프리미티브로만. 의도적 예외는 `ALLOW_HEX`로만. 끝나면 더미 위반으로 가드레일 1회 검증·보고.

### B. 기존 프로젝트 — 리트로핏 (하드코딩 이미 누적)
```
이 프로젝트의 색 하드코딩을 토큰 단일 소스로 리트로핏한다. 빅뱅 금지 — 점진·검증 가능하게.
```
→ §4.1(C-001) 4단계 그대로. 실측 → 지배값 토큰화(무변경) → codemod(드리프트 흡수 보고) → 가드레일 + 슬라이스 PR.

### C. 방법론 학습 훅
- 이 패턴은 **이미 active Catalog `C-001`로 승급**됨(gamblescan 실세계 + 방법론 canonical, N≥2) → 스켈레톤 `bakes-in.json`에 합류(자동 주입). 새 도메인에서 또 겪으면 `C-001`에 그 날짜를 `seen_in` hit으로 추가.
- 모델/스택이 다르면 `C-001`의 `verified_with`/`deps_implicated` 갱신.

## 7. 안티패턴·교훈 (정직 모드)

- **"일단 hex" → 드리프트**: 같은 회색이 6종으로 갈라진 뒤의 정리는 day-1 가드레일보다 수십 배 비싸다.
- **가드레일 미검증**: 켰다고 끝이 아니다 — 더미 위반으로 *실제 차단*을 확인하지 않으면 죽은 가드레일.
- **이름에 값 박기**: `--color-slate-800` 같은 값-이름 토큰은 리브랜딩 때 *이름까지* 거짓이 된다. 역할로 명명.
- **거대 시스템 욕심**: 4기둥을 넘는 컴포넌트 라이브러리·문서 사이트는 day-1 범위 아님 — 필요해질 때 승급.
- **회색 예외 남발**: `ALLOW_HEX`는 *진짜 의도적 예외*(순백 max-contrast, 메달 gold/silver)만. 귀찮아서 넣으면 가드레일이 무력화.
- **빅뱅 리트로핏**: 기존 프로젝트를 한 PR로 전환 금지 — 화면 단위 슬라이스 + 회귀 게이트(렌더 비교).

## 8. 상호운용 표준 — W3C DTCG (필요할 때만, METH-094)

> 4기둥은 *한 코드베이스 안*의 규율이다. **여러 도구·플랫폼이 토큰을 주고받아야 할 때**(Figma ↔ 코드, 웹+iOS+Android 공유, 디자이너가 토큰 소유)만 표준 교환 포맷을 도입한다. 단일 웹앱이면 4기둥으로 충분 — 아래는 *승급 시점*의 지도.

- **표준**: **W3C Design Tokens Community Group(DTCG)** — 토큰의 JSON 교환 포맷. 핵심: `$value`·`$type`(color/dimension/fontFamily/shadow 등)·`$description`, 그룹 중첩, **별칭 참조 `{group.token}`**(= 시맨틱→프리미티브 앨리어싱을 표준 문법으로), composite 타입(typography·shadow·gradient), 테마/모드.
- **툴**: **Style Dictionary**(빌드 — DTCG JSON → CSS 변수·iOS·Android·JS 다중 타깃 생성) · **Tokens Studio**(Figma 플러그인 — 디자이너가 토큰 소유, DTCG로 export). 이 둘이 "디자인↔코드 단일 출처" 파이프라인의 사실상 표준.
- **우리 4기둥과의 매핑**: §2 시맨틱 토큰 = DTCG 시맨틱 레이어(`$value: "{color.slate.800}"` 앨리어스), §3 4색 그룹 = DTCG color 그룹. 즉 **4기둥은 DTCG의 최소 부분집합** — 나중에 DTCG로 승격해도 이름·역할 구조가 그대로 이어진다(리트로핏 아님).
- **도입 판정(승급 트리거)**: ① 멀티플랫폼(웹+네이티브) ② 디자이너가 토큰을 Figma에서 직접 소유 ③ 토큰을 여러 repo가 소비. 하나라도 참이면 DTCG+Style Dictionary. **셋 다 거짓이면 4기둥 유지**(거대 시스템 욕심 = §7 안티패턴).
- **가드레일 연속성**: DTCG로 가도 §4 fail-closed 가드레일(하드코딩 차단)은 그대로 — 표준은 *교환 포맷*이지 *규율 대체*가 아니다.
