# 02. 식별자 및 버전 관리 규칙

> 이 문서는 프로젝트 전반에서 사용하는 모든 식별자·버전·번호의 **부여 기준과 형식**을 정의한다.
> 새 식별자를 만들기 전에 이 문서를 먼저 확인한다.

---

## 1. 전체 구조 관계

```
페이즈 (M0, M1 ...)              전략 단위 — 사용자에게 가치를 전달할 수 있는 기능 묶음 (MASTER_PLAN)
  └── TODO (PREFIX-001 ...)      작업 단위 — 페이즈에 속한 개별 작업 (TODO 칸반)
```

> **2층 구조 (METH-086).** 이전 3층(페이즈→스프린트→TODO)의 중간 "스프린트" 층은 폐지됐다. solo+AI에서 스프린트는 잉여 중간층이자 명칭 모순(기간 고정 안 하는 sprint)이었다. cadence·예측은 **flow 메트릭**(WIP·cycle time·throughput·Monte Carlo, `20_guides/15` §6.19~6.20)이 담당하고, 배치 그룹핑이 필요하면 TODO 항목에 `milestone: M1` 태그를 단다. 게이트는 페이즈(MASTER_PLAN)가 소유한다.

번호는 **프로젝트 전체에서 연속**으로 이어진다. 페이즈가 바뀌어도 리셋하지 않는다.

---

## 2. 페이즈 (M0, M1 ...)

### 형식
`M` + 0부터 시작하는 정수. `M0`, `M1`, `M2` ...

### 부여 기준
| 페이즈 | 의미 | 진입 조건 |
|--------|------|-----------|
| **M0** | 프로젝트 초기화 · 기획 완료 | 항상 존재. 코드 없음. 서비스기획·개발명세서 승인이 종료 조건. |
| **M1+** | 사용자에게 실제로 가치를 전달할 수 있는 기능 묶음 완성 | 이전 페이즈 휴먼 게이트 통과 |

### 페이즈 경계 판단 기준
- 이 묶음이 완성되면 **사용자가 실제로 쓸 수 있는가?**
- Yes → 새 페이즈의 종료 단위
- No → 같은 페이즈 안의 TODO 작업으로 처리

### 규칙
- 페이즈는 MASTER_PLAN.md에서 정의한다. TODO.md에서 임의로 신설하지 않는다.
- 페이즈 번호는 재사용하지 않는다. 취소된 페이즈는 `cancelled` 상태로 둔다.
- 페이즈 진입·종료는 반드시 휴먼 게이트를 통과해야 한다.

---

## 3. TODO ID (PREFIX-NNN)

### 형식
`[프로젝트 접두어]-` + 3자리 제로패딩 정수.
- 접두어: 프로젝트명 약어 2~5자 대문자. 예: `MYAPP`, `GBLE`, `SCAN`
- 예: `MYAPP-001`, `MYAPP-042`

### 부여 기준
- 실제로 작업할 단위로 분해될 때 부여한다.
- 아이디어 단계(Backlog)에서는 임시 제목만 있어도 된다. ID는 Ready로 올라올 때 확정한다.

### 규칙
- 한번 부여한 ID는 변경하지 않는다 (stable ID).
- 완료·취소된 항목의 ID는 재사용하지 않는다.
- ID는 TODO.md 안에서 유일해야 한다.

### Backlog vs Ready 기준
| 상태 | 조건 |
|------|------|
| **Backlog** | 아이디어 수준. acceptance criteria 미완성. |
| **Ready** | acceptance criteria 확정. 선행 의존성 해소 확인. 담당자 확정. (선택: `milestone: M1` 태그) |

---

## 4. ADR (ADR-NNN)

### 형식
`ADR-` + 3자리 제로패딩 정수. `ADR-001`, `ADR-002` ...

### 부여 기준 — 다음 중 하나 해당 시 작성
- **Class B 변경** 자동 트리거 (DB migration, 새 외부 API, 인증 변경, destructive data, 백그라운드 잡)
- **Class C 변경** 자동 트리거 (가격, 법무, 브랜드, 공개 릴리스, 외부 약속)
- 코드만으로는 **"왜 이렇게 결정했는가"를 역산할 수 없는** 아키텍처 결정
- 두 가지 이상의 선택지를 비교 검토한 결정

### 부여하지 않는 경우
- Class A 변경 (일반 기능 구현·버그 수정)
- 코드 자체가 이유를 설명하는 결정
- PR description으로 충분한 단순 선택

### 규칙
- ADR은 `40_dev/adr/ADR-NNN-제목.md` 에 저장한다.
- 한번 `accepted` 된 ADR은 삭제하지 않는다. 결정이 바뀌면 새 ADR을 만들고 이전 ADR을 `superseded by ADR-NNN`으로 표시한다.
- MASTER_PLAN.md의 모든 결정 항목은 ADR 링크가 있어야 한다. 없으면 "ADR 필요"로 표시.

## 5. 문서 버전 (vN)

### 형식
`v` + 1부터 시작하는 정수. `v1`, `v2`, `v3` ...

### 부여 기준 — 버전업하는 경우
- 기획 내용이 **실질적으로 변경**될 때 (기능 추가·삭제·정책 변경)
- 휴먼 게이트 승인 후 내용이 바뀔 때
- 재기획(re-plan) 결과로 기존 내용이 달라질 때

### 버전업하지 않는 경우
- 오탈자·띄어쓰기·표현 수정
- 서식·레이아웃 변경
- 링크 업데이트

### 스냅샷 파일명 규칙
```
40_dev/snapshots/plans/{type}/v{N}-{YYYY-MM-DD}.md
40_dev/snapshots/dev-specs/v{N}-{YYYY-MM-DD}.md
```

### frontmatter 규칙
모든 기획 문서 상단에 아래 필드를 포함한다.
```yaml
---
version: v1
last_updated: YYYY-MM-DD
status: draft | active | superseded | archived
---
```

## 6. AI 기능 ID (AI-NNN)

### 형식
`AI-` + 3자리 제로패딩 정수. `AI-001`, `AI-002` ...

### 부여 기준
- `16_AI_기능_기획서`에서 AI 기능 단위 명세가 작성될 때 부여한다.
- 일반 기능(AI 미포함)에는 부여하지 않는다.

### 규칙
- AI 기능 ID는 MASTER_PLAN.md §8.1의 AI 기능 페이즈 배치 표에서 참조된다.
- 16번에 명세가 없는 AI 기능은 MASTER_PLAN.md에 페이징하지 않는다.
- 한번 부여한 ID는 변경하지 않는다. 기능이 폐기되면 `deprecated` 상태로 표시.

## 7. 지침 문서 번호 (guide-NN) — 번호 범위 예약

### 형식
`20_guides/NN_<이름>.md`, frontmatter `doc_id: guide-NN`. 2자리 제로패딩.

### 번호 범위 (예약)
- **00–89 — 상류(방법론 레포) 소유.** `sync`로 다운스트림에 전파되는 공용 지침. 다운스트림은 *이 범위에 파일을 새로 만들지 않는다* — sync 시 상류 파일과 번호·`doc_id`가 충돌한다.
- **90–99 — 다운스트림 프로젝트-커스텀 예약.** 특정 프로젝트에만 필요한 지침은 이 범위에서 부여(`doc_id: guide-9N`). 상류가 절대 점유하지 않으므로 sync 안전.

### 규칙
- 상류 신규 지침 번호는 *상류 내 연속·미중복*만 확인하면 된다. 다운스트림-커스텀은 90+ 예약이 충돌을 막는다.
- 다운스트림-커스텀 지침은 반드시 90+.
- **비준수 레거시 커스텀**(예: ai-icons·icons-invest `20_guides/04_프로젝트_문서_보관_및_분류_규칙`, doc_id `guide-04`)은 이 규약 신설 *전*에 만들어진 것. 다음 정비 시 90+로 마이그레이션 대상이며, 마이그레이션 전까지 상류는 그 번호(04)를 피한다. (그래서 상류 산출물 채널 분리 지침은 04가 아닌 **05**로 부여됨 — METH-053.)

## 8. 전체 식별자 요약표

| 식별자 | 형식 | 단위 | 새로 만드는 시점 | 저장 위치 |
|--------|------|------|-----------------|-----------|
| 페이즈 | M0, M1 ... | 기능 묶음 (사용자 가치 전달 가능 단위) | 새로운 사용자 가치 묶음이 정의될 때 | MASTER_PLAN.md |
| TODO | PREFIX-001 ... | 개별 작업 | Backlog → Ready 전환 시 확정 | TODO.md |
| ADR | ADR-001 ... | 아키텍처 결정 | Class B/C 또는 역산 불가 결정 | 40_dev/adr/ |
| 문서 버전 | v1, v2 ... | 기획서·명세서 | 내용 실질 변경 시 | 각 문서 frontmatter |
| AI 기능 | AI-001 ... | AI 기능 단위 | 16번 기획서 작성 시 | 16_AI_기능_기획서 |
| 지침 번호 | guide-NN | 지침 문서 | 신규 지침 작성 시 (상류 00–89 / 다운스트림 90–99) | 20_guides/ |
