# 16_AI_기능_기획서_작성_지침

## 1. 문서 개요

### 1.1 문서명
AI 기능 기획서 작성 지침

### 1.2 문서 목적
본 문서는 AI(LLM/에이전트/생성형 모델)가 제품 기능의 일부 또는 전부를 차지하는 경우, 그 기능을 어떻게 정의·평가·운영할지를 정의하는 표준 작성 기준을 제공한다.
서비스기획서가 사용자 흐름과 기능 구조의 원본이라면, AI 기능 기획서는 **AI 기능의 입력·출력 계약, 평가 기준, 가드레일, fallback, 비용·관측 구조의 원본**이다.

### 1.3 문서의 역할
이 문서는 서비스기획서의 자식 문서다.
서비스기획서가 "사용자가 무엇을 하는가"를 정의한다면, AI 기능 기획서는 그중 AI가 관여하는 지점에 대해 **"무엇을 입력으로 받아 무엇을 출력하며, 틀렸을 때 어떻게 동작하는가"** 를 정의한다.

### 1.4 적용 범위
본 지침은 아래 경우에 적용한다.

- LLM 기반 챗봇/어시스턴트
- 생성형 콘텐츠(텍스트/이미지/음성/코드) 기능
- 분류·추출·요약 기능
- 추천/랭킹에 LLM이 관여하는 기능
- AI 에이전트(다단계 도구 호출) 기능
- RAG 기반 답변 기능
- 자동 의사결정 기능 (사람 검토 없이 시스템에 영향 주는 경우)

### 1.5 적용하지 않는 경우
다음 경우에는 본 문서가 필요하지 않다.

- AI를 *작성 도구*로만 쓰고 산출물에 AI가 포함되지 않을 때 (`ai_relevance: tool`)
- 결정론적 룰엔진/통계 모델만 사용할 때
- 외부 SaaS의 AI 기능을 그대로 임베드하기만 할 때 (이 경우 운영기획서로 충분)

### 1.6 예상 독자
- PM/PO
- AI 엔지니어
- 백엔드/프론트엔드 개발자
- 평가(QA) 담당자
- 운영팀
- 보안/컴플라이언스 담당자

## 2. AI 기능 기획서의 정의

### 2.1 기본 정의
AI 기능 기획서는 **AI가 관여하는 단일 기능 단위에 대해, 입력 contract / 출력 contract / 평가 / 가드레일 / fallback / 운영 관측을 정의하는 실행 명세 문서**다.

### 2.2 한 문장 정의
"AI가 무엇을 입력으로 받아 무엇을 출력하고, 잘못됐을 때 어떻게 행동하며, 잘 됐는지 어떻게 측정하는가"를 정의하는 문서.

### 2.3 다른 문서와의 차이
- 서비스기획서: 사용자 시나리오와 화면 정책 — AI 기능 기획서는 그 *AI 부분만* 깊게 본다
- 운영기획서: 운영자/관리자 처리 절차 — AI 기능 기획서는 *시스템 동작*을 정의한다
- 평가·가드레일 지침(17): 조직 표준과 카탈로그 — AI 기능 기획서는 *이 기능에 적용된 인스턴스*

## 3. AI 기능 기획서의 목적

1. AI 기능의 입력·출력을 명세 가능한 계약으로 만든다
2. 평가 기준을 *구현 전에* 합의한다 (Eval-First)
3. 가드레일을 아키텍처 차원에서 강제한다
4. 실패·환각·지연 시 사용자 경험을 정의한다
5. 비용·지연·관측 가능성을 운영 가능한 수준으로 만든다

## 4. 핵심 질문

AI는 AI 기능 기획서를 작성할 때 반드시 아래 질문에 답해야 한다.

### 4.1 AI 가치 가설 (Value Hypothesis)
- 왜 *반드시* GenAI/LLM이어야 하는가
- 룰/검색/통계 모델로 대체 가능한 부분은 어디까지인가
- AI 도입으로 얻는 사용자 가치를 한 줄로 어떻게 표현하는가

### 4.2 입력·출력 계약 (I/O Contract)
- 입력 형식과 필수 필드는 무엇인가
- 출력 스키마는 무엇인가 (구조화 출력 권장)
- 길이·언어·형식 제약은 무엇인가

### 4.3 평가 기준 (Eval Spec)
- 무엇을 "성공"으로 정의하는가
- 골든 데이터셋은 어디 있는가, 몇 개인가
- 누가 채점하는가 (룰 / LLM-as-Judge / 사람)
- 회귀 기준은 무엇인가

### 4.4 실패 모드 (Failure Modes)
- AI가 환각/오답을 낼 가능성이 어느 정도인가
- 사용자가 잘못된 출력을 받았을 때 어떻게 인지하고 어떻게 대응하는가
- 완전 실패(타임아웃/모델 다운) 시 fallback은 무엇인가

### 4.5 가드레일 (Guardrails)
- 입력 측 가드: prompt injection, PII, 금지 주제
- 출력 측 가드: 콘텐츠 안전, 형식 검증, 사실 검증
- 행동 측 가드: tool 사용 권한, write-action 승인

### 4.6 비용·지연 (Cost & Latency)
- 단일 호출 비용 / 평균 토큰
- p50/p95 지연 목표
- 월간 예상 호출량과 비용 캡

### 4.7 관측 (Observability)
- 무엇을 로깅하는가 (입력, 출력, 모델, 토큰, 평가 점수)
- 누가 트레이스를 보는가
- 어떤 알람을 켜는가 (비용 폭주, 오답률 증가, 지연 급증)

## 5. 포함 범위

### 5.1 기능 개요
- 기능명
- 한 줄 정의
- 부모 서비스기획서 링크
- AI 가치 가설
- ai_relevance / guardrail_class

### 5.2 사용자 시나리오 (AI 관점)
- AI 호출 트리거 (사용자 액션, 자동, 스케줄)
- 사용자가 결과를 받기까지의 흐름
- 결과 수정/재생성/거절 액션 (의무 — 5.7 참조)

### 5.3 입력 계약
- 입력 스키마 (JSON Schema 또는 OpenAPI)
- 필수/선택 필드
- 입력 정제(redaction, normalization)
- 컨텍스트 주입 방식 (RAG라면 검색 소스, 청크 크기, 임베딩 모델)

### 5.4 모델·프롬프트 설계
- 사용 모델 (provider, model id, 버전 핀)
- 프롬프트 핵심 의도 (system prompt 의도, few-shot 사용 여부)
- 프롬프트 자체는 별도 파일로 관리 (frontmatter `machine_readable_links.prompts`)
- 결정론 정도 (temperature, seed 사용 여부)

### 5.5 출력 계약
- 출력 스키마 (구조화 출력이 가능하면 강하게 권장)
- 길이/형식 제약
- 출력 검증 단계 (schema validation, content filter, 사실 검증)
- 출력 메타데이터 (모델 ID, 평가 점수, 토큰 수)

### 5.6 평가 명세 (Eval Spec)
- 골든 데이터셋 위치와 크기 (권장 최소 100개)
- 메트릭 (정확도, 정밀도, BLEU/ROUGE, LLM-as-Judge 점수, latency, cost 등)
- 채점 방식 (rule / LLM-as-Judge / human)
- LLM-as-Judge 사용 시 judge 프롬프트 위치와 calibration 데이터
- 합격 기준 (예: judge 점수 4/5 이상 90%, p95 latency 3s 이하)
- 회귀 기준 (이전 버전 대비 어느 메트릭이 어느 정도 떨어지면 차단)

### 5.7 AI 실패 UX (의무 항목)
사용자가 잘못된 출력을 받았을 때 반드시 다음 액션 중 하나 이상이 가능해야 한다.

- **Edit** — 사용자가 결과를 직접 수정
- **Regenerate** — 다시 생성 요청
- **Reject** — 결과를 버리고 원상복귀
- **Escalate** — 사람 검토로 전환 (high-stakes)

또한 다음을 명시한다.
- AI 출력임을 사용자에게 어떻게 표시하는가 (라벨, 출처, 신뢰도)
- 사용자가 잘못된 출력으로 손해를 본 경우 책임 경계
- 환각 가능성에 대한 사용자 고지 방식

### 5.8 가드레일 (4-카테고리)
17번 지침의 표준 카탈로그를 참조하여 본 기능에 적용한 인스턴스를 명시한다.

- **콘텐츠 안전** — 유해/혐오/성인/폭력 출력 차단
- **보안** — prompt injection, jailbreak, tool abuse 방어
- **데이터 보호** — PII 마스킹, secret leak 차단
- **컴플라이언스** — 도메인 법규 (의료/금융/EU AI Act 등)

각 카테고리별로 (1) 어떤 통제가 적용되는가 (2) 통제 위반 시 시스템 동작은 무엇인가를 적는다.

### 5.9 Fallback 체인
1차 시도 → 2차 시도(다른 모델/프롬프트) → 룰 기반 → 사람 → 정중한 거절
각 단계의 트리거 조건과 대기 시간 임계값을 명시한다.

### 5.10 Tool/Action 권한 (에이전트인 경우)
3-Tier 경계로 정의한다.

- ✅ **Always** — 사람 승인 없이 실행 가능 (예: 읽기, 검색)
- ⚠️ **Ask** — 사용자/사람 승인 후 실행 (예: 결제 정보 입력, 메시지 발송)
- 🚫 **Never** — 절대 실행 금지 (예: 비밀번호 입력, 자금 이체, 공개 게시 자동화)

### 5.11 비용·지연
- 단일 호출 평균 토큰 (입력/출력)
- 단일 호출 비용 추정
- p50/p95 latency 목표
- 월간 호출량 가설 / 비용 캡 / 알람 임계

### 5.12 관측 명세
- 트레이스에 기록할 필드 (request_id, user_id_hash, model, tokens, latency, cost, eval_score)
- 보존 주기와 PII 처리
- 대시보드 위치
- 알람 룰

### 5.13 모델·프롬프트 변경 절차
- 모델 교체는 Class B (회귀 평가 통과 필수)
- 시스템 프롬프트 변경은 Class B
- few-shot 변경은 Class A (단, 회귀 평가는 권장)
- 프롬프트 버전 관리와 롤백 절차

### 5.14 데이터·학습 정책
- 사용자 입력을 학습에 사용하는가 (yes/no, opt-in/out)
- 외부 모델 제공자에게 데이터를 보내는가, 어디까지 보내는가
- 보존 기간

### 5.15 에이전트 아키텍처 (§5.10 Tool 권한의 상위 결정)
> **Workflow냐 Agent냐 게이트**(Anthropic 'Building Effective Agents'): 고정 제어흐름=**워크플로우**(프롬프트 체이닝·라우팅·병렬·orchestrator-worker·evaluator-optimizer, *코드가 흐름 소유*). 동적 라우팅·개방형 탐색 필요할 때만 **에이전트**("LLM이 루프에서 tool 사용"). 가장 단순한 것부터.
- 에이전트면 명시: **loop 패턴**(ReAct / plan-execute / reflection) · 단일 vs 멀티(orchestrator-worker) · **정지 조건**(max iterations·max tool calls·토큰/비용 캡·no-progress·사람 에스컬레이션). → 지침 07 자율예산의 feature-level 인스턴스 · OWASP **LLM10**(무한 소비) 방어.

### 5.16 에이전트 메모리
- **short-term**(컨텍스트/작업 상태) vs **long-term**(벡터스토어·파일·노트, 세션 넘어 지속): 무엇을 저장·검색 트리거·TTL/축출·**메모리 내 PII** 처리.

### 5.17 MCP 통합 (§5.10 매트릭스를 통합 계약으로)
> **MCP**(Model Context Protocol, 모델↔도구/데이터 표준 — "AI용 USB-C"). 서버 primitive: tools·resources·prompts(+sampling·elicitation).
- 각 외부 tool/데이터마다: 쓰는 **MCP 서버/tool/resource**(vs 자체 통합) · **인증**(OAuth Resource Server·RFC 8707 Resource Indicators) · **스코프**(최소권한) · tool 응답을 **미신뢰 입력**으로 취급(프롬프트 인젝션 표면 = LLM01).

### 5.18 RAG 설계 + RAG-eval (RAG 기능 시 — §5.6 Eval Spec 심화)
- **검색 설계**: chunking·임베딩 모델·하이브리드(키워드+벡터)·reranking · **grounding/citation** 의무.
- **RAG-eval(RAGAS)**: 검색=context precision/recall · 생성=**faithfulness(groundedness)**·answer relevancy → 인수 임계로. (faithfulness+context recall이 최고 신호). 단일 검색 부족 시 **agentic RAG**(검색을 에이전트 루프 내 tool로).
- 조직 eval 카탈로그는 17번, 여기는 *이 기능*의 검색 품질 바.

### 5.19 구조화 출력 메커니즘 (§8.3 심화)
> "권장"에서 *메커니즘*으로: **JSON Schema(Pydantic/Zod) + constrained decoding**으로 *스키마 유효 보장*("보통 유효" 아님). 모드: `response_format`(사용자 대면 답) vs **strict function calling**(앱 함수 호출). 실패 계약: refusal·length-truncation 시 동작(스키마 유효 ≠ 의미 정확 — 검증 별도). OWASP **LLM05**(부적절 출력 처리) 방어.

### 5.20 컨텍스트 엔지니어링 + 모델 적응 결정
> 프롬프트 엔지니어링의 후속(Anthropic): 컨텍스트 창 *전체* 관리로 "고신호 최소 토큰".
- **컨텍스트 관리**: 컨텍스트 예산 + prompt caching·compaction/요약(장기 작업)·메모리·서브에이전트 격리 중 무엇을 쓰나. **context rot**(길수록 성능 저하 — 더 많은 토큰≠좋음) 인지. 프로젝트 지침 06(컴팩션)의 feature-level 짝.
- **모델 적응 결정트리**: **프롬프트 → RAG → 파인튜닝 → 추론(extended-thinking) 모델** 사다리 중 선택 근거 + 소/대 모델·멀티모달(vision/audio)·비용/품질 라우팅. 기본 휴리스틱: *RAG-first, fine-tune-second*; long-context는 검색의 무료 대체가 아님.

### 5.21 OWASP LLM Top 10 (2025) 위협 체크 (§5.8 가드레일의 feature-level 패스)
> 4-카테고리 가드레일을 표준 위협 택소노미에 매핑(조직 카탈로그는 17번). 기능마다 triage:
- LLM01 프롬프트 인젝션(간접·tool/RAG 콘텐츠 경유) · LLM02 민감정보 노출 · LLM05 부적절 출력 처리 · **LLM06 과도한 권한**(Tool 매트릭스 §5.10) · LLM07 시스템 프롬프트 유출 · LLM08 벡터/임베딩 취약(RAG) · LLM09 오정보(grounding+citation+고위험 사람검토) · **LLM10 무한 소비**(정지조건 §5.15).

## 8. 작성 원칙

### 8.1 Eval-First
평가 기준이 합의되기 전에는 프롬프트를 길게 쓰지 않는다.
6번(평가 명세)을 1~5번보다 먼저 채워보면, 1~5번이 자연히 정돈된다.

### 8.2 Plan / Generate / Evaluate 분리
세 영역은 서로 다른 사람·세션·에이전트가 담당한다.
같은 사람이 같은 세션에서 프롬프트와 평가를 동시에 만들면 자기참조 오류가 발생한다.

### 8.3 구조화 출력 강하게 권장
가능한 한 자유 문장이 아닌 JSON Schema/typed output 형태로 출력을 제약한다.
이는 가드레일·평가·후속 처리를 모두 단순화한다.

### 8.4 결정론 정도를 의식하라
temperature와 seed 사용 여부에 따라 평가 재현성이 달라진다.
평가용 호출은 가능한 한 결정론적으로 운영한다.

### 8.5 모든 출력은 추적 가능해야 한다
사용자에게 보여진 모든 AI 출력은 (request_id, model, prompt_version, eval_score)로 역추적 가능해야 한다.

### 8.6 Fallback 없는 AI 기능은 출시하지 않는다
모델 다운/지연/가드레일 차단 시 사용자는 어떤 형태로든 결과를 얻어야 한다 (정중한 거절도 결과다).

### 8.7 가드레일은 프롬프트가 아니라 시스템에 둔다
"사용자가 욕설을 하면 응답하지 마세요"와 같은 프롬프트 차원 통제는 보조 수단이다.
실제 차단은 입력/출력 단의 별도 가드 컴포넌트가 수행해야 한다.

## 9. 자주 하는 실수

### 9.1 평가 데이터셋 없이 출시
"잘 동작하는 것 같다"로 출시하면 회귀를 감지할 수 없다.

### 9.2 결정론적 인터페이스 위에 비결정론 모델
사용자가 "정확한 숫자"를 기대하는 곳에 LLM을 그대로 노출하면 신뢰가 무너진다.

### 9.3 fallback 없는 외부 모델 의존
모델 제공자 장애가 곧 제품 장애가 된다.

### 9.4 프롬프트가 곧 정책 문서
프롬프트는 코드 일부다. 정책 변경은 별도 ADR/문서로 추적해야 한다.

### 9.5 비용 캡 없는 운영
악성 사용자나 무한 루프 한 건이 월 비용을 수백 배 부풀릴 수 있다.

### 9.6 Edit/Regenerate 없는 생성형 UX
사용자에게 거부 권한이 없는 AI 출력은 신뢰를 잃는다.

### 9.7 self-eval에 의존
같은 모델이 자기 출력을 평가하면 잘못된 출력을 잘했다고 판단할 수 있다.
Judge 모델은 가능한 한 다른 모델을 쓴다.

## 11. 좋은 문장 패턴

### 11.1 AI 가치 가설 패턴
본 기능은 [작업]을 [전통 방식]으로 처리할 경우 [한계]가 있어, [LLM 능력]을 활용해 [가치]를 제공한다.

### 11.2 입력 계약 패턴
본 기능의 입력은 [스키마]를 따르며, 필수 필드는 [필드 목록]이다. [필드명]은 PII 가능성이 있으므로 redaction 후 모델에 전달한다.

### 11.3 출력 계약 패턴
본 기능의 출력은 [JSON Schema]를 따르며, 스키마 검증에 실패하면 [재시도/거절] 동작을 수행한다.

### 11.4 평가 패턴
본 기능의 합격 기준은 [judge 점수] [임계값] 이상의 비율이 [%] 이상이며, p95 latency가 [N초] 이하인 것이다. 회귀 기준은 [메트릭]이 [임계값]을 하회하면 배포를 차단하는 것이다.

### 11.5 실패 UX 패턴
사용자가 결과에 만족하지 못할 경우 [edit/regenerate/reject/escalate] 액션을 [제공 방법]으로 제공한다. 모든 AI 출력에는 [라벨]을 부착해 사용자가 AI 생성물임을 인지할 수 있도록 한다.

### 11.6 Fallback 패턴
1차 호출이 [조건]에서 실패하면 [2차 모델]로 재시도하고, 그조차 실패하면 [룰/사람/거절]로 강등한다.

## 12. EARS 표기법 (권장)

기능 동작을 EARS(Easy Approach to Requirements Syntax) 형태로 적으면 검증 가능성이 높아진다.

기본 패턴:
- "When [trigger], the system shall [response]."
- "While [precondition], the system shall [response]."
- "If [condition], then the system shall [response]."

예시:
- "When the user submits a question longer than 4000 tokens, the system shall reject with a 'too long' message."
- "If the model returns content flagged by the safety guard, then the system shall replace the response with the standard refusal message."

## 13. AI 실행용 축약 지시문

앞으로 AI 기능 기획서를 작성할 때, 이를 AI 기능의 입력·출력 계약, 평가, 가드레일, fallback, 관측의 원본 문서로 취급하라.
가장 먼저 AI 가치 가설과 평가 명세를 합의하고, 그 다음에 입력·출력 계약과 모델·프롬프트 설계를 정의하라.
모든 AI 출력에는 사용자가 edit/regenerate/reject 할 수 있는 경로를 의무로 포함하라.
가드레일은 프롬프트가 아니라 시스템 컴포넌트로 강제하라.
fallback이 없는 AI 기능은 정의하지 말라.
모든 출력은 (request_id, model, prompt_version, eval_score)로 역추적 가능해야 하며, 비용 캡과 알람을 동시에 정의하라.
프롬프트, 평가셋, 가드 정책은 기계 판독 가능한 별도 파일(prompts/, evals/, policy.yaml)로 관리하고 본 문서의 frontmatter `machine_readable_links`에 연결하라.

## 15. 실무 craft 부록 (현장 패턴 · 적용 프로젝트 환류)

### 15.1 추천·ML 기능 기획 수명주기 7단계
- §5의 LLM 진입점 외에, 추천·예측형 ML 기능은 7단계로 기획한다: **데이터 수집 → 특징 추출·전처리 → 피처 엔지니어링 → 모델 선택·학습 → 평가·검증(정확도/정밀도/재현율/ROC-AUC) → 배포·재학습 → 초개인화(퍼소나)**.
- **변수 정의 스키마**: `변수명 · 데이터 정의 · 사용구분(Input/Target/Rejected) · 측정수준(Nominal/Interval)` — 어떤 변수를 모델 입력/목표/제외로 쓰는지 명시. §14.2 빠른 작성 템플릿(입력/출력 스키마)의 ML판.

### 15.2 2025-26 에이전트·RAG·컨텍스트 표준 (웹리서치 환류)
> Anthropic(Building Effective Agents·context engineering)·MCP 스펙·OpenAI structured outputs·RAGAS·OWASP LLM Top10 2025 1차 소스로 §5.15~5.21 신설(지침 ~2026-05 이후 발전분). 핵심: **workflow vs agent 게이트·정지조건 · MCP 통합 계약 · RAG-eval(faithfulness) · constrained decoding · 컨텍스트 엔지니어링(context rot)·모델 적응 사다리 · OWASP LLM Top10 feature 패스**.
- **상호 배선**: 정지조건(§5.15)↔LLM10 · MCP 미신뢰 응답(§5.17)↔LLM01 · grounding(§5.18)↔LLM09 · 구조화 출력(§5.19)↔LLM05 — 고립 필드 아닌 교차참조로 일관성 유지.
- **한 줄 원칙**: 가장 단순한 것(프롬프트→워크플로우)부터, 에이전트는 필요할 때만·정지조건과 함께. 컨텍스트는 최소 고신호. 조직 eval/guard 카탈로그는 17번.
