# 21. 개발명세 작성 지침

> 개발명세 계열 템플릿(architecture · data-model · user-flow · wireframe-spec · functional-spec · api-contract)을 *어떻게 조합·작성하고 개발자에게 인계하는지*의 단일 표준.
> 위상: 기획 계열이 지침 10~17로 "어떻게 쓰나"를 규율하듯, 개발명세 계열의 대응 지침. 지침 18(빌드 순서)·19(코드 품질)·20(시각 품질)이 규율하지 않던 *명세 작성·조합·인계* 층을 채운다.

---

## 1. 왜 — "개발기획서"를 단일 문서로 만들지 않는다

전통 SI의 **개발기획서**는 한 문서에 기술 접근 + 모듈 분해 + 일정 + 담당 + 기술 결정을 모두 담았다. 방법론은 이를 **의도적으로 분해**해 둔다 — 스토리보드를 정책/화면/기능으로 분해한 것과 같은 원리(단일출처·중복 금지, `CLAUDE.md` §4 File Roles).

| 개발기획서가 담던 것 | 방법론의 단일 위치 |
|---|---|
| 제품 요구("무엇") | `prd.md` · `requirements-spec.md` |
| 기술 접근·시스템 구조 | `architecture.md` |
| 데이터 구조 | `data-model.md` |
| **시스템 인터페이스 계약** | **`api-contract.md`** |
| 기능 단위 규칙·예외 | `functional-spec.md` |
| 화면 흐름·화면 | `user-flow.md` · `wireframe-spec.md` |
| 누가·언제·무엇을 | `wbs.md` (직무 레인 + 담당자 1명) |
| 페이즈·게이트 | `MASTER_PLAN.md` (지침 18) |
| 기술 의사결정·근거 | `ADR` |

> **개발기획서를 하나로 다시 묶지 않는다.** 재번들은 위 원본들을 중복시켜 단일출처를 깨뜨린다. "개발자에게 던지는 개발기획서"의 실체는 *이 문서들의 조합 + 아래 §3의 읽는 순서*다.

## 2. 개발명세 6종 지도 (원본 경계)

| 템플릿 | 원본으로 다루는 것 | 다루지 *않는* 것 (다른 문서 소관) |
|---|---|---|
| `architecture` | 어떻게(설계 원칙·시스템 구조·as-built→목표→이전 경로) | 데이터 상세(→data-model) · API 계약(→api-contract) |
| `data-model` | 엔티티·필드·관계·무결성·마이그레이션 | 전송 형태(→api-contract §4) |
| `api-contract` | **시스템 인터페이스 계약** — 엔드포인트·요청/응답·에러·버전·인증 | 기능 내부 규칙(→functional-spec) · 타입 정의 원본(→data-model) |
| `functional-spec` | 기능 *단위* 비즈니스 규칙·권한·예외(레이어별) | 시스템 간 인터페이스(→api-contract) |
| `user-flow` | 정상/대안/실패(+복구)/분기 흐름 | 화면 레이아웃(→wireframe-spec) |
| `wireframe-spec` | 화면별 5블록·3-state | 흐름 로직(→user-flow) |

핵심 경계: **functional-spec = 기능 안에서 무엇이 일어나나 / api-contract = 기능들 사이 계약.** 병렬 개발(FE/BE 분리)의 조율 축은 api-contract다.

---

## 3. 개발자용 "여기서 시작" — 읽는 순서 (kickoff)

개발리드가 개발자에게 인계할 때, 산출물 더미가 아니라 **읽는 순서**를 준다(지침 09 — 사람 독자는 순차적으로 읽는다).

1. **무엇을·왜** — `prd` / `requirements-spec` (목표·범위·우선순위 Pn)
2. **기술 접근** — `architecture` (설계 원칙·구조·이전 경로)
3. **데이터** — `data-model` (엔티티·관계)
4. **계약** — `api-contract` (병렬 착수의 단일 진실 — FE/BE 여기서 합의)
5. **기능 규칙** — `functional-spec` (FS-ID별 비즈 규칙·예외)
6. **화면·흐름** — `user-flow` → `wireframe-spec`(또는 목업)
7. **누가·언제** — `wbs` (직무 레인·담당자·의존성)

> 각 단계에 **"왜(의도)" 1~2줄**을 붙인다 — 사람 개발자는 이유를 알아야 마이크로 판단을 옳게 한다. 빈틈은 강박적으로 닫지 말고 **api-contract §5 / 각 문서의 Open 질문**으로 남겨 질문 루프를 연다(지침 09 소통 계약).

---

## 4. 작성 순서·게이트

- **작성 순서**: architecture 초안 → data-model → api-contract(계약 합의) → functional-spec → user-flow/wireframe. api-contract는 **FE/BE 착수 전 합의 게이트**(`dev-spec-review`)를 통과해야 병렬 착수.
- **비가역·고위험 결정**은 명세 본문이 아니라 **ADR**로 분리(기술 스택 확정·데이터 격리 방식·인증 모델 등).
- **자동 Class B 트리거**(스키마 변경·인증 변경·외부 API 계약 변경)가 api-contract/data-model에서 발생하면 PR에 근거·영향·롤백을 명시(`CLAUDE.md` §3).

---

## 5. 다른 지침과의 경계

- **지침 18(개발 마스터 플랜)**: 기획서를 *빌드 순서·페이즈·게이트*로 정렬. 본 지침은 그 안에서 *각 개발명세를 어떻게 쓰고 조합하나*. 18=일정 축, 21=명세 축.
- **지침 19(클린코드)·20(디자인 토큰)**: *코드·시각 산출물의 품질* 가드레일. 본 지침은 *명세 문서*의 작성. 명세→코드 순.
- **지침 09(핸드오프 재포맷)**: 산출물 독자가 사람 개발자일 때의 재포맷. 본 지침의 §3 읽는 순서·"왜" 첨부·Open 질문 루프가 09의 사람 레이어를 개발명세에 인스턴스화한 것.
- **`planning-handoff` 모드**: 기획 전담자는 §3의 1~6을 넘기되 `architecture`·`data-model`·`api-contract`는 개발자 소유라 기본 제외(필요 시 데이터 요구까지 확장). `_CATALOG.md` 참조.

---

## 6. 유래·향후

- **유래**: 방법론이 기획-헤비(지침 10~17)·개발명세-라이트였고, "개발리드→개발자 인계의 단일 진입점·API 계약 부재"가 확인됨. 사용자 발의(2026-07-08, METH-062). api-contract 템플릿 신설과 짝.
- **향후**: 실사용에서 병렬 개발 조율 craft가 축적되면 §3 읽는 순서·§4 게이트를 사례로 보강. api-contract의 OpenAPI 기계판독 강제(스펙-문서 일치 CI)를 N≥2 재현 시 검토.
