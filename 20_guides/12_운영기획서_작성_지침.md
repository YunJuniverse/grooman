# 12_운영기획서_작성_지침

## 1. 문서 개요

### 1.1 문서명
운영기획서 작성 지침

### 1.2 문서 목적
본 문서는 AI가 운영기획서를 작성할 때 따라야 하는 표준 작성 기준을 정의한다.  
운영기획서는 서비스 출시 이후 또는 출시 준비 단계에서, 서비스가 안정적으로 운영되기 위한 운영 기준, 처리 절차, 역할 분담, 검수 및 제재 기준, CS 기준, 관리자 요구사항을 정리하는 문서다.

### 1.3 문서의 역할
운영기획서는 서비스기획서에서 정의한 기능과 정책이 실제 운영 환경에서 지속 가능하게 작동하도록 만드는 기준 문서다.  
즉, 누가 무엇을 어떤 기준으로 처리하는지, 문제가 생겼을 때 어떤 절차로 대응하는지, 운영팀이 어떤 기능과 도구를 필요로 하는지를 정의한다.

### 1.4 적용 범위

- 운영기획서

## 2. 운영기획서의 정의

### 2.1 기본 정의
운영기획서는 **서비스를 실제로 관리, 통제, 대응, 유지하기 위한 운영 기준과 절차를 정의하는 문서**다.  
이 문서는 단순한 매뉴얼이 아니라, 운영의 구조와 원칙을 정리하는 상위 기준 문서다.

### 2.2 이 문서가 다루는 핵심
운영기획서는 아래를 다룬다.

- 운영 조직과 역할
- 운영 대상과 처리 단위
- 운영 프로세스
- 검수 기준
- 승인 기준
- 제재 기준
- CS 처리 기준
- 이슈 대응 기준
- 관리자 기능 요구사항
- 운영 KPI
- 운영 리스크와 대응 원칙

### 2.3 다른 문서와의 차이
운영기획서는 사업기획서처럼 시장성과 BM을 원본으로 다루지 않는다.  
서비스기획서처럼 사용자 플로우와 기능 상세를 원본으로 다루지 않는다.  
마케팅기획서처럼 채널 전략이나 캠페인 구조를 중심으로 다루지 않는다.  
브랜드기획서처럼 메시지나 브랜드 정체성을 원본으로 다루지 않는다.

운영기획서는 **운영 기준, 처리 절차, 책임 주체, 운영 도구 요구사항의 원본 문서**다.

### 2.4 한 문장 정의
운영기획서는 **“서비스를 누가 어떤 기준과 절차로 안정적으로 운영할 것인지 정의하는 운영 기준 문서”**다.

## 3. 운영기획서의 목적

### 3.1 핵심 목적
운영기획서의 목적은 아래와 같다.

1. 서비스 운영의 기준과 책임 구조를 명확히 한다
2. 검수, 승인, 제재, CS, 이슈 대응의 원칙을 정의한다
3. 운영팀과 관리자 도구 개발의 기준을 제공한다
4. 서비스 정책이 실제 운영 현실과 충돌하지 않도록 조정한다

### 3.2 이 문서가 해결하는 문제
운영기획서가 없으면 다음 문제가 자주 발생한다.

- 운영자가 무엇을 기준으로 판단해야 하는지 모름
- 신고, 제재, 검수의 처리 기준이 사람마다 다름
- 관리자 기능이 필요한데 요구사항이 정리되지 않음
- CS 응대 품질이 들쭉날쭉함
- 정책 위반 상황에서 일관성이 없음
- 서비스기획은 되어 있는데 운영 현실과 맞지 않음
- 장애, 분쟁, 악성 행위 대응 기준이 없음

### 3.3 필요한 시점
아래 단계에서 운영기획서가 필요하다.

- 서비스 런칭 준비 단계
- 관리자 기능 개발 전
- 검수형/커뮤니티형/거래형 서비스 설계 단계
- CS/운영팀 온보딩 전
- 운영 이슈가 빈번하게 발생하는 서비스의 리뉴얼 시점
- 운영 정책의 표준화가 필요한 시점

## 4. 핵심 질문

AI는 운영기획서를 작성할 때 반드시 아래 질문에 답해야 한다.

### 4.1 누가 무엇을 처리하는가
- 어떤 운영 역할이 존재하는가
- 각 역할의 책임 범위는 무엇인가
- 어떤 상황에서 누가 개입해야 하는가

### 4.2 무엇을 어떤 기준으로 검수/승인/제재하는가
- 어떤 항목이 운영 대상인가
- 허용 기준과 제한 기준은 무엇인가
- 승인/반려/차단/제재의 조건은 무엇인가

### 4.3 문의, 신고, 분쟁, 장애는 어떻게 처리하는가
- 주요 운영 이슈 유형은 무엇인가
- 이슈별 처리 절차는 무엇인가
- 긴급도와 우선순위는 어떻게 구분하는가

### 4.4 운영팀은 어떤 도구와 정보가 필요한가
- 어떤 관리자 화면이 필요한가
- 어떤 로그와 이력이 보여야 하는가
- 어떤 액션을 관리자 도구에서 수행할 수 있어야 하는가

### 4.5 무엇을 KPI로 관리할 것인가
- 운영 품질을 어떻게 측정할 것인가
- 응답 시간, 처리율, 오탐/누락, 재발률 등을 어떻게 볼 것인가

### 6.12 AIOps 관제 (AI 기능이 있을 때 필수)
AI 기능을 포함한 서비스의 런타임 관제 기준이다.

반드시 포함:
- **트레이스 조회 권한**: 누가 어느 트레이스를 볼 수 있는가 (PII 보호 포함)
- **비용 모니터링**: 시간당/일별/월별 비용 캡과 알람 임계
- **품질 모니터링**: online judge 점수, refusal rate, 가드레일 trigger 비율
- **drift 감지**: 입력 분포·출력 분포가 베이스라인에서 어느 정도 벗어나면 알람할지
- **모델 deprecation 대응**: 외부 모델 제공자가 모델을 폐기할 때의 마이그레이션 절차

각 항목의 원본 정의는 17 평가/가드레일 지침을 참조하고, 본 절에서는 *운영 적용 형태*만 다룬다.

### 6.13 인간 검토 게이트 운영
high-stakes AI 액션이 있는 경우, 다음 운영 기준을 명시한다.

- 어떤 액션이 인간 검토 대상인가 (16번에서 정의된 게이트 ID)
- 누가 검토하는가 (역할/팀)
- 검토 SLA (예: 일반 24h, 긴급 1h)
- 검토 거부 시 사용자에게 전달되는 메시지
- 검토 이력 보존 정책

### 6.14 AI Incident 분류
운영 인시던트에 AI 관련 분류를 추가한다.

| 분류 | 정의 | 1차 대응 |
|------|------|----------|
| Hallucination | 사실과 다른 출력으로 사용자 영향 | 출력 회수, 영향 범위 측정 |
| Wrong Answer | 환각은 아니나 잘못된 추천/응답 | 골든셋 보강, 회귀 평가 |
| Cost Spike | 비용 캡 초과 | 즉시 차단, 원인 식별 |
| Security (Injection/Jailbreak) | 보안 가드 우회 | 차단, 보안팀 알람, 패턴 추가 |
| PII Leak | 출력에서 개인정보 노출 | 즉시 차단, 보안/법무 알람 |
| Bias | 특정 집단에 차별적 출력 | 출력 차단, 공정성 평가 |
| Model Outage | 외부 모델 장애 | fallback 발동, 제공자 상태 확인 |

각 분류별로 (대응 SLA, 보고 라인, 사후 분석 책임자, 재발 방지 절차)를 정의한다.

### 6.15 신뢰성 목표 — SLO / Error Budget
> 운영 KPI(6.9)를 *신뢰성 목표 + 리스크 예산*으로 격상(Google SRE). "모니터링 있음"을 거버넌스 레버로 바꾼다.

- **SLI**(측정: 성공률·p99 지연·가용성) → **SLO**(내부 목표, 예 "28일 롤링 99.9%", 사용자 기대·비즈 영향에서 도출, 현 성능 약간 아래에서 시작해 분기 후 조임) → **SLA**(외부 계약, SLO보다 *느슨하게* — 안전 마진).
- **Error Budget = 1 − SLO** = *리스크 감수 허가*(기능 출시에 쓴다). **Error-budget 정책 필수**: 예산 소진 시 *무엇을 하는가* — 기능 프리즈·신뢰성 작업만, 강제 권한자·예외 승인자 명시(6.9 KPI + 승인 거버넌스 §19.4 연결).

### 6.16 인시던트 관리 (형식) — 6.7 이슈 대응의 형식화
- **심각도 매트릭스 SEV1~SEV4**(SEV1 코어 다운=즉시 페이지·전원 / SEV2 중대 저하·우회 존재 / SEV3 경미 / SEV4 미미). 심각도가 *대응 속도·페이지 대상·커뮤니케이션 주기*를 결정.
- **역할 분리**: Incident Commander(조율·결정, *디버깅 안 함*) · Comms Lead · Ops/Tech Lead.
- **라이프사이클**: 탐지→분류→완화→해결→리뷰(체크리스트).
- **지표**: **MTTD·MTTA·MTTR**(·MTBF)를 심각도별 운영 KPI로.

### 6.17 On-call · 에스컬레이션 · 포스트모템
- **On-call/에스컬레이션 정책**: 1차 ~5분 ack → 미ack 시 2차 자동 승격 → breach 시 리드 알림. 로테이션 주기·follow-the-sun·**교대 핸드오프 30~60분 오버랩**(open 인시던트·최근 배포·임시 우회 요약).
- **블레임리스 포스트모템**(SEV1/2 필수): 타임라인·영향·근본원인(사람 아닌 프로세스/시스템)·기여요인·**추적되는 액션아이템(담당·기한)** → §19.3 추적 매트릭스 연결.

### 6.18 알림·관측 (SLO 기반) — 6.12 AIOps 관제의 알림 품질
- **3 pillars**: metrics(대시보드·SLO·알림)·logs·traces.
- **증상에 알림, 원인 아님**(에러율>1%·p99>2s). **multi-window·multi-burn-rate 알림**(예: 1h에 2%·6h에 5% 소진 → 페이지 / 3일 10% → 티켓)으로 스파이크와 지속 소진 구분.
- **알림 피로 통제**: 모든 알림은 actionable, 상관·중복제거로 단일 인시던트화, 주기적 노이즈 알림 삭제.

### 6.19 계층 지원 (L1/L2/L3) — 6.6 CS 정책의 계층화
- **L1**(프론트·흔한 이슈·분류, FCR 70~80% 목표) / **L2**(기술 전문·로그·설정) / **L3**(전문 엔지니어·근본원인·인프라, 티켓 >5~10% 유입 시 상위 문제 신호).
- **계층별 SLA**: L1=first-response, L2/L3=resolution. 에스컬레이션 타이머(66% 경고·85% 자동승격·breach 리드알림). **디플렉션**(셀프서브 KB, 디플렉션율 추적). 지표 **CSAT·CES·FCR**. L3→엔지니어링 경로를 6.16 인시던트로 연결.

### 6.20 변경·릴리스 운영 (Change/Release Ops)
- **ITIL 변경 유형**: Standard(사전승인·저위험·반복) / Normal(위험평가+승인) / Emergency(인시던트 중 ECAB). CAB는 자문, 정의된 Change Authority(자동 CI/CD 게이트 포함)가 인가.
- **점진 배포**(blast radius 축소): feature flag · canary/blue-green · staged rollout. **롤백 계획 필수 필드**(CLAUDE.md Class B 롤백을 운영에 미러). **변경 프리즈 창**(피크·민감 시기).

### 6.21 Toil 예산 · 자동화
- **Toil**(수동·반복·자동화 가능·전술적·무가치·선형 증가) 정의. **Toil 캡**(SRE <50% 가이드 또는 팀 규모 맞춤) + 주기적 측정. 고빈도 절차는 runbook → 점진 자동화. "제거한 toil"을 운영 KPI로.

### 6.22 AI 프로덕션 운영 (AI 기능 운영 시 — 6.12~6.14 심화)
- **프로덕션 eval/품질 모니터링**: RMSE/F1은 평평해도 환각은 오를 수 있음 → groundedness·응답품질·의미유사도·**가드레일 위반율**·drift를 *추론 호출 단위*로.
- **가드레일 위반 = 인시던트**: 독성·PII 유출·정책 위반을 SEV 매핑되는 1급 인시던트로(6.14 AI Incident 연결).
- **프롬프트 인젝션/남용 대응**(OWASP LLM01 2025 #1): 탐지+대응, 미신뢰 입력 후 에이전트 tool-use 제약, 고위험 행동(발송·구매·데이터 변경·코드 실행)은 **HITL 승인**(6.13 인간 검토 게이트 연결).
- **모델/제공자 장애 failover**: 멀티 provider fallback 체인·헬스체크·타임아웃·자동 failover·**tool 호출 멱등키**(재시도 중복 실행 방지).
- **HITL 워크로드**: 검토 큐를 실제 운영 부하로 계획(처리량·검토 지연 SLA·백로그 에스컬레이션).
- **AI 비용 운영(토큰 FinOps)**: feature/model/provider별 실시간 토큰·비용 귀속 + 급증 이상탐지 + 비용 KPI(급증을 alertable 조건으로).

## 10. 작성 원칙

### 10.1 실제 운영자가 판단 가능한 수준으로 쓴다
운영기획서는 선언문이 아니라 판단 기준 문서다.  
실제 운영자가 읽고 처리할 수 있는 수준으로 작성해야 한다.

### 10.2 책임 주체를 명확히 쓴다
누가 처리하는지가 빠지면 문서 효용이 급격히 떨어진다.

### 10.3 기준은 가능한 한 조건형으로 쓴다
“필요 시”, “상황에 따라”, “적절히” 같은 표현은 피하고, 어떤 조건에서 어떤 조치를 하는지 쓴다.

### 10.4 예외와 이슈를 중심에 둔다
운영은 정상 상황보다 예외 상황에서 진가가 드러난다.  
따라서 신고, 분쟁, 장애, 악성 행위, 어뷰징 같은 상황을 적극적으로 포함해야 한다.

### 10.5 운영 현실과 서비스 정책을 연결한다
서비스기획의 정책이 운영에서 과도한 비용이나 모호함을 만들지 않는지 검토해야 한다.

### 10.6 관리자 도구 요구사항까지 연결한다
운영 기준만 있고 도구가 없으면 실행되지 않는다.  
따라서 운영기획서는 관리자 기능 요구사항까지 이어져야 한다.

## 11. 자주 하는 실수

### 11.1 서비스기획서 내용을 그대로 가져오는 실수
운영기획서는 기능 정의 문서가 아니라 처리 기준 문서다.

### 11.2 기준 없이 원칙만 적는 실수
예:
- 부적절한 게시물은 제재한다
- 문의는 빠르게 응답한다

이런 문장은 실제 운영 기준이 되기 어렵다.

### 11.3 책임 주체가 없는 실수
누가 승인하고 누가 반려하고 누가 최종 판단하는지가 빠지면 운영 문서로서 약하다.

### 11.4 관리자 기능 요구사항이 빠지는 실수
운영기획은 툴 설계와 연결되지 않으면 실행력이 떨어진다.

### 11.5 CS와 운영과 정책 대응을 섞는 실수
문의 대응, 신고 처리, 정책 제재는 비슷해 보여도 다른 구조다.  
문서에서 구분해야 한다.

### 11.6 긴급도와 우선순위가 없는 실수
운영 이슈는 모든 것을 같은 수준으로 다루면 안 된다.

## 12. 다른 문서와의 경계

### 12.1 서비스기획서와의 경계
운영기획서는 서비스기획서의 정책을 입력값으로 받을 수 있으나, 사용자 흐름과 기능 상세 자체를 원본으로 다루지 않는다.

### 12.2 사업기획서와의 경계
운영기획서는 운영 방식의 실행 기준을 다루지만, 왜 이 사업을 하는지와 수익 구조는 사업기획서가 원본이다.

### 12.3 마케팅기획서와의 경계
운영기획서는 캠페인 유입 이후 문의/이슈 대응의 기준을 일부 다룰 수 있으나, 채널 전략과 성과 구조는 마케팅기획서가 원본이다.

### 12.4 브랜드기획서와의 경계
운영기획서는 응대 톤이나 민감 이슈 대응 원칙에서 브랜드 기준을 참조할 수 있으나, 브랜드 메시지 원본은 브랜드기획서가 담당한다.

### 12.5 프로젝트 관리 기획서와의 경계
운영기획서는 운영 기준과 처리 절차의 원본을 정의하고, 프로젝트 관리 기획서는 운영 준비 일정, 산출물 제출, 승인 흐름, 이슈와 변경 관리 같은 프로젝트 관점의 통제를 담당한다.

## 14. 금지 패턴

### 14.1 추상적인 운영 원칙만 나열
예:
- 신속하고 정확하게 운영한다
- 고객 중심으로 대응한다
- 부적절한 콘텐츠는 제한한다

이런 문장은 구체 기준 없이 단독 사용 금지

### 14.2 처리 주체 없는 문장
예:
- 신고 건은 검토 후 조치한다
- 문의는 우선순위에 따라 처리한다

누가 하는지가 빠져 있으므로 불완전하다

### 14.3 제재 기준 없는 정책 문장
예:
- 반복 위반 시 강한 제재를 한다
무엇이 반복이고 무엇이 강한 제재인지 불분명하다

### 14.4 툴 없는 운영 계획
운영자가 어떤 관리자 기능으로 처리하는지 빠져 있으면 문서 완성도가 낮다

### 14.5 마케팅/서비스 내용을 과다 삽입
운영기획서에서 기능 설계나 채널 전략을 과도하게 다루지 않는다

## 16. 품질 체크리스트

작성 후 아래를 점검한다.

- 운영 목적과 범위가 명확한가
- 누가 무엇을 처리하는지 보이는가
- 운영 대상이 구체적으로 정의되었는가
- 검수/승인/제재 기준이 명확한가
- 문의와 신고와 분쟁과 장애를 구분했는가
- 예외 상황과 긴급도 기준이 포함되어 있는가
- 관리자 기능 요구사항이 충분한가
- KPI가 측정 가능한 수준인가
- 서비스기획과 운영기획의 경계가 보이는가
- 실제 운영팀이 문서만 보고 판단 가능한 수준인가

**2025-26 신뢰성·인시던트 체크(§6.15~6.22)**:
- **SLO + Error-budget 정책**이 있는가 — 예산 소진 시 행동(기능 프리즈)·강제 권한자 명시(§6.15)
- **인시던트 심각도(SEV1~4)·IC 역할·MTTD/MTTA/MTTR**가 정의됐는가(§6.16)
- **on-call/에스컬레이션 + 블레임리스 포스트모템**(액션아이템 추적)이 있는가(§6.17)
- 알림이 **증상 기반·multi-burn-rate**이고 알림 피로 통제가 있는가(§6.18)
- **계층 지원 L1/L2/L3 + 계층별 SLA(CSAT/CES/FCR)**가 있는가(§6.19)
- **변경 유형(Std/Normal/Emergency)·롤백·프리즈**가 있는가(§6.20)
- (AI 운영 시) **프로덕션 eval·가드레일=인시던트·provider failover·HITL 워크로드·토큰 FinOps**(§6.22)

## 17. AI 실행용 축약 지시문

앞으로 운영기획서를 작성할 때, 이를 서비스가 실제로 안정적으로 운영되기 위한 기준과 절차의 원본 문서로 취급하라.  
먼저 운영 목적, 운영 대상, 역할 구조, 핵심 운영 프로세스, 검수/승인/제재 기준, CS 기준, 이슈 대응 절차, 관리자 기능 요구사항, 운영 KPI를 정의하라.  
정상 처리뿐 아니라 신고, 분쟁, 장애, 악성 행위, 어뷰징 같은 예외 상황을 반드시 포함하라.  
정책은 모호한 원칙이 아니라, 누가 어떤 조건에서 무엇을 하는지 판단 가능한 수준으로 작성하라.  
서비스기획의 기능 상세 원본과 사업기획의 시장/수익 구조 원본은 중복하지 말고 필요 시 참조만 하라.  
운영기획서는 실제 운영팀과 관리자 도구 설계에 바로 활용 가능한 수준으로 구체화하라.

### 19.1 독립성·이해상충 방지 설계
- 평가·판정·인증 기능과 *수익* 기능이 한 조직에 공존하면 구조적 독립성을 명문화한다: ① **보상구조 분리** ② **정보차단(Chinese Wall)** ③ **알고리즘 공개 + 외부 감사** ④ **자동 분리(사람 개입 최소화)**. 신뢰성을 다루는 모든 평가형·인증형·심사형 서비스에 전이. 메커니즘별 `위험 → 차단 장치 → 검증 방법` 표로.

### 19.2 운영 게이트 정의 (DoR / DoD)
- 착수 가능 상태(**Definition of Ready**)와 완료·릴리스 가능 상태(**Definition of Done**)를 빌드 전·후 *객관적 체크리스트*로 표준화. 프로세스별 시작/종료 조건을 횡단 표준으로 격상. `Exceptions(승인자·만료)` 표를 동반해 예외도 통제.

### 19.3 요구사항 추적 매트릭스
- 요구 ID를 `기획 출처 | 개발 스펙 | 빌드 산출물 | 테스트 증거 | milestone`로 한 줄에 연결. **Missing Links**(누락)·**Deferred**(이월)를 별표로 분리 → 요구를 *정의*가 아니라 *구현·테스트 증거까지* 추적.

### 19.4 의사결정·승인 거버넌스
- **임원 의사결정 브리프(8블록)**: 요청 결정 / 현재 상태 / 완료된 것 / 학습 / 리스크·트레이드오프 / 권고안 / 승인 요청 / 결정 기록 — 승인자가 *한 장*으로 판단(CLAUDE.md 인간 승인 게이트의 문서 양식).
- **승인 로그 + 폐기 추적**: 승인 건을 `ID·산출물·버전·변경유형·요청자·승인자·상태`로 누적하고, **Open Approval**(차단 영향)·**Superseded**(폐기→대체 사유)를 분리 → 결정의 *수명주기*를 감사 가능하게.
