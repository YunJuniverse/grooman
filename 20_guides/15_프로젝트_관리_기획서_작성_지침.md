# 15_프로젝트_관리_기획서_작성_지침

## 1. 문서 개요

### 1.1 문서명
프로젝트 관리 기획서 작성 지침

### 1.2 문서 목적
본 문서는 AI가 프로젝트 관리 기획서를 작성할 때 따라야 하는 표준 작성 기준을 정의한다.  
프로젝트 관리 기획서는 프로젝트의 목표, 범위, 일정, 역할, 보고 체계, 이슈, 리스크, 의사결정, 산출물, 변경사항을 통합적으로 관리하기 위한 메타 운영 문서다.

### 1.3 문서의 역할
프로젝트 관리 기획서는 사업기획, 서비스기획, 운영기획, 마케팅기획, 브랜드기획 등 프로젝트 내 개별 기획 문서와 실행 조직을 **일정, 책임, 보고, 통제, 추적**의 관점에서 연결하는 허브 문서다.  
즉, 이 문서는 “무엇을 기획할 것인가”보다 “그 기획과 실행을 프로젝트로 어떻게 관리할 것인가”를 정의한다.

## 2. 프로젝트 관리 기획서의 정의

### 2.1 기본 정의
프로젝트 관리 기획서는 **프로젝트를 일정, 범위, 역할, 보고, 이슈, 리스크, 의사결정, 산출물, 변경사항 관점에서 통합적으로 관리하기 위한 메타 운영 문서**다.

### 2.2 이 문서가 다루는 핵심
프로젝트 관리 기획서는 아래를 다룬다.

- 프로젝트 목표와 완료 기준
- 프로젝트 범위와 단계
- 조직과 역할
- 타임라인과 마일스톤
- 주간 / 월간 보고 체계
- 이슈 관리 체계
- 리스크 관리 체계
- 의사결정 관리 체계
- 산출물 관리 체계
- 변경 관리 체계
- 커뮤니케이션 및 회의 운영 체계
- 관리 KPI

### 2.3 다른 문서와의 차이
프로젝트 관리 기획서는 사업기획서처럼 사업 방향과 BM을 원본으로 다루지 않는다.  
서비스기획서처럼 기능과 정책을 원본으로 다루지 않는다.  
운영기획서처럼 서비스 운영 정책을 원본으로 다루지 않는다.  
마케팅기획서처럼 채널 전략과 캠페인을 원본으로 다루지 않는다.  
브랜드기획서처럼 메시지와 인식 체계를 원본으로 다루지 않는다.

프로젝트 관리 기획서는 **프로젝트 전체를 관리하는 구조와 통제 기준의 원본 문서**다.

### 2.4 한 문장 정의
프로젝트 관리 기획서는 **“프로젝트를 목표, 일정, 보고, 이슈, 리스크, 의사결정, 산출물 관점에서 메타적으로 통제하는 운영 관리 문서”**다.

## 3. 프로젝트 관리 기획서의 목적

### 3.1 핵심 목적
프로젝트 관리 기획서의 목적은 아래와 같다.

1. 프로젝트의 목표와 범위와 완료 기준을 명확히 한다
2. 프로젝트 일정과 단계와 마일스톤을 관리 가능하게 만든다
3. 역할, 책임, 보고, 의사결정 구조를 명확히 한다
4. 이슈, 리스크, 변경사항을 추적 가능한 상태로 만든다
5. 산출물과 회의와 커뮤니케이션을 통합적으로 관리한다

### 3.2 이 문서가 해결하는 문제
프로젝트 관리 기획서가 없으면 다음 문제가 자주 발생한다.

- 일정은 있는데 범위가 계속 흔들림
- 주간보고는 있는데 액션 정리가 안 됨
- 월간보고는 있는데 의사결정 이력이 남지 않음
- 이슈는 공유되는데 오너가 없음
- 리스크는 알고 있지만 대응이 관리되지 않음
- 산출물이 흩어져 있고 버전 관리가 안 됨
- 회의는 많은데 결정사항이 축적되지 않음
- 프로젝트 상태를 한눈에 볼 수 없음
- 변경 요청이 누적되며 일정과 범위가 무너짐

### 3.3 필요한 시점
아래 단계에서 프로젝트 관리 기획서가 필요하다.

- 프로젝트 착수 직전
- 착수 후 초기 정렬 단계
- 다부서 협업이 필요한 프로젝트
- 외부 파트너 / 수행사가 있는 프로젝트
- 리뉴얼 / 구축 / 런칭형 프로젝트
- 일정 지연, 범위 확장, 의사결정 지연이 자주 발생하는 프로젝트
- PM 체계가 필요하지만 아직 문서 기준이 없는 프로젝트

## 4. 핵심 질문

AI는 프로젝트 관리 기획서를 작성할 때 반드시 아래 질문에 답해야 한다.

### 4.1 이 프로젝트의 목표와 완료 기준은 무엇인가
- 프로젝트가 끝났다고 볼 수 있는 기준은 무엇인가
- 성공 기준은 무엇인가
- 제외 범위는 무엇인가

### 4.2 어떤 범위와 단계로 진행되는가
- 이번 프로젝트에 포함되는 것은 무엇인가
- 포함되지 않는 것은 무엇인가
- 어떤 단계로 나눌 수 있는가

### 4.3 누가 무엇을 책임지는가
- 프로젝트 오너는 누구인가
- PM은 누구인가
- 각 팀과 담당자의 역할은 무엇인가
- 누가 의사결정하는가

### 4.4 일정은 어떻게 관리되는가
- 주요 마일스톤은 무엇인가
- 선행 조건은 무엇인가
- 어떤 일정이 전체 일정에 가장 큰 영향을 주는가

### 4.5 보고는 어떻게 관리되는가
- 주간보고와 월간보고는 무엇을 위해 하는가
- 누가 언제 보고하는가
- 보고 후 어떤 액션이 이어지는가

### 4.6 이슈와 리스크는 어떻게 관리되는가
- 어떤 것을 이슈로 볼 것인가
- 어떤 것을 리스크로 볼 것인가
- 누가 책임지고 언제까지 대응하는가

### 4.7 산출물과 변경사항은 어떻게 통제하는가
- 어떤 산출물이 나오는가
- 검토 / 승인 / 버전 관리는 어떻게 하는가
- 범위 변경은 어떤 절차를 거치는가

### 5.4 외부 파트너 / 수행사

- 평가 주기 (offline 회귀 주기, online 샘플링 주기)
- 회귀 차단 이력 (어느 PR이 어느 메트릭에서 차단됐는지)
- judge calibration 점수 추이
- 데이터셋 갱신 책임자와 갱신 주기

본 절은 *PM 관점의 메타 관리*이고, 메트릭 정의·채점 방식의 원본은 17 지침이다.

### 6.16 인간 검토 게이트 카탈로그 (Human Approval Gate Catalog)
프로젝트 전체에서 운영 중인 인간 검토 게이트를 단일 카탈로그로 관리한다.

| 게이트 ID | 시점 | 승인자 | 증거 형식 | SLA | 출처 문서 |
|-----------|------|--------|-----------|-----|-----------|

방법론의 표준 게이트(Phase 1/2/3 등)와 AI 관련 게이트(eval-spec-approval, model-swap, regression-override, high-stakes-action 등)를 모두 포함한다.

게이트 ID는 PM기획서에서 *카탈로그*로 정의되며, 18번 마스터플랜에서 *이 프로젝트의 페이즈에 인스턴스화*된다 (예: `phase-entry/M1`, `master-plan-approval`).

### 6.17 AI 위험 등록부 (AI Risk Register)
일반 리스크 로그(6.8)와는 별도로 AI 특화 리스크를 분리 관리한다.

| 위험 유형 | 정의 | 발생 가능성 | 영향도 | 완화 전략 | 오너 |
|-----------|------|-------------|--------|-----------|------|
| 환각 (Hallucination) | 사실과 다른 출력 | | | 골든셋·judge·UX 라벨 | |
| Prompt Injection | 시스템 프롬프트 우회 | | | injection detector, fail-closed 가드 | |
| 데이터 유출 | PII/secret 외부 송출 | | | redactor, output scanner | |
| 모델 deprecation | 외부 모델 폐기 | | | 다중 provider, fallback 체인 | |
| 비용 폭주 | 토큰 사용량 급증 | | | 비용 캡, 알람 | |
| Bias/공정성 | 특정 집단 차별 | | | 공정성 평가, 다양한 골든셋 | |
| 법규 변경 | EU AI Act 등 새 규제 | | | 분기별 컴플라이언스 리뷰 | |
| 외부 API SLA 위반 | 모델 제공자 장애 | | | fallback chain, status 모니터링 | |

### 6.18 딜리버리 모델 선언 (PMBOK 7 — 예측/하이브리드/애자일)
> PMBOK 7이 프로세스(5그룹×10영역)에서 **12원칙 + 8 성과영역 + 테일러링**으로 전환. 이 지침의 기본 구조(보고·산출물·변경 로그)는 *예측형(predictive) 디폴트*일 뿐 — 플랜은 **개발 접근법을 명시 선언·정당화**해야 한다("Development Approach & Life Cycle" 도메인).
- **스펙트럼**: 예측(요구 안정+외부 하드 데이트) ⟷ 하이브리드(예측 마일스톤+애자일 실행) ⟷ 적응(불확실·발견 중심, Scrum 스프린트/백로그/velocity[목표 아님]·Kanban).
- 플랜 상단에 [접근법·근거·테일러링] 선언.

### 6.19 플로우 메트릭 + 확률 예측 (%-complete/간트 대안)
- **Kanban 플로우 4지표**: WIP(시작-미완)·Cycle time·Work item age·Throughput(+Lead time·Flow efficiency·CFD). WIP 한도가 주 제어 레버.
- **Monte Carlo 확률 예측**: 단일 간트 날짜 대신 과거 throughput 리샘플로 **신뢰구간**("20건이 85% 확률로 12~18일") — false-precision %-complete가 숨기는 분산을 드러냄. 반복/연속흐름·**AI 에이전트 작업**엔 플로우+확률 예측 우선.

### 6.20 DORA 딜리버리 헬스 (소프트웨어 산출물)
> 프로젝트 KPI·운영 MTTR과 구분되는 *딜리버리* 성능(처리량+안정성).
- **처리량**: 배포 빈도 · 변경 리드타임(commit→prod) · 실패 배포 복구시간. **불안정성**: 변경 실패율 · **배포 재작업률**(2024 신규 5번째).
- **2025 DORA 경고**: AI 도입은 *처리량↑ 그러나 불안정성↑* → AI 에이전트가 코드 쓸 때 변경실패율·재작업률을 특히 추적.

### 6.21 아웃컴 / OKR 레이어 (산출물 위에)
> 산출물 완료가 아니라 **아웃컴**(고객 행동·결과 변화, Seiden/Cagan) 관리. "feature factory" 회피.
- **OKR**: Objective(정성) + 3개 내외 KR(측정가능 *아웃컴*, task 체크리스트 아님). 모든 WBS/산출물은 상위 아웃컴에 링크("왜 존재?"). **선행 vs 후행** 지표 구분. *아웃컴 연결 없는 산출물 = red flag*. 기존 KPI(6.13)를 가치에 연결.

### 6.22 레퍼런스클래스 예측 (WBS 3점추정 보완)
> 3점/PERT는 여전히 *inside view*(낙관 편향). **outside view 추가**: 확정 전 유사 과거 프로젝트 class의 실제 결과와 비교 → inside가 더 낙관적이면 **낙관편향 상향(uplift)** 명시 적용(Flyvbjerg: 도로 ~20%·교량/터널 ~33%·철도 ~44% 초과). inside(3점)+outside를 *조정된* 추정으로.

### 6.23 RAID 로그 + 프리모템 (이슈/리스크/결정 로그 통합)
- **RAID**: Risks·Assumptions·Issues·Dependencies 단일 등록부. 기존 이슈/리스크/결정 로그(6.7~6.9)에 없는 **가정(Assumptions, 조용히 깨지는 전제)·크로스팀 의존(Dependencies)**을 1급·리뷰주기 항목으로. 크리티컬 패스/체인 관리.
- **프리모템**(kick-off에 필수): "이 프로젝트가 대실패했다 가정, 원인 역산"(prospective hindsight — 정확한 원인 식별 ~30%↑). 임원 없이 20~30분 → RAID 리스크/가정으로.

### 6.24 AI 증강 PM + AI 에이전트 작업 거버넌스 (2025-26 최우선)
> 이 방법론 자체가 AI 에이전트 실행으로 돌아감 — 플랜은 **human+AI 팀** 플랜이어야.
- **(a) AI copilot(PM 운영)**: 상태 롤업·일정 편차/크리티컬패스 리스크 예측·회의 요약·백로그 그루밍. 규칙: *사람이 accountability 보유*(copilot 보조, 소유 아님).
- **(b) AI 에이전트 실행 작업 거버넌스**: **작업 스코핑**(변경클래스×오너·태스크별 인수기준) · **자율 예산+정지조건**(파일/PR/반복 캡 → 초과 시 사람 체크인, CLAUDE.md "예산 내 자율 초과 시 보고"를 플랜에 표면화, 지침 07) · **인간 검토 게이트**(6.16 카탈로그를 에이전트 산출물로 확장) · **에이전트 throughput/품질 추적**(완료 건수 + DORA 변경실패율·재작업률을 품질 가드, AI 불안정성 대비). 팬아웃은 지침 08.

## 10. 작성 원칙

### 10.1 프로젝트를 문서가 아니라 관리 시스템으로 본다
프로젝트 관리 기획서는 한 번 작성하고 끝내는 문서가 아니다.  
보고, 이슈, 리스크, 산출물, 변경사항을 연결하는 상위 시스템 문서여야 한다.

### 10.2 일정은 날짜 나열이 아니라 의존성 관리다
타임라인은 마일스톤, 선행 조건, 승인 지점, 병목 구간까지 포함해야 한다.

### 10.3 보고는 정렬 장치여야 한다
주간 / 월간 보고는 상태 공유만이 아니라, 의사결정 요청, 이슈 가시화, 다음 액션 정렬 기능을 가져야 한다.

### 10.4 이슈는 추적 가능해야 한다
이슈에는 반드시 오너, 상태, 해결 목표일, 에스컬레이션 기준이 있어야 한다.

### 10.5 리스크는 사전 대응 관점으로 본다
리스크는 발생 후 처리 대상이 아니라, 미리 인지하고 완화하는 관리 대상이다.

### 10.6 변경은 통제 대상이다
범위, 일정, 산출물, 정책의 변경은 프로젝트 붕괴의 가장 큰 원인 중 하나다.  
따라서 변경 관리 체계를 반드시 포함해야 한다.

### 10.7 의사결정 이력을 남긴다
무엇을 왜 결정했는지 기록하지 않으면 같은 논의가 반복된다.

### 10.8 책임과 승인 체계를 명확히 한다
실행자와 검토자와 승인자를 분리해서 생각해야 한다.

## 11. 자주 하는 실수

### 11.1 일정표만 있고 관리 체계가 없는 실수
프로젝트 관리 문서가 단순 일정표로 끝나면 통제력이 약하다.

### 11.2 주간보고가 상태 나열로 끝나는 실수
주간보고는 액션과 의사결정 연결이 있어야 한다.

### 11.3 이슈를 회의 안건 수준으로만 다루는 실수
이슈는 로그와 상태값과 오너를 갖는 관리 대상이다.

### 11.4 리스크와 이슈를 혼동하는 실수
리스크는 잠재 문제, 이슈는 이미 발생한 문제다.

### 11.5 결정사항 기록이 없는 실수
회의는 했지만 무엇이 결정되었는지 남지 않으면 프로젝트 기억이 사라진다.

### 11.6 산출물 버전 관리가 없는 실수
같은 파일의 여러 버전이 흩어지면 검토와 승인 흐름이 꼬인다.

### 11.7 변경 요청을 비공식적으로 처리하는 실수
범위와 일정이 계속 흔들리는 가장 큰 원인이다.

## 12. 다른 문서와의 경계

### 12.1 사업기획서와의 경계
프로젝트 관리 기획서는 사업기획서의 상위 방향을 입력값으로 받을 수 있으나, 시장성과 BM 원본은 사업기획서가 담당한다.

### 12.2 서비스기획서와의 경계
프로젝트 관리 기획서는 서비스기획 산출물의 일정과 검토 흐름을 관리할 수 있으나, 기능과 정책 원본은 서비스기획서가 담당한다.

### 12.3 운영기획서와의 경계
프로젝트 관리 기획서는 운영 준비 일정과 산출물 흐름을 관리할 수 있으나, 운영 정책 원본은 운영기획서가 담당한다.

### 12.4 마케팅기획서와의 경계
프로젝트 관리 기획서는 마케팅 실행 일정과 보고 체계를 관리할 수 있으나, 채널 전략과 캠페인 원본은 마케팅기획서가 담당한다.

### 12.5 브랜드기획서와의 경계
프로젝트 관리 기획서는 브랜드 작업의 일정과 승인 흐름을 관리할 수 있으나, 메시지와 인식 원본은 브랜드기획서가 담당한다.

## 14. 금지 패턴

### 14.1 일정표만 붙여놓는 방식
타임라인만 있고 마일스톤, 의존성, 승인 데드라인이 없으면 불완전하다.

### 14.2 보고를 형식적으로만 운영하는 방식
보고 항목이 액션과 결정으로 이어지지 않으면 관리 체계로서 약하다.

### 14.3 이슈를 회의록 안에만 남기는 방식
이슈는 별도 로그와 상태 추적이 필요하다.

### 14.4 리스크를 선언적으로만 쓰는 방식
예:
- 일정 지연 가능성 있음
- 리소스 부족 우려 있음

이런 문장은 대응 오너와 완화 방안 없이 단독 사용 금지

### 14.5 변경 관리를 구두로만 처리하는 방식
범위 변경과 일정 변경은 반드시 기록과 승인 절차가 있어야 한다.

### 14.6 책임 주체 없이 “팀”으로만 적는 방식
누가 책임지는지 अस्प확하면 실행력이 떨어진다.

## 16. 품질 체크리스트

작성 후 아래를 점검한다.

- 프로젝트 목적과 완료 기준이 명확한가
- 범위와 제외 범위가 분명한가
- 조직과 역할과 승인 체계가 보이는가
- 일정과 마일스톤과 의존성이 정리되어 있는가
- 주간 / 월간 보고 체계가 목적과 연결되는가
- 이슈와 리스크가 분리되어 있는가
- 의사결정 관리 방식이 있는가
- 산출물과 버전 관리 원칙이 있는가
- 변경 요청을 통제하는 구조가 있는가
- 실무자가 바로 운영 가능한 수준인가

**2025-26 딜리버리 체크(§6.18~6.24)**:
- **딜리버리 모델(예측/하이브리드/애자일)**이 명시 선언·테일러링됐는가(§6.18)
- 반복/에이전트 작업에 **플로우 메트릭 + Monte Carlo 확률 예측**을 쓰는가(§6.19)
- 소프트웨어 산출물에 **DORA 5지표**(변경실패율·재작업률 포함)가 있는가(§6.20)
- 산출물이 **아웃컴/OKR**에 링크되는가 — feature factory 아님(§6.21)
- 추정에 **레퍼런스클래스(outside view)·낙관편향 상향**이 있는가(§6.22)
- **RAID**(가정·의존 포함) + kick-off **프리모템**이 있는가(§6.23)
- **AI 에이전트 작업**에 스코핑·자율예산/정지·검토 게이트·throughput/품질 추적이 있는가(§6.24)

## 17. AI 실행용 축약 지시문

앞으로 프로젝트 관리 기획서를 작성할 때, 이를 프로젝트를 일정, 보고, 이슈, 리스크, 의사결정, 산출물, 변경사항 관점에서 통합적으로 관리하는 메타 운영 문서로 취급하라.  
먼저 프로젝트 목적, 완료 기준, 범위, 단계, 역할, 타임라인, 보고 체계, 이슈 관리, 리스크 관리, 의사결정 관리, 산출물 관리, 변경 관리 구조를 정의하라.  
주간 / 월간 보고는 상태 공유가 아니라 액션과 정렬 장치로 설계하라.  
이슈에는 반드시 오너, 상태, 해결 목표일을 포함하라.  
리스크는 잠재 문제로 별도 관리하고, 변경 요청은 구두가 아니라 기록과 승인 절차로 통제하라.  
개별 기획 문서의 원본 내용을 중복하지 말고, 프로젝트 관점의 관리 구조와 실행 통제 기준에 집중하라.

## 19. 실무 craft 부록 (현장 패턴 · 적용 프로젝트 환류)

> **출처**: 적용 프로젝트(ICONS) 학습 코퍼스(WBS·Kick-off·제안 실무 강의) 정제본을 방법론으로 환류(2026-06-23). 일반 craft만 수록.

### 19.1 WBS 작성법
- **Task = 실행 가능한 일정 단위.** WBS = "예측 가능한 일"을 만드는 구조. 계층형 1.0 → 1.1 → 1.1.1.
- **1 Task = 0~5일**(초과 시 분해). 일정은 업무시간 기준 **0.8 진수**(점심 제외). 전체에 **리스크 버퍼 0.3~0.5배**.
- **3대 규칙**: ① 결과 중심(산출물이 보여야 — "모델설계"❌ → "모델 정의서 v1.01"✅) ② Task마다 담당자 1명 ③ 예외(리스크·의존성·지연대응) 포함.
- **작성 = 합의 절차**: 초안(기획자) → 리뷰에서 직무별 수치 조정 → '팀의 버전'으로(자기가 만든 일정은 지킨다). 일정 질문은 구체적으로("GET API+응답처리 기준 며칠?").
- 운영: 데일리 스탠드업(15분) · 주간 WBS 리뷰 · 회의록 공유. (템플릿 `50_resources/templates/wbs.md`)

### 19.2 Kick-off 운영
- 정의: 프로젝트 공식 시작 신호탄 — 수행범위·인원·1차 일정·계획 확정 + 이슈·채널 일원화.
- 정하는 것: 일정·인원·소통채널·사업목적 인식공유·담당 지정.
- 문서 8목차: 사전조사 / 핵심가치(고객·사업 각 3) / 전체 플로우 / 단위 플로우 / 디자인 컨셉 / 수행인력 / 커뮤니케이션 채널(채널수=n(n-1)/2) / 일정.
- **Magic Number 3**(가장 강조): 핵심가치·디자인컨셉·수행일정. "내용 중심" 사고 우선.

### 19.3 제안(Proposal) 5단계
환경구성 → 전략수립 → 제작 → 발표 → 종료. **품질 게이트 = 전략점검(전략 흔들리면 전체 흔들림) + 제작 3중 품질검토.** 발표 단계에 **견적 수익율 산출·승인**(고정비 반영) 필수. 종료 단계에 REVIEW(skill 향상 목적, 비난 금지).

### 19.4 단계별 산출물 매핑
제안 진입(제안여부검토서) → 전략수립(자료수집표·전략서·시나리오) → 제작(품질 1·2·3차·목차·제안서) → 발표(예상질문표·검수계획서·견적서) → 착수(킥오프·수행계획서) → 수행(요구사항정의서·일정계획서(WBS)·IA·스토리보드) → 운영(운영명세서·리스크관리·수익율관리).

### 19.5 자주 하는 실수
전략점검 생략 · 견적 수익율 누락 · Task 없이 일정 산정(통합테스트 누락) · 나홀로 M/M 계산 · 막연한 낙관·비공식 진행 · 숨겨진 의존성(선행 미완에 후속 시작) · QA 일정 미배정 · 리스크 미공유(대응=사실+대안).

### 19.6 AI 협업 작업 분해 (클래스 × 담당)
- 모든 Task에 **`클래스(A/B/C)` × `담당(AI / Human / Human+AI / 운영·데이터)`** 2축을 동시 부착 — 변경클래스는 *승인 게이트*를, 담당축은 *실행 주체·병목*을 분리 가시화.
- **인간 액션 레지스트리**: 인간만 할 수 있는 작업(시드 데이터·법률 착수·예산 승인·계약·번역)을 한 표에 모아 `필요 Sprint | 상태 | 비고` 추적 → 인간이 자기 병목을 한눈에. 승인 게이트와는 별개의 *실행* 작업.
- **게이트 대기 SLA**: "게이트 요청 → 인간 승인까지 시간"을 KPI로(예: 72h 이내) — 인간이 율속단계가 되는 구조를 *건수가 아니라 시간*으로 방어.

### 19.7 변경·우선순위의 인라인 통제
- **전역 우선순위 정책 오버레이**: "AI는 목업으로 완료 → 개발 완료 후 실데이터 일괄 QA" 같은 정책을 *날짜 스탬프*로 선언하고 개별 태스크에 태그로 일관 적용 — 우선순위를 태스크별이 아니라 *정책 레이어*로 통제.
- **전략 폐기 인라인 도장**: 전략 전면 폐기 시 "⚠️ 전략 변경(날짜): X 폐기"를 본문에 도장 + *그로 삭제된 태스크 ID 동시 명기* → 별도 변경로그가 아니라 작업 문맥에서 추적.

### 19.10 수행·운영 거버넌스
- **수행계획서**(`execution-plan.md`): 착수 시점 범위·인력·일정·역할 합의 스냅샷(WBS 상위). M/M 매트릭스 + **RACI 업무분장**(단계×역할, ●책임/◎문서작성) + 산출물 4계층(절차→단계→세부→산출물명).
- **커뮤니케이션 채널 공식**: 소통 채널 수 = **n(n−1)/2**(15명→105채널)로 오버헤드 정량화 → 채널 일원화 근거. Kick-off 8목차(사전조사·핵심가치·전체플로우·단위플로우·디자인컨셉·수행인력·커뮤니케이션채널·수행일정).
- **WBS 보강**: 3점 추정(비관/낙관/근사 3컬럼) · 직무 8레인(기획·디자인·퍼블리셔·FE·BE·QA·운영·보안 — 보안·운영 누락 주의). 템플릿 `wbs.md`.
- **회의 ≠ 결과보고**: 회의(방향성) vs 결과보고(결론 합의) 분리. 아이디어 수렴 = KJ법 5단계·블루캡(타이머 1~3분). 회의록 모호어 금지("~같음").
- **운영 산출물**: 운영명세서(`operation-spec.md`) · 오픈후 모니터링(`post-launch-monitoring.md`) · 업무요청처리서(`work-request-ticket.md`) · 용어규약집(`glossary.md`).
