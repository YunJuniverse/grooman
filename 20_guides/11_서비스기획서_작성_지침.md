## 1. 문서 개요

### 1.1 문서명
서비스기획서 작성 지침

### 1.2 문서 목적
본 문서는 AI가 서비스기획서를 작성할 때 따라야 하는 표준 작성 기준을 정의한다.  
서비스기획서는 사용자 경험, 기능 구조, 정보 구조, 정책, 권한, 예외처리를 실제 구현 가능한 수준으로 정리하는 문서다.

### 1.3 문서의 역할
서비스기획서는 사업기획서의 상위 방향을 실제 사용자 경험과 제품 구조로 전환하는 문서다.  
즉, 이 문서는 사용자가 어떤 흐름으로 서비스를 이용하는지, 어떤 기능이 필요한지, 어떤 정책과 예외 기준 아래 서비스가 동작하는지를 정의한다.

## 2. 서비스기획서의 정의

### 2.1 기본 정의
서비스기획서는 **사용자 경험과 서비스 기능 구조를 실제 구현 가능한 수준으로 정의하는 문서**다.  
이 문서는 서비스가 어떤 목적을 가지고, 누가 어떤 흐름으로 사용하며, 어떤 규칙과 정책 아래 동작하는지를 정리한다.

### 2.2 이 문서의 위상 — 컨테이너가 아니라 오케스트레이팅 부모(척추 + 인덱스)

> **핵심 원칙(2026-07 개정)**: 서비스기획서는 IA·기능·정책·화면을 *자기 안에 담는 컨테이너*가 아니라, **의도·결정·경계를 원본으로 소유하고 자식 산출물을 인덱스로 엮는 부모**다. 전통 화면설계서(SB)가 IA+기능+정책+화면을 한 문서에 담던 모놀리스는 우리가 이미 자식 템플릿(`ia-spec`·`functional-spec`·`service-policy`·`user-flow`·`wireframe-spec`·`user-story`)으로 분해해 뒀다. 서비스기획서를 그 재컨테이너로 되돌리면 단일출처가 깨진다. (`CLAUDE.md` §4, 지침 21 개발명세와 동형)

경계 규칙: **부모는 *의도·결정·경계*를 소유하고, 자식은 *상세·열거*를 소유한다. 상세가 전용 템플릿에 SSOT로 살 수 있으면 부모는 복사하지 않고 링크한다.**

| 부모(서비스기획서)가 **원본으로 소유** | 자식 산출물이 **원본**(부모는 링크·결정만) |
|---|---|
| 서비스 정의·목적·문제, 사용자 정의, 상위 시나리오, **MVP 범위·비목표(Non-Goals)**, 크로스-기능 연결 로직, 핵심 제품 결정, AI 제품 로직(진입점·실패 UX·Eval bar), **산출물 인덱스** | IA→`ia-spec` · 기능 상세→`functional-spec` · 정책 상세(ON/OFF)→`service-policy` · 플로우→`user-flow` · 화면→`wireframe-spec` · 유저스토리→`user-story` · API 계약→`api-contract` · AI 상세→16 |

### 2.3 다른 문서와의 차이
서비스기획서는 사업기획서처럼 시장성, BM, 재무를 원본으로 다루지 않는다.  
운영기획서처럼 검수 절차, 제재 단계, CS 처리 프로세스를 상세 원본으로 다루지 않는다.  
마케팅기획서처럼 채널, 퍼널, 캠페인 전략을 중심으로 다루지 않는다.  
브랜드기획서처럼 메시지 체계와 톤앤매너를 원본으로 관리하지 않는다.

서비스기획서는 **사용자가 접하는 경험의 *의도·결정·경계*를 소유하고, 그 상세(IA·기능·정책·화면)를 자식 산출물로 엮는 오케스트레이터**다 — 상세의 원본은 자식 템플릿이다(§2.2).

### 2.4 한 문장 정의
서비스기획서는 **“사용자가 무엇을 어떤 흐름으로 이용할지의 *의도·결정·경계를 정의하고, 상세 산출물을 인덱스로 엮는* 부모 문서”**다.

## 3. 서비스기획서의 목적

### 3.1 핵심 목적
서비스기획서의 목적은 아래와 같다.

1. 서비스의 전체 구조를 사용자 관점에서 정의한다
2. 필요한 기능과 정책을 정리한다
3. 디자이너와 개발자가 구현 가능한 수준의 기준을 제공한다
4. 운영기획과 QA, 후속 협업의 기준이 되는 문서를 만든다

### 3.2 이 문서가 해결하는 문제
서비스기획서가 없으면 다음 문제가 발생하기 쉽다.

- 기능이 많지만 사용자 흐름이 정리되지 않음
- 화면은 있는데 정책이 없음
- 정책은 있는데 권한과 예외 기준이 없음
- 개발팀마다 해석이 달라짐
- 운영 기준과 기능 구조가 충돌함
- MVP와 후속 기능의 구분이 안 됨

### 3.3 필요한 시점
아래 단계에서 서비스기획서가 필요하다.

- 사업 방향이 정리된 후 제품 구조를 설계할 때
- MVP 범위를 확정할 때
- 디자인 착수 전 기능 구조를 정리할 때
- 개발 착수 전 정책과 예외를 정리할 때
- 운영기획과 QA 기준의 상위 입력값을 만들 때
- 기존 서비스 리뉴얼 구조를 재정리할 때

## 4. 핵심 질문

AI는 서비스기획서를 작성할 때 반드시 아래 질문에 답해야 한다.

### 4.1 사용자는 누구인가
- 어떤 유형의 사용자가 존재하는가
- 사용자별 목적과 기대는 무엇인가
- 권한에 따라 어떤 차이가 있는가

### 4.2 사용자는 어떤 흐름으로 서비스를 이용하는가
- 처음 진입부터 핵심 액션까지 어떤 흐름을 밟는가
- 주요 use case는 무엇인가
- 반복 사용 흐름은 어떻게 설계되는가

### 4.3 어떤 기능이 필요한가
- 핵심 문제 해결에 필요한 기능은 무엇인가
- 기능 간 관계는 어떻게 되는가
- 어떤 기능이 필수이고 어떤 기능이 후순위인가

### 4.4 어떤 정책과 권한이 필요한가
- 회원/비회원/관리자/파트너 등은 무엇을 할 수 있는가
- 등록, 수정, 삭제, 공개, 비공개 조건은 무엇인가
- 승인, 제한, 신고 관련 기준은 무엇인가

### 4.5 어떤 예외처리가 필요한가
- 실패 상황은 무엇인가
- 제한 상황은 무엇인가
- 사용자가 막히는 케이스는 무엇인가
- 시스템이 어떻게 반응해야 하는가

### 4.6 MVP 범위는 어디까지인가
- 반드시 포함해야 하는 핵심 기능은 무엇인가
- 나중으로 미룰 수 있는 기능은 무엇인가
- 단계별 확장 구조는 어떻게 되는가

## 5. 주 독자별 기대 포인트

### 5.1 PM / PO
- 상위 사업 방향이 실제 기능 구조로 정리되었는가
- MVP 범위가 명확한가
- 정책과 권한이 빠지지 않았는가
- 운영과 개발로 자연스럽게 연결되는가

### 5.2 디자이너
- 사용자 흐름이 분명한가
- 어떤 화면이 필요한가
- 각 화면의 목적은 무엇인가
- 권한과 정책에 따른 UX 분기가 보이는가

### 5.3 개발자
- 기능 정의가 명확한가
- 입력값/출력값/조건이 보이는가
- 예외처리가 정리되어 있는가
- 권한과 상태값이 분명한가

### 5.4 QA
- 정상 흐름과 예외 흐름이 모두 정의되어 있는가
- 테스트 포인트가 보이는가
- 정책 기준이 검증 가능한 형태인가

### 5.5 운영 협업 담당자
- 운영과 충돌하는 정책이 없는가
- 관리자 개입 지점이 보이는가
- 후속 운영기획서 작성의 기반이 되는가

- AI 기능 목록 (각 기능의 한 줄 정의)
- 각 기능의 ai_relevance 등급 (`tool` / `feature` / `core`)
- 각 기능별 16번 문서 링크 (없으면 *작성 예정*으로 표기)
- 사용자 흐름 중 AI 호출 트리거 위치

본 서비스기획서에는 다음을 *넣지 않는다*.
- 프롬프트 본문 (별도 prompts/ 디렉토리)
- 모델 선택과 비용 추정 (16번 원본)
- 평가 데이터셋 (17번 카탈로그 + 16번 인스턴스)

### 6.14 AI 실패 UX (AI 기능이 있을 때 의무)
AI 출력이 잘못된 경우 사용자가 다음 중 하나 이상의 액션을 *반드시* 수행할 수 있어야 한다.

- **Edit** — 사용자가 결과를 직접 수정
- **Regenerate** — 다시 생성 요청
- **Reject** — 결과를 버리고 원상복귀
- **Escalate** — 사람 검토로 전환 (high-stakes)

또한 모든 AI 출력에는 다음을 명시한다.
- AI 생성물임을 표시하는 라벨/배지
- 출처 또는 신뢰도 표기 (가능한 경우)
- 사용자 책임 경계 (잘못된 출력으로 인한 손해 책임)

상세 UX 흐름과 화면 정책은 본 절에서 다루되, 모델·프롬프트·평가는 16번에 분리한다.

### 6.15 Eval Spec 진입점 (AI 기능이 있을 때)
서비스 단위에서 다음을 합의한다.

- 어떤 사용자 시나리오가 AI 평가 대상인가
- 골든셋의 책임자 (작성·갱신 주체)
- 운영 시 online 평가 표본 비율 (예: 1% 트래픽 샘플링)
- 회귀 발견 시 누가 의사결정하는가

상세 메트릭과 채점 방식은 17번 지침을 참조하고, 인스턴스는 16번에서 정의한다.

## 9. 항목별 작성 기준

### 9.1 서비스 개요 작성 기준
반드시 포함:
- 서비스 목적
- 한 줄 정의
- 해결하려는 문제
- 이번 문서 범위

권장 질문:
- 이 서비스는 사용자에게 어떤 가치를 주는가
- 이 문서에서 어디까지 정의할 것인가

### 9.2 사용자 정의 작성 기준
반드시 포함:
- 사용자 유형
- 사용자 목적
- 역할
- 권한 차이

권장 방식:
- 단순 “회원/비회원”을 넘어서 실제 행동 기준으로 구분한다
- 사용자마다 기대 액션을 함께 정리한다

### 9.3 사용자 시나리오 작성 기준
반드시 포함:
- 시작 조건
- 주요 행동 흐름
- 목표 액션
- 종료 조건

권장 방식:
- 시나리오는 기능보다 행동 흐름 중심으로 쓴다
- 핵심 시나리오와 보조 시나리오를 구분한다

### 9.4 IA 작성 기준
반드시 포함:
- 상위 메뉴
- 하위 메뉴
- 콘텐츠 구조
- 탐색 관계

권장 방식:
- 서비스 전체를 한눈에 볼 수 있게 작성한다
- 사용자가 어디서 어디로 이동하는지 연결성을 드러낸다

### 9.5 기능 정의 작성 기준
반드시 포함:
- 기능명
- 기능 목적
- 사용 주체
- 핵심 액션
- 결과

권장 방식:
- 기능을 나열하지 말고 목적과 연결한다
- 유사 기능은 군집화한다

### 9.6 기능 상세 명세 작성 기준
반드시 포함:
- 진입 조건
- 동작 방식
- 입력 조건
- 처리 로직
- 결과 상태
- 제한 조건

권장 방식:
- 정상 흐름과 예외 흐름을 함께 본다
- 가능한 경우 상태 변화 관점을 포함한다
- 검증 가능성을 높이기 위해 EARS(Easy Approach to Requirements Syntax) 표기법을 권장한다

EARS 기본 패턴:
- "When [trigger], the system shall [response]."
- "While [precondition], the system shall [response]."
- "If [condition], then the system shall [response]."

예시:
- "When the user submits a comment longer than 1000 characters, the system shall display a length warning and prevent submission."
- "If the moderation flag is set, then the system shall hide the post from the public feed."

### 9.7 화면 정책 작성 기준
반드시 포함:
- 화면 목적
- 주요 정보 요소
- 핵심 액션
- 진입/이탈 조건

권장 방식:
- 와이어프레임 설명서가 아니라, 화면의 역할과 정책을 중심으로 쓴다

### 9.8 정책 정의 작성 기준
반드시 포함:
- 등록 가능 조건
- 수정 가능 조건
- 삭제 가능 조건
- 공개 범위
- 제한 조건
- 권한 차이

권장 방식:
- 정책은 반드시 조건문 수준으로 구체화한다
- “관리자 판단”으로 뭉개지지 않게 쓴다

### 9.9 예외 처리 작성 기준
반드시 포함:
- 어떤 상황이 예외인지
- 사용자에게 무엇을 보여줄지
- 시스템은 어떻게 반응할지
- 후속 조치가 필요한지

권장 방식:
- 예외는 나중에 쓰는 부록이 아니라 핵심 설계 요소로 본다
- 실패 상황과 권한 이슈를 빠뜨리지 않는다

### 9.10 권한 체계 작성 기준
반드시 포함:
- 역할 구분
- 역할별 가능한 액션
- 제한 액션
- 관리자 개입 필요 지점

권장 방식:
- 권한표 형태로 작성하면 좋다
- 사용자 유형과 권한을 혼동하지 않는다

### 9.11 MVP 범위 작성 기준
반드시 포함:
- MVP 필수 기능
- 후순위 기능
- 제외 기능
- 향후 검토 기능

권장 방식:
- MVP는 “최소 기능 나열”이 아니라, 핵심 가치 검증 범위로 정의한다
- 왜 제외하는지 이유도 붙이면 좋다

## 10. 작성 원칙

### 10.1 사용자 흐름을 중심에 둔다
서비스기획서는 기능 나열 문서가 아니다.  
사용자가 어떤 맥락에서 어떤 흐름으로 서비스를 이용하는지를 중심으로 작성해야 한다.

### 10.2 기능보다 목적을 먼저 쓴다
각 기능은 반드시 왜 필요한지가 먼저 설명되어야 한다.

### 10.3 정상 흐름과 예외 흐름을 함께 본다
정상 시나리오만 적으면 개발과 QA와 운영에서 충돌이 생긴다.  
예외 상황, 제한 상황, 실패 상황을 반드시 포함해야 한다.

### 10.4 정책은 모호하지 않게 쓴다
“필요 시 제한한다” 같은 표현보다, 어떤 조건에서 무엇을 제한하는지 구체적으로 적는다.

### 10.5 권한과 상태를 분리해서 본다
권한은 누가 할 수 있는가의 문제고, 상태는 현재 객체나 프로세스가 어떤 상황인가의 문제다.  
둘을 혼동하지 않게 작성해야 한다.

### 10.6 MVP와 확장을 명확히 나눈다
모든 아이디어를 한 번에 넣지 않는다.  
핵심 가치 검증에 필요한 범위를 우선 분리한다.

### 10.7 다른 문서와 연결될 수 있어야 한다
서비스기획서는 운영기획, 마케팅기획, 디자인, 개발, QA로 이어질 수 있는 중간 허브 문서여야 한다.

## 11. 자주 하는 실수

### 11.1 기능만 나열하고 시나리오가 없는 실수
기능 목록은 있지만 사용자가 어떻게 흐르는지 없으면 UX 구조가 무너진다.

### 11.2 화면 설명만 있고 정책이 없는 실수
화면은 있는데 등록, 수정, 삭제, 공개, 권한 기준이 없으면 구현과 운영이 흔들린다.

### 11.3 정책은 있는데 예외가 없는 실수
정상 상황만 있으면 실제 서비스에서 빈틈이 커진다.

### 11.4 사업기획 내용을 반복하는 실수
시장성, BM, 경쟁 분석을 길게 반복하면 문서 목적이 흐려진다.

### 11.5 운영기획과 경계가 없는 실수
서비스 정책과 운영 절차를 구분하지 않으면 문서가 과도하게 복잡해진다.

### 11.6 MVP 기준이 없는 실수
기능은 많은데 우선순위가 없으면 실행이 늦어진다.

### 11.7 권한 체계가 빠지는 실수
권한 정의가 없으면 개발과 운영에서 해석이 달라진다.

### 11.8 AI 출력을 결정론처럼 다루는 실수
"AI가 알려주는 정확한 가격", "AI가 자동으로 결정한 승인" 같은 표현은 AI의 확률적 특성을 무시한다.
AI 출력은 항상 *불확실성을 가진 추천*으로 다루고, 사용자가 검토·수정·거절할 수 있는 경로를 제공해야 한다.

### 11.9 AI 기능 진입점 없이 AI를 끼워넣는 실수
AI 기능을 6.13~6.15 절에서 진입점으로 분리하지 않고 본문 곳곳에 분산시키면, 평가·가드레일·운영 책임이 흐트러진다.

### 11.10 AI 실패 UX 누락
edit/regenerate/reject 액션 없이 AI 출력을 그대로 노출하면 사용자 신뢰가 무너진다.
AI 기능이 있는데 6.14절이 비어 있으면 미완성 문서로 취급한다.

## 12. 다른 문서와의 경계

### 12.1 사업기획서와의 경계
서비스기획서는 사업 방향을 입력값으로 받지만, 시장 분석과 수익 구조를 원본으로 다루지 않는다.

### 12.2 운영기획서와의 경계
서비스기획서는 서비스 내 정책과 권한을 정의할 수 있으나, 운영팀의 검수 절차와 제재 프로세스 원본은 운영기획서가 담당한다.

### 12.3 마케팅기획서와의 경계
서비스기획서는 랜딩, 가입, 전환 관련 서비스 포인트를 정의할 수 있으나, 채널 전략과 캠페인 실행안은 마케팅기획서가 담당한다.

### 12.4 브랜드기획서와의 경계
서비스기획서는 브랜드 경험 반영 포인트를 일부 포함할 수 있으나, 메시지 체계와 표현 원칙의 원본은 브랜드기획서가 담당한다.

### 12.5 프로젝트 관리 기획서와의 경계
서비스기획서는 기능 구조와 정책의 원본을 정의하고, 프로젝트 관리 기획서는 해당 서비스기획 산출물의 일정, 검토, 승인, 변경과 같은 실행 관리 구조를 담당한다.

### 12.6 AI 기능 기획서(16)와의 경계
서비스기획서는 *어떤 AI 기능이 어디에 들어가는가*까지만 정의한다.
AI 기능 각각의 입력·출력 계약, 모델·프롬프트, 평가 명세, 가드레일, fallback은 16번 AI 기능 기획서가 원본이다.
서비스기획서에서 6.13~6.15 절은 AI 기능 *진입점*만 표기하고 상세는 모두 16번으로 분리한다.

### 12.7 평가 및 가드레일 지침(17)과의 경계
서비스기획서는 17번을 *언급하고 참조*만 한다.
조직 표준 메트릭, 가드레일 카탈로그, 인간 검토 게이트의 원본은 17번이다.

### 12.8 개발 마스터 플랜(18)과의 경계
서비스기획서는 기능을 *정의*한다 (시나리오·입출력·정책).
그 기능을 *어느 페이즈에 어떤 순서로 빌드하는가*는 18번 마스터플랜이 결정한다.
18에서 새 기능 ID를 만들지 않으므로, 빌드 페이즈에 필요한 모든 기능은 11번에 먼저 정의되어야 한다.

### 14.4 예외 없는 정상 흐름만 설명

예:
- 신고 접수 후 관리자 A가 1차 검수하고 24시간 내 처리

## 16. 품질 체크리스트

작성 후 아래를 점검한다.

- 서비스 목적이 첫 부분에 명확한가
- 사용자 유형이 충분히 구체적인가
- 핵심 시나리오가 기능보다 먼저 보이는가
- IA와 기능 구조가 연결되는가
- 기능마다 목적이 드러나는가
- 정책과 권한이 명확한가
- 예외 처리와 실패 상황이 포함되어 있는가
- MVP와 후속 기능이 구분되는가
- 운영기획과 서비스기획의 경계가 보이는가
- 디자인/개발/QA가 실제로 활용 가능한 수준인가

**인덱스 모델 체크(§2.2 — 단일출처)**:
- **산출물 인덱스(§6.0)가 있고 최신인가** — 자식 산출물마다 상태·링크·소유자가 채워졌는가
- **자식과 중복이 없는가** — IA·기능 상세·정책 ON/OFF·화면을 여기에 *전수 복사*하지 않았는가([인덱스] 항목은 링크+결정만)
- **비목표(Non-Goals)가 1급으로 있는가** — 하지 않을 것 + 후속 진입 조건
- (AI 있으면) **품질 bar가 있는가** — Eval 임계·실패 UX(Edit/Regenerate/Reject/Escalate)·행동 제약(절대 금지)

## 17. AI 실행용 축약 지시문

앞으로 서비스기획서를 작성할 때, 이를 사용자가 무엇을 어떤 흐름으로 이용하게 될지를 정의하는 구현 기준 문서로 취급하라.  
먼저 사용자 유형, 핵심 시나리오, 서비스 구조, IA, 주요 기능, 정책, 권한, 예외처리, MVP 범위를 정의하라.  
기능은 나열하지 말고 사용자 목적과 흐름 중심으로 설명하라.  
정상 흐름뿐 아니라 실패 상황과 제한 조건도 반드시 포함하라.  
상세 시장 분석과 재무 구조는 사업기획서로, 운영 절차 원본은 운영기획서로, 채널 실행안은 마케팅기획서로 분리하라.  
문서는 디자인, 개발, QA, 운영 협업이 가능한 수준으로 구체화하라.

## 19. 실무 craft 부록 (현장 패턴 · 적용 프로젝트 환류)

> **출처**: 적용 프로젝트(ICONS) 학습 코퍼스(서비스기획 강의·실무 자료·표준 산출물 케이스) 정제본을 방법론으로 환류(2026-06-23). 일반 craft만 수록.

### 19.1 핵심 원리
- **서비스기획서 ≠ 화면설계서.** `ASIS분석 → 벤치마킹 → 요구사항/정책 → IA → 프로세스/플로우 → 화면설계`의 묶음. 근거(ASIS 문제점) 없는 화면 금지.
- 요구사항은 **Needs(What)가 아니라 Purpose(Who·How)**로 받는다(현업 UI 요청을 그대로 받지 말고 목적 역추적).
- MVP 중심·**측정 가능한 UX**: ① 만족 타깃/전체 타깃 ② Σ(중요도×만족)/전체 타깃.

### 19.2 산출물 흐름 + 12단계 산출물 체계
ASIS→TOBE 흐름: ASIS 현황분석 → 벤치마킹 → 인사이트 도출 → 요구사항 정의 → 정책 정의 → TOBE IA → 프로세스/플로우차트 → 화면설계(SB). UI 정밀도 4단계: **와이어프레임 → 목업 → 프로토타입 → 스토리보드(SB)**.
표준 산출물 12단계: ① 서비스기획서 ② 유저스토리/시나리오 ③ 정책정의서 ④ IA(정보구조기능정의서) ⑤ 메뉴트리 ⑥ 플로우차트 ⑦ 기능요구사항정의서 ⑧ 스토리보드 ⑨ 화면설계 ⑩ UIUX (+ ⑪ 인사이트보고서 ⑫ 참고데이터). 서비스기획서는 이 중 *한 장*이므로, 나머지 산출물 작성 현황을 함께 표기한다 — **현황 표기의 정본 = §6.0 산출물 인덱스**(부모=인덱스, §2.2·§19.14).

### 19.3 데이터 무결성 원칙 (이력 보존)
**정보 변경이 과거 이력에 영향을 주어서는 안 된다.** 변경은 *시점부터* 적용하고 과거 기록은 그대로 둔다(가격·정책·확률 변경 시 과거 주문·결과·로그는 당시 값 보존 = 스냅샷/버전 박제). 이력성 데이터 설계의 1순위 원칙.

### 19.4 정책정의 실무
- 정책 = 반복된 행위에서 발견한 규칙(서비스의 "법"). 두 관점: 고객/운영자가 자주 만드는 문제.
- 두 출력: **행동가이드(OFF=수동 해결)** vs **해결책(ON=자동화→서비스화)**.
- 작성 패턴(조건문): "반복되는 [무엇을] → 어떻게 [해야 한다] → [새 문제]". 상담안내는 단계 서술, 근거·예외를 불릿 구분. (2-시트: 관리대장 + 상세, 템플릿 `service-policy.md`)

### 19.5 스토리보드(화면설계) 실무
- 화면 영역 표준(공동작업 기준서): `Global(GNB)/Top/Identity/LNB·Intro·RNB/Bottom/Footer/Personalize`.
- SB 페이지: 좌=화면(목업)+마커 ①②③ / 상=제목·Date·Ver / 우=Description(같은 번호 매칭). 파일명 `[영역]_StoryBoard_YYMMDD_Vx.xx`, 마스터 템플릿 고정 후 추가.
- **디스크립션 8요소**: 요약 / 입력폼 상태(전·중·후) / 버튼 상태(활성·비활성·선택) / 로직 / 에러 / 제한개수 / 외부액션 / 별도문서. 시각: 한 화면 그룹 3개↑ 혼선(근접성), 드래그 대신 수치 입력·정렬.

### 19.6 관리자(Admin) 설계
- Admin은 Front에서 역도출(프론트 관리 / DB 관리 / DB 확인=대시보드·통계). Input→Output 프로세스로 정의.
- IA: INDEX=GNB 주요 데이터 노출 · **DB 변화 적은 것→많은 것 순**(회원>상품>게시판>운영>통계) · 3depth 초과 금지 · 1순위 원칙 = 데이터 무결성(§19.3).

> **추가 환류(2026-06-23, GambleScan / METH-040)**: 아래 §19.7~19.9 는 적용 프로젝트 GambleScan 실전 서비스기획·개발명세 산출물에서 정제. 일반 craft만.

### 19.7 요구사항 4층 추적 (BR→UR→FR→NFR)
- 요구사항을 **사업요구(BR) → 사용자요구(UR) → 기능요구(FR) → 비기능요구(NFR)** 4층으로 분리, 각 항목에 ID·우선순위·사유. BR엔 `Reason`, UR엔 **`Acceptance Signal`**, FR엔 `Input/Output` 컬럼 → 상위 의도가 하위 명세까지 추적.
- **Acceptance Signal**: 사용자 요구를 추상 니즈가 아니라 *검증 가능한 신호*로. "간단한 폼"❌ → "3단계 이내 제출 + 접수확인 메일"✅, "빠름"❌ → "로드 2초 내·핵심지표 above the fold"✅. QA가 그대로 테스트 케이스로 사용. (EARS는 시스템측, 이건 사용자측)

### 19.8 다면(N-sided) 서비스 설계 (권한·인계·계층)
- **권한 매트릭스 2축**: `역할 × 화면영역(Surface)`로, 셀에 액션 강도(읽기만/전체/참여/응답의무/불가). 다면 서비스의 *비대칭 권한*(자기 Surface는 전체, 타 Surface엔 "응답 의무" 등)을 담음 — 1축 역할×CRUD 표의 확장.
- **컨텍스트 인계 테이블**: 영역 간 행동의 연결을 `트리거 → 연결 방식 → 효과` 3열로. 정적 sitemap이 못 담는 *상태 인계*(이동 시 무엇을 들고 가는가)를 설계.
- **사용자 피라미드 90/9/1**: 참여 강도 × 모집단 비율(탐색형~90 / 기여형~9 / 수익형~1)로 계층화 → 기능 우선순위(다수 읽기 vs 소수 거버넌스) 직결.

### 19.9 산출물 디테일 규약
- **라벨링 결정 기록**: IA에 용어 표기 정책을 `채택 라벨 / 회피 라벨 + 사유` 2열로 박제(ADR화) → 산출물·코드·디자인 용어 일관성 강제.
- **MVP 제외 = 1급 산출물**: 뺀 기능마다 *제외 사유* + *후속 Phase 진입 조건*(예: "파트너 N개 확보 시")을 게이트 표로. "In Scope"만큼 "Out of Scope + 조건"을 명시.

> **압축 보충(2026-06-23, ICONS / METH-041)**: METH-039 압축 시 빠진 서비스기획 실무 디테일을 ICONS knowledge 원본에서 보충.

### 19.10 서비스 정의·평가 실무 보충
- **서비스 정의 3종**: Target은 *측정 가능*하게 구체화(예: "월 N만원 자기관리에 쓰는 20~30대 싱글 직장인") · **Core(핵심 성공요인) vs Support** 구분 · **출시유형 3종**(기존 개선 / 실물 개선 / 신개념)별 로드맵 보폭 차등.
- **ASIS UIUX 평가 7항목 루브릭**: 기능성 · 사용성 · 직관성/일관성 · 편의성 · 인터페이스 · 접근성 · 정보구조 (ASIS 화면 평가의 구체 척도).

> **추가 환류(2026-06-23, 원본 코퍼스 / METH-042)**: 원본 서비스기획 학습 자료(강의 교안·실전 케이스) 직접 정독분.

### 19.11 ASIS 분석·정량화
- **ASIS 3축 프레임**: 현황분석을 `IA(menu depth·page type·permission·update·admin) / UIUX(layout·structure·channel·identity·contents·interface) / Function(key·added·admin)` 3축 고정 체크.
- **UIUX 정량화**: §19.10 7루브릭을 *0~점 채점 → 레이더 차트 → 준수율 %*로. 자사×타사 오버레이로 갭 가시화.

### 19.12 요구사항·시나리오 실무
- **요구사항 접수·수용판정 대장**: BR→UR(§19.7) *입구*에 현업 raw 요구를 ID화 — `요구ID·명·내용·중요도(5점)·요청자(VOC/설문)·협의일/작성자·수용여부·버전`. 수용여부 **enum=검토/수용/불가/재협의/진행안함**(부정 시 사유 의무).
- **유저 시나리오 7요소**: user-story 카드와 별개로 `사용자그룹·시놉시스·니즈·불편·대안·서비스구성·감성요소(톤앤매너)` 표 → 시나리오→기능/콘텐츠 추출→Grouping으로 동선 도출.
- **서비스 개요 = 목적(정성)/기대효과(정량) 분리**: 문서 최상단 개요에서 *목적은 정성, 기대효과는 그 목적의 정량화*로 이항 분리(§19.7 Acceptance Signal과 짝).

### 19.13 화면설계·산출물 거버넌스
- **디스크립션 문체 5원칙**: 간단히·짧게·중복회피·중요한 것 앞에·불필요 설명 금지(각 사례/불가사례 대조). 라벨 사전: 회원(←사용자/이용자)·확인(←저장/등록)·취소(←캔슬)·이전(←이전보기). (§19.9 라벨링 결정 기록의 본문)
- **시나리오 3상태 병치**: 한 화면을 입력 *전/중/후* + 유형 A/B/C 변형을 한 지면에 마커(①②③)로 병치 → 상태 전이를 그림으로 검증(§19.5 입력폼 상태의 배치 규약).
- **화면설계서 7구성 + 표지 거버넌스**: SB 문서 = `개정이력·정책정의·TOBE IA·프로세스/플로우·레이아웃·시나리오·디스크립션` 7요소. 표지에 **승인사항 테이블(작성/승인자 × 성명·직급·일자·서명란)** + Revision History·Policy Summary + 페이지 코드 `FO-SU-01-01`(영역-기능-Depth-순번). 인간 승인 게이트의 산출물 내 증거. 템플릿 `wireframe-spec.md`.
- **DB 3종 분류로 Admin 구동**: 데이터를 비정형/반정형/정형 3종으로 분류 → Admin 메뉴·저장구조 설계 구동. Admin INDEX(GNB)=각 메뉴 핵심 지표 노출(§19.6 보완).
- **IA = 진척·SB·산출물 통합 보드**: ia-spec에 `SB No.·개발 Point(스토리포인트)·직무별 작업체크(기획/디자인/코딩/개발)·milestone·코딩 URL·개발 URL·접근성 체크` 컬럼 추가 → 정보구조+진척+산출물 링크를 한 행에서.
- **마이크로카피/UX 라이팅**: 에러·확인·공백 상태별 카피 규칙(템플릿 `microcopy.md`). 화면설계 States 칸에 *무엇을 쓸지*까지.
- **용어 통일·페이퍼 목업·신개념 온보딩**: *자사 용어사전*으로 어휘 통일(이용자→회원·캔슬→취소; 산출물·코드·디자인 일관) · 그리기 전 손스케치 → **페이퍼 목업**으로 사용성 선검증 · 신개념 = "나쁘다(안 익숙함) × 매력적(고유가치)" → 온보딩으로 *익숙함 비용*을 상쇄.
