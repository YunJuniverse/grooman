# 03. AI 관찰 로그 작성 규칙 (L1)

> 본 가이드는 L1 관찰 로그(`50_resources/ai_observations/`)의 *단일 작성 규칙* 이다.
> 위상: [10_foundation/WHITEPAPER.md](../10_foundation/WHITEPAPER.md) §5 L1.
> 핵심 가치: **AI가 자동 기록한다. 사용자는 채우지 않는다. 부패해도 OK — 정제는 L3가 한다.**

---

## 1. 의무 사항 (Hard Rules)

| 규칙 | 위반 시 |
|---|---|
| AI가 매 세션 종료 시 *반드시* 1개 파일 작성 | CI fail (`최근 7일 관찰 0건이면 fail`, 백서 §9 리스크 1) |
| 사용자가 손으로 작성·수정 금지 | 원천성 훼손 — L3 신호 신뢰 불가 |
| 작성된 파일은 절대 삭제 금지 | 학습 데이터 손실 |
| 시간 표기는 모두 UTC ISO 8601 | 환경 이식성(백서 §0 제0원칙) |
| 파일명은 `YYYY-MM-DD_<slug>.md` (UTC 날짜 + 영문 kebab-case slug) | 인덱싱·정렬 단절 |

---

## 2. 파일 위치·이름

```
50_resources/ai_observations/YYYY-MM-DD_<slug>.md
```

slug 가이드:
- 영문 소문자 + 하이픈
- 그 세션의 *작업 요지* 를 3~5단어로
- 예: `2026-05-07_l0-bootstrap-and-handoff-sim.md`, `2026-06-12_supabase-auth-fix.md`

같은 날 여러 세션 시: `2026-06-12_supabase-auth-fix.md`, `2026-06-12_dashboard-ui-pass.md` — slug로 구분.

---

## 3. 스키마 (전체)

YAML frontmatter 6 필드 + 자유서술 1단락. 그 외 절대 추가하지 않는다.

```yaml
---
session_id: 2026-05-07_l0-bootstrap-and-handoff-sim
authored_by:
  agent: claude-sonnet-4-6        # 모델 ID
  tool: claude-code-cli           # 도구 (cursor / codex / web-ui / ...)
  host_os: darwin-25.4            # 환경
domain: meta                       # 도메인 식별자 (skeleton과 일치)
task_type: bootstrap               # bootstrap | feature | bugfix | refactor | research | docs
stack_used:                        # 실제 사용한 스택·버전 (개략)
  - "python3"
  - "git@2.46"
flow_used: ad-hoc                  # skeleton:<id>-<version> | ad-hoc
friction:                          # 시간 잡아먹은 마찰점 (없으면 빈 배열)
  - id: F-001                      # 세션 내 로컬 ID
    where: "git mv WHITEPAPER.md (untracked file)"
    cost_minutes: 2
    resolution: "use plain mv then git add at new location"
    repeat_of: null                # 기존 Catalog 엔트리 hit이면 ID 기록 (예: C-014)
prompt_patterns:                   # 효과적이었거나 비효율이었던 프롬프트 패턴
  - intent: "folder restructure proposal"
    success: true
    rounds: 2
---

(자유서술 한 단락 — 다음 번 같은 도메인을 시작하는 자기 자신에게 줄 한 마디. 50~150자.)
```

### 3.1 필드 의미

| 필드 | 형식 | 설명 |
|---|---|---|
| `session_id` | string | 보통 파일명(.md 제외)과 동일 |
| `authored_by.agent` | string | 모델 ID (예: claude-sonnet-4-6, gpt-5, gemini-2.5-pro) |
| `authored_by.tool` | string | 호스팅 도구 ID |
| `authored_by.host_os` | string | OS+버전 (darwin-25.4, linux-6.5, win-wsl2 등) |
| `domain` | string | Skeleton 도메인 식별자와 동일 표기 |
| `task_type` | enum | 6종 중 1 — *반드시* enum 내 |
| `stack_used` | array<string> | 사용한 핵심 의존성·도구. 너무 자세히 쓰지 말고 *재현에 필요한 정도* |
| `flow_used` | string | 적용한 Skeleton ID 또는 `ad-hoc` |
| `friction[].cost_minutes` | int | 정직하게 추정. 0~5 / 5~30 / 30+ 구간 |
| `friction[].repeat_of` | string \| null | 기존 Catalog 엔트리 hit이면 ID, 아니면 null |
| `prompt_patterns[].rounds` | int | 의도 한 가지를 완수하는 데 든 turn 수 |

## 4. 작성 규율 (Discipline)

### 4.1 금지

- ❌ "오늘 우리는 X를 잘 했다", "사용자가 이런 점을 좋아할 것이다" — *서술·칭찬·예측*
- ❌ 5단락짜리 회고문 — *간결성*
- ❌ "어쩌면", "아마도", "~할 수도 있다" — *추측*
- ❌ 절대 경로 (`/Users/<name>/...`) — 환경 이식성 위반
- ❌ AI/도구별 슬랭 (Claude만 아는 표현 등) — 다른 모델이 못 읽음
- ❌ 빈 friction 배열에 `[]` 대신 `[None]` 또는 의례적 항목 채우기

### 4.2 권장

- ✅ 코드 식별자·파일 경로(저장소 상대)는 그대로 인용
- ✅ 친근한 어조 금지, *전보문* 어조
- ✅ "재현 가능한 사실"과 "시간 비용"만
- ✅ friction이 없으면 빈 배열 (`friction: []`)
- ✅ 자유서술은 *다음 세션 자기 자신에게 보내는 메모* 어조

### 4.3 좋은 예시 (자유서술)

> ✅ "0_foundation 컨벤션은 백서 §부록 C로 명문화됨. 새 폴더 추가 시 빈 슬롯(50/60/70/80) 사용 가능 — 사용 전 백서 갱신(Class C)."

### 4.4 나쁜 예시

> ❌ "오늘은 백서를 작성하고 폴더 구조를 정리하는 보람찬 작업이었습니다. 다음에는 더 잘 할 수 있을 것 같아요!"

## 5. 작성 방법 — `observe` CLI (정본)

### 5.1 반드시 CLI로 작성 (직접 `cat >` 금지)

관찰 로그는 **`methodology.py observe` CLI로만** 작성한다. CLI가 frontmatter·enum·길이·slug를 atomic하게 검증하므로, 직접 `cat > .md`로 쓰면 형식 오류로 **wrap·CI가 fail**한다. (헌법 CLAUDE/AGENTS §2 ④가 이 규칙을 세션 종료 절차로 의무화.)

```
python3 60_tools/methodology.py observe \
  --slug "<kebab-case-slug>" \
  --task-type <bootstrap|feature|bugfix|refactor|research|docs> \
  --summary "<50~150자 자유서술>"
```

- `--agent`/`--tool`/`--host-os`/`--domain`은 생략 시 `.ai/context.json`·`uname`에서 자동 채움(명시 시 override).
- `--stack`(복수 가능)·`--flow-used`·`--intent`/`--rounds`는 선택.
- 종료 절차: observe 작성 → `wrap`(→`4/4 ✓`) → `ship`. observe를 빠뜨리면 wrap이 관찰 로그 미갱신으로 통과 못 함.

### 5.2 friction 캡처 — `--friction` (학습 루프의 원료)

**비자명한 문제·재발·막힘**을 겪었으면 `--friction`도 남긴다. **형식은 positional 파이프**(`key=value` 아님):

```
--friction "<where>|<cost_minutes:int>|<resolution>|<repeat_of>"
# 예: --friction "git ls-tree 한글경로 octal-escape로 grep 오탐|8|core.quotepath=false로 재확인|null"
```

- `cost_minutes`는 정수. `repeat_of`는 기존 Catalog 엔트리 hit이면 그 ID(예: C-001), 아니면 `null`.
- **여러 번 지정 가능**(마찰 N개 = `--friction` N개).
- **마찰 없으면 생략**(강제 아님 — 노이즈 방지). 5분 미만 사소한 건 보통 생략, *진짜 마찰*만.
- **`where`는 재발 판정 키** — 같은 마찰은 *같은 표현*으로 적어야 thinktank가 ≥2회로 집계한다.

## 6. 누가 읽는가 + 학습 루프에서의 위치

| 독자 | 시점 | 목적 |
|---|---|---|
| **L3 씽크탱크** | 회고 직전 `methodology.py thinktank` | friction 클러스터링 → Catalog 승급 후보 |
| **다음 세션의 AI** | 부팅 시 (선택적) | 같은 도메인 최근 N건을 컨텍스트로 로드 |
| **사람** | 분기 회고 (드물게) | L3 리포트 신뢰도 검증 |

### 6.1 관찰 로그 = 학습 루프의 *원료*

이 로그는 방법론이 스스로 학습하는 파이프라인의 **진입점**이다:

```
observe --friction (L1 원료, 이 문서)
   ↓  thinktank (반복 마찰 where ≥2회 → 승급 후보 마킹, 자동 승급 아님)
_pending/P-NNN (사람 작성)
   ↓  승급 머지
C-NNN active catalog (검증된 정설)
   ↓  bakes-in.json
skeleton (코드로 구움) → 새 프로젝트 주입
```

- **마찰을 안 남기면 루프가 굶는다** — thinktank가 집계할 게 없어 승급 후보가 안 나오고 catalog가 빈다. 그래서 §5.2의 `--friction` 캡처가 자가발전의 실질 연료다.
- 상세: `50_resources/catalog/_README.md` §3(원료 수집)·`50_resources/skeletons/_README.md`. 승급 규칙은 catalog `_README` §3.
- L3 마이닝 인프라는 v0(문자열 카운트) — 정교화 전까지는 *읽히지 않더라도 적재만* 해도 가치(데이터 누적이 자가발전의 전제).

## 7. 흔한 실수 (시드 단계)

| 실수 | 교정 |
|---|---|
| 작성을 깜빡 | 다음 세션 첫 행동으로 *직전 세션*의 관찰을 거꾸로 작성 (가능하면) |
| friction에 너무 많이 채움 | 시간 비용 5분 미만은 보통 생략. 진짜 마찰만. |
| `repeat_of` 항상 null | Catalog 검색 안 함. 작성 전 grep 의무. |
| 자유서술 5단락 | 1단락 강제. 길면 잘라낸다. |
| 한국어/영어 혼용으로 검색성 저하 | 식별자·경로는 그대로, 서술은 일관 언어. *L3가 다국어 처리 가능하지만 일관성이 클러스터링 품질을 높임*. |

---

## 8. 본 가이드 자체의 진화

L3 마이닝이 가동되면, *본 가이드가 효과적인지* 도 측정 대상:

- 작성된 관찰의 평균 길이 (너무 길면 본 가이드 §4 강제 강화)
- friction 필드 누락률
- `repeat_of` 채워진 비율 (낮으면 grep 절차 강화)

본 가이드 변경은 Class B (백서 §3 변경 클래스 — 자가발전 신호 품질에 영향).
