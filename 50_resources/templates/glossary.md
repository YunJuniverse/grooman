# 용어규약집 (Glossary) — 템플릿

> 용도: 프로젝트/계약 **표면 용어 통일** — 영업~운영 단계에서 고객·수행사가 같은 말을 같은 뜻으로 쓰게 한다(제안서·산출물·회의록 라벨 일관). 지침 `20_guides/15_프로젝트_관리_기획서_작성_지침.md` §6 · `20_guides/11_서비스기획서_작성_지침.md` §19.9(라벨링) 참조.
> **경계(SSOT, 중복 금지)**: 이 문서 = *SI 단계별 표준어* 통일(계약·산출물 표면). 도메인 개념의 *코드까지 흐르는 canon*(유비쿼터스 언어·바운디드 컨텍스트·Code/UI 매핑·린트 훅)은 `context-glossary.md`가 정본 — 개념 정의는 거기서, 여기선 링크만(둘 다 채우지 말 것).

> **단계** enum = `영업` / `제안` / `착수` / `설계` / `구현` / `운영`.
> **상태** enum = `Approved`(정본) / `Deprecated`(폐기 — 유사용어로 강등, 신규 사용 금지). 폐기어도 지우지 말고 남겨 추적성 유지.

**원칙**: ① 같은 개념에 **표준용어 1개** — 유사용어 칸에 *버려진 표현(영문·약어 포함)* 을 모아 라벨링 일관성을 강제(검색엔 잡히되 문서 표면엔 표준어만). ② **예시(용례)** 를 붙인다 — 정의보다 용례가 오해를 줄인다. ③ 도메인 개념 자체(코드·UI·바운디드 컨텍스트)는 여기 늘리지 말고 `context-glossary.md`로 — 이 문서는 *계약 표면 라벨*만 얇게 유지.
