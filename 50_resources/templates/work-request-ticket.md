## 3. 완료 기준 (Definition of Done)

> 무엇이 충족되면 이 티켓을 닫는가 — 검증 가능하게(재현 불가 확인·회귀 통과·요청자 확인 등). 빈칸 방치 시 "닫혔지만 안 고쳐짐" 발생.

## 4. 변경관리 (티켓유형=변경일 때만)

| 항목 | 내용 |
|---|---|
| **변경 유형** | `표준(Standard, 사전승인·저위험)` / `정상(Normal, 승인 필요)` / `긴급(Emergency, 사후 리뷰)` (지침 §6.20) |
| **Change Class** | `A` / `B` / `C` (CLAUDE.md §3 — 스키마·인증·외부계약·과금·정책 변경은 B/C) |
| **롤백 계획** | (변경 & Class≥B는 **필수** — 실패 시 되돌리는 절차) |
| 배포 방식 | (canary/blue-green/staged·변경 프리즈 창 확인) |

> 승인 게이트는 *위험 변경만*: 표준/Class A는 승인 없이 진행, **정상·긴급 또는 Class B/C는 승인권자 서명 필수**(과잉 게이트 방지 ↔ 위험 무게이트 방지 균형).

> **상태 워크플로** = `접수` → `검토(영향·우선순위)` → `승인/반려` → `처리중` → `검증(완료기준)` → `완료`

**원칙**: 운영 요청은 *접수 → 영향·우선순위 → (변경이면)변경관리 → 결정 → 검증* 흐름을 한 장에 남긴다. ① **티켓유형**으로 경로를 가른다(요청/장애/변경). ② **우선순위=영향×긴급**으로 순서를 정한다(요청구분≠우선순위). ③ **변경은 Class·롤백·승인**을 CLAUDE.md 변경 클래스와 연결 — 임의 수용 금지, 위험 변경은 서명 근거 필수.
