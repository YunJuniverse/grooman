## 검수 결과

> 결과 enum: **1 정상 / 2 경미오류(오타 등) / 3 심각오류(미작동)**. · 태그: 정상경로 / 부정(negative) / 경계(boundary). · 작성: AI초안 / 사람검토 표기.

| ID | 요구사항 | 검수항목 | 단계(시나리오형) | 예상결과 | 실제결과 | 결과 | 태그 |
|----|------|------|------|------|------|------|------|
| TC-001 | REQ-… | | | | | | |

- **케이스 ID·요구사항 ID**는 필수 — 결함·RTM이 개별 케이스를 참조하려면 안정 ID가 있어야 함(ISTQB).
- **실제결과**는 결함 증거 — 결과 enum(1/2/3)과 별개로 *관찰된 실제 동작*을 적는다(ISO 29119-3).
- **부정·경계 케이스**를 태그로 표시해 실패경로 커버리지를 보이게 한다(무작위형은 태그 생략 가능).
- 케이스별 **우선순위/심각도**가 필요하면 별도 열 추가(계획서의 위험 우선순위에서 상속).

### (선택) Given-When-Then — 시나리오형만
복잡한 흐름은 Gherkin으로 단계를 구조화(선택, lean 유지):
```
Given <사전조건>  When <행동>  Then <예상결과>
```
> 무작위형에 GWT 강제 금지 — 탐색적 검수의 자유도를 죽인다.

## 2모드 안내

| 모드 | 설명 | ISTQB 대응 |
|------|------|------|
| 무작위형 | 임의 검수자가 자유롭게 항목을 골라 실사용 관점으로 결함 발견 | 경험 기반(experience-based) |
| 시나리오형 | 영역별 사전정의 검수표(Header·Main·Footer 등)로 빠짐없이 순회 | 명세 기반(specification-based) |

**원칙**: 무작위형으로 실사용 결함을, 시나리오형으로 누락 결함을 잡는다. 심각오류(3)는 1건이라도 검수 미통과 — 경미오류(2)와 섞어 묻지 말 것.
