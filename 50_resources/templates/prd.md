## 5. 기능 요구사항

> 표기: **P0~P3** = §10 출시(빌드) 순서. **`M`** = 필수(MUST), **`S`** = 권장(SHOULD).
> 각 요구는 **테스트 가능한 인수기준**(원자적 술어, 예 `image.size_mb <= 5`) + 입출력 예시 1-2개(ground truth)를 붙인다 — AI 코딩 에이전트가 바로 소비. 상세 추적은 `requirements-spec`, 구현 명세는 `functional-spec`으로 **링크(복사 금지)**.

## 9. AI 제품 요구 (AI/LLM 기능이 있을 때만)

> AI는 *확률적* — 이진 인수기준(§5)으로 품질을 표현할 수 없다. 아래를 요구로 명시. *상세*(프롬프트·모델·평가셋)는 `20_guides/16_AI_기능_기획서` / 평가·가드는 `17`이 원본 — 여기엔 제품 레벨 결정·경계·bar만.

### 9.1 품질 bar (Eval 임계)
> 차원별로 (지표 · 측정법[알고리즘/AI-judge/사람] · 평가셋) + **임계 3단(launch / target / aspirational)**. 등식 아닌 분포·신뢰구간으로.

| 품질 차원(정확·완결·관련·톤·안전·형식) | 측정법 | 평가셋 | launch | target | aspirational |
|---|---|---|---|---|---|
| (예: 사실성) | 사람 200표본 | golden-200 | 90% | 95% | 99% |

### 9.2 가드레일 (행동 제약 — "절대 안 함")
- **입력 필터**(거부할 프롬프트) · **출력 검증**(차단할 응답) · **행동 경계**(실행 가능/불가) · **에스컬레이션 트리거**(사람 전환).

### 9.3 실패 모드 → Fallback
| 실패 유형 | 트리거 | 시스템 응답(재시도/저모델 강등/캐시/사람) | 사용자 UX |
|---|---|---|---|
| 품질 실패 | 저신뢰·나쁜 출력 | | Edit/Regenerate/Reject/Escalate |
| 가용성 실패 | API 다운·rate limit·지연(5~15s) | | 스트리밍/진행표시/비-AI 경로 |

### 9.4 모델 전략·비용
- 주 모델 / fallback / 저비용 대안 · provider 다변화 · 업그레이드 프로토콜 · **최대 허용 인터랙션당 비용** · 데이터 흐름·보존·동의.

## 11. 성공 지표 (metric tree)

> 목표마다 *측정 가능한 타깃·방향·임계*를 묶는다("engagement 개선" ❌ → "D7 리텐션 ≥ X%" ✅).

- **North Star**(핵심 가치·성장 예측 1지표):
- **Input metrics (HEART)**: Happiness · Engagement · Adoption · Retention · Task success — 각 Goal→Signal→Metric.
- **선행/후행**: 선행 1 + 후행 1 짝.
- **가드레일/카운터 지표(절대 나빠지면 안 됨)**: (예: 지연·마진·에러율·이탈·기존 지면 engagement)
- **(AI면) 모델 품질 지표 vs 제품 성과 지표 분리**: 정확·환각율·안전준수·P95 지연·쿼리당 비용 ↔ task success·채택·리텐션.

> **RAT(Riskiest Assumption Test)**: 실패 시 기능이 죽는 *가장 위험한 가정*을 1개 표시 → 빌드 *전에* 싼 테스트로 검증.

**원칙**: 이 문서는 *무엇*만 담는다 — *어떻게*는 `architecture.md`. 코드와 충돌하면 코드가 진실이고, 문서는 별도 요청 시 갱신한다. 규제·비가역 결정은 ADR로 분리. AI 상세는 16/17로, 요구 상세는 requirements-spec/functional-spec으로 **링크(복사 금지)**. lean 유지(2-3p).
